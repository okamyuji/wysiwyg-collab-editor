# 要求分析書

本書は、複数人で同時にリアルタイム編集できる装飾付きWYSIWYGエディタの要求分析を整理したものである。本書の目的は、ステークホルダーが達成したいビジネス上のゴールと、解決すべき本質的な課題を明確にし、後続の要件定義の入力として確定させることにある。

## 1. プロジェクト背景

社内文書や提案書、議事録、設計書などの作成において、ファイル添付方式やオフライン編集では版管理の取り違えと変更内容の上書きが頻発する。利用者は、同じ画面を見ながら同時に書き込めるGoogle文書のような体験を求めている。一方で、外部SaaSに依存することなく、自社の管理下でデータを保持しながら同等の編集体験を実現したいという要望が背景にある。

本プロジェクトでは、操作変換(Operational Transformation、以下OT)を採用することにより、楽観的に編集を許容し、競合をサーバー側で後から解決する方式とする。これにより、編集者が他者の入力を待たずに自由に文字入力できる体験を提供する。

## 2. ステークホルダー

本システムの関係者を以下の表に整理する。

| ステークホルダー                               | 主な関心事                                                                                                   |
| ---------------------------------------------- | ------------------------------------------------------------------------------------------------------------ |
| 文書編集者(一般利用者)                         | 入力遅延を感じずに、装飾豊かな文書を編集し、他者の編集を即時に確認できることを求める                         |
| レビュアー                                     | コメントを追記し、提案モードで変更を提示し、版を比較し、変更履歴を閲覧できることを求める                     |
| 文書オーナー                                   | 共有権限を細かく制御し、共有リンクを発行および失効し、文書を削除および復元できることを求める                 |
| 運用管理者                                     | 認証連携を維持し、監査ログを保全し、稼働状況を監視し、データのバックアップと復元を運用できることを求める     |
| 開発者                                         | 機能を容易に拡張し、テストを書きやすく、運用を単純化し、技術的負債を抑えた状態で開発を継続できることを求める |
| プロジェクト主査(プロジェクトマネジメント担当) | スケジュール、予算、リスクを管理し、本書を含む成果物の品質を確保することを求める                             |

| 情報セキュリティ担当                           | 脆弱性管理を継続し、ペネトレーションテストを実施し、インシデントの一次対応窓口を担えることを求める           |
| カスタマーサポート担当                         | 利用者からの問い合わせに対応し、共有リンクの失効を代理操作し、復旧を支援できることを求める                   |

## 3. ビジネス要求

本システムが達成すべきビジネスゴールを以下に整理する。

1. 文書作成の所要時間を、現行のメール添付運用と比較してリリース後6か月時点で30パーセント短縮する。比較基準値Xは、本書承認後に利用者調査(N=20)で測定し、要件定義書のNFR章に確定値として記録する。利用者調査(N=20)で事前事後比較を行い、片側p<0.05で有意差を確認する。
2. 同時編集による上書き喪失をゼロにする。本目標はNFR-03整合性で必達条件とする。
3. 自社管理下にデータを保持し、外部SaaSへの本文保存依存を排除する。具体的には、自社運用のS3互換ストレージ(MinIOなど)に画像と添付を保存することは本目標に違反しない。クラウド事業者のマネージドサービスを採用する場合は、契約上のデータ取扱条項を法務・コンプライアンス担当が事前審査する。
4. 利用者が初回ログインから30分以内に最初の文書を作成できることを目標とする。利用者調査(N=20)で計測する。
5. 将来の機能拡張(PDF出力、外部認証連携、AI補完など)に対し、本MVPの実装を破棄せずに追加開発できる基盤を整える。各機能の追加は後続の要件定義書と基本設計書とADR群の決定範囲内でのみ行う。

## 4. 利用者の要求

利用者が直接欲しがっている体験を以下に整理する。

1. 文字入力の遅延が感じられないこと。
2. 他者の入力が即時に画面へ反映され、誰が今どこを編集しているかが見えること。
3. 文字装飾、見出し、箇条書き、画像、表、リンクを学習コストの低い操作で挿入できること。
4. 任意の範囲にコメントを残し、議論のスレッドを保持できること。
5. 変更を直接反映せず提案として残し、オーナーが受諾または却下できること。
6. 過去の版に戻れること。
7. リンクを発行して社外の特定の相手と限定的に共有できること。

## 5. 機能要求(高レベル)

利用者の要求から導出される高レベルの機能を以下にMECEに整理する。詳細は要件定義書で定める。文書中の構造化要素(画像、表、ハイパーリンク)の挿入操作は装飾エディタ機能から分離し、別カテゴリ(画像は機能13、表は機能14、ハイパーリンクは機能15)に集約して扱う。

1. リアルタイム共同編集機能。編集者がブラウザに入力した操作を即時に他の編集者へ伝播する仕組みを提供する。競合は操作変換アルゴリズムで自動解消する。
2. 装飾エディタ機能。テキストに対する太字、斜体、下線、取り消し線、文字色、背景色、見出し階層、箇条書き、番号付きリスト、引用、コードブロック、整列、フォントサイズ段階指定を扱う。
3. プレゼンス表示機能。接続中の編集者一覧と、それぞれのカーソル位置および選択範囲を画面に示す。
4. コメント機能。範囲指定でコメントを残し、返信スレッドと解決状態を管理する。本文への反映を伴わない議論用の情報付与を扱う。コメント本文に他の利用者をメンションでき、メンションされた利用者に通知が届く。メンションの解決は利用者ID(内部識別子)を保持し、画面表示時に表示名へ変換する。同名利用者の衝突は内部IDによって一意に解決される。
5. 提案(サジェスト)モード機能。編集を本文に直接反映せず提案として保留する。受諾された提案のみ本文へ反映する。本文反映を伴う変更提案を扱う点でコメントと棲み分ける。提案にも前項と同じメンション方式(内部ID解決、表示名表示)のメンションを許可する。
6. 版履歴機能。一定間隔および明示的なスナップショット時点の版を保存し、復元と差分閲覧ができる。
7. 共有と権限管理機能。文書ごとに閲覧、コメント、編集、所有の権限を扱う。共有リンクの発行と失効、有効期限の設定ができる。
8. 認証機能。利用者を一意に識別し、操作の主体を記録する。
9. オフライン閲覧機能。ネットワーク切断時にも直前の文書を閲覧できる。MVPの段階では編集はサポートしない。
10. エクスポート機能。文書を、PDF、DOCX、Markdownのいずれかの形式で外部へ書き出せる。エクスポートは印刷を含む(PDF経由で対応する)。
11. 文書管理機能。文書の作成、タイトル変更、削除、復元、所有者変更、自分がアクセス可能な文書の一覧表示を扱う。
12. 文書検索機能。文書タイトルと本文のテキスト検索、所有者または最終更新時刻による絞り込みを扱う。本MVPでは全文検索ではなくPostgreSQLのpg_trgmによる部分一致から始める。
13. 画像と添付機能。文書中に画像を貼り付け、サムネイル付きで表示できる。
14. 表機能。文書中に表を挿入する。MVP範囲は行と列の追加削除と基本装飾とする(具体ライブラリは要件定義書以降で定める)。
15. ハイパーリンク機能。URLとアンカーテキストを指定する。外部リンクと内部リンクを区別して表示する。
16. 通知機能。コメント追加、提案、共有リンク発行、権限変更を当該文書の関係者へ通知する。MVPでは画面内の通知センターとメール送信(コメントと提案のメンション時のみ)に限定する。
17. サポート運用機能。カスタマーサポート担当が、利用者の依頼に応じて共有リンクの代理失効、文書の代理復元、監査ログの当該文書範囲閲覧を行える。MVPでは個別アカウントへの権限付与で運用する。

## 6. 非機能要求(高レベル)

非機能の観点での要求を以下に整理する。

1. 性能
   - 想定する同時編集規模は、1文書あたり同時編集者30名以下、システム全体5000同時接続以下とする。30名はGoogle文書の常識的な同時編集規模を上限根拠とし、5000同時接続は社内の利用者総数約2万名のピーク同時接続率25パーセントから算定する。
   - 入力から他クライアントの画面へ反映されるまでの目標遅延は、計測区間を発信クライアントの入力確定から受信クライアントの画面再描画完了までとし、社内ネットワーク上(RTT 10ミリ秒前提)で平均150ミリ秒以内、95パーセンタイル400ミリ秒以内とする。
2. 可用性
   - 平日業務時間帯(JST 9時から18時)を計測対象とした業務時間帯月次稼働率99.5パーセント以上を目標とする。夜間および休日はベストエフォート運用とし、計画停止を可能とする。
   - 災害復旧の目標としてRPO(Recovery Point Objective)は24時間、RTO(Recovery Time Objective)は4時間を上限とする。
   - 日次の論理バックアップで上記RPOを達成することを基本方針とする。具体的なDB種別は要件定義段階で確定する。
3. 整合性
   - 正常系では、編集途中の操作が失われず、全クライアントが最終的に同じ文書状態へ収束する。異常系(永続化失敗、復旧不能な切断)では、エラー応答と再ロード指示によって利用者へ既知の例外を返す。
   - ネットワーク切断と再接続を伴う場合でも、未ACKの操作を喪失しないこと。具体的な再接続プロトコルは後続の要件定義および基本設計で定める。
4. 安全性
   - 通信は全てTLSで暗号化する。
   - 認証または有効な共有リンクトークンによる認可がない利用者は、文書へアクセスできない。
   - 共有リンクには有効期限と権限を必ず付与する。
   - 認証(利用者の身元確認)と認可(リソースへのアクセス可否判定)を概念として明確に区別する。
5. 監査性
   - 誰がいつどの操作を行ったかを追跡できる。
6. 拡張性
   - 装飾種別の追加、新たな埋め込み要素(動画、図形)の追加が比較的小さな改修で行える構造を持つ。
7. 運用容易性
   - バックアップ、復元、稼働監視、ログ集約の手順が標準化されている。
8. 利用者体験
   - 主要操作はキーボードショートカットで行える。
   - 画面表示は装飾の意図を損なわず、印刷時にも崩れない。
9. プライバシーとコンプライアンス
   - 個人情報(メールアドレス、表示名、IPアドレス)の取扱を法令(個人情報の保護に関する法律、通称: 個人情報保護法)に準拠した範囲に限定する。
   - 監査ログのIPアドレスはHMAC-SHA-256でハッシュ化して保存する。HMAC鍵は環境変数または外部シークレットマネージャから注入し、年1回ローテーションする。
   - 機微情報の取扱は要件定義書NFR-04で具体化する。
10. 脆弱性管理

- 依存ライブラリの既知脆弱性をCIで継続的に検査する(gitleaksおよび依存スキャナ)。
- 本番リリース前および主要機能追加時に外部委託のペネトレーションテストを実施する。
- 検出された脆弱性は、Criticalは72時間以内に是正、Highは30日以内に是正、Mediumは90日以内に是正することを目標とする。期限超過時はリリース停止または機能制限を判断する。

11. 品質ゲート(開発プロセス上の必須要件)

- 秘密情報の混入検知としてgitleaksを必ず適用する。
- フロントエンドの開発と本番ビルド、テスト、フォーマッタ、linterはVite+(素のViteではなく、VoidZeroが提供するVite+を指す)を採用し、Vite+が提供するフォーマッタとlinterを必ず適用する。
- ユニットテストはVite+標準のテストランナー(一次出典はVoidZero公式文書を参照する)を用い、ステートメントカバレッジ80パーセント以上を必達としつつ、テストはAAA(Arrange-Act-Assert)パターンで意味のある振る舞いを検証するように書く。
- E2EテストはPlaywrightを採用し、UX上の主要導線をすべて自動化し、CI上で必ず全件成功させる。
- 上記の品質ゲートは、CIパイプライン上でブロッキング条件として扱い、いずれかが失敗した場合は本番デプロイを禁じる。

## 7. 制約条件

制約条件は以下のとおりである。

1. 技術スタックの方針として、バックエンドにNode.jsとTypeScript、OT基盤にShareDB、フロントエンドのエディタにQuill.jsを採用する。
2. 標準ライブラリと業界標準のオープンソースを優先する。
3. データベースは標準的な選択肢の中から選ぶ。確定値は第13章および要件定義書 第8章を参照する。
4. 自社管理のサーバーまたはプライベートクラウドで運用する想定とする。
5. プロジェクトの開発体制は少人数を想定し、過度な分散構成を避ける。
6. フロントエンドのビルド基盤と開発時のフォーマッタおよびlinterは、Vite+(素のViteではなく、VoidZeroが提供するVite+を指す)のエコシステムに統一する。
7. CI品質ゲートの具体内容は第6章第11項に集約し、ここでは参照のみに留める。

## 8. 前提条件

前提条件は以下のとおりである。

1. 利用者は、Chrome、Edge、Firefox、Safariのいずれも直近12か月以内にリリースされた版を用いる。直近2メジャー版は基本的に12か月以内のリリースに収まる。Internet ExplorerとレガシーEdge(EdgeHTML)は対象外とする。
2. 利用者の端末はネットワークへ安定して接続できる。
3. 認証連携先や運用基盤の詳細は、本MVP範囲外とし、別途のインフラ設計書(本プロジェクトの後続で起票)で扱う。
4. OTのアルゴリズムは、添付資料に示された挿入操作の変換規則を起点として、削除と装飾属性の変換規則を加えて完備する。
5. クライアントとサーバーは同じ操作モデルを共有し、内部表現としてはQuill.jsのDelta形式を採用する。

## 9. 用語定義

主要な用語を以下に定義する。

| 用語             | 定義                                                                                                                                                                                                     |
| ---------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| OT               | 操作変換。中央サーバーが操作を全クライアント共通の順序に確定し、後発の操作を先発の操作に応じて自動的に補正することで、全クライアントの最終状態を一致させる仕組みである。                                 |
| CRDT             | 競合フリー複製データ型。各操作に全順序化可能な識別子を埋め込み、ピア同士の同期だけで決定的に同じ最終状態へ収束させる方式である。                                                                         |
| 操作(Operation)  | 文書に対する最小単位の編集行為であり、挿入、削除、属性変更を含む。                                                                                                                                       |
| Delta            | Quill.jsが採用する操作表現形式であり、保持(retain)、挿入(insert)、削除(delete)の連続として文書差分を記述する形式である。                                                                                 |
| プレゼンス       | 接続中の利用者のカーソル位置、選択範囲、表示名などの状態情報である。                                                                                                                                     |
| スナップショット | 特定時点の文書全体の状態を表す保存物である。                                                                                                                                                             |
| ACK              | サーバーがクライアントへ返す操作確定応答である。当該操作がサーバー側で永続化済みかつ全クライアントへブロードキャスト済みであることを示す応答である。                                                     |
| 収束             | 全クライアントが同じ操作集合を同じ順序で適用したのち、同一の文書状態に到達すること。                                                                                                                     |
| 全体順序         | サーバーが付与する操作の単調な順序であり、全クライアントが共通して採用する順序である。                                                                                                                   |
| 利用者           | 本システムに認証済みでログインしている人(認証済み利用者)、または共有リンクからゲストセッションでアクセスしている人(ゲスト)の総称である。                                                                 |
| 編集者           | 文書に対しeditor以上の権限を持つ認証済み利用者である。                                                                                                                                                   |
| 文書オーナー     | 文書に対しowner権限を持つ認証済み利用者である。文書作成者の初期状態である。                                                                                                                              |
| 文書作成者       | 文書を新規作成した認証済み利用者である。作成と同時にowner権限が付与される。                                                                                                                              |
| ゲスト           | 共有リンクの認証不要型(anyone)から発行されたゲストセッションでアクセスする人である。本システムへのログインは行わない。                                                                                   |
| 権限階層         | 文書アクセス権限の階層であり、低位から高位の順にviewer(閲覧)、commenter(コメント)、editor(編集)、owner(所有)の4段階を持つ。上位は下位の権限をすべて包含する(viewer ⊂ commenter ⊂ editor ⊂ owner)。       |
| 社内利用者       | 本システムへの認証アカウントを持つ自社利用者である。                                                                                                                                                     |
| 社外利用者       | 本システムへの認証アカウントを持たないが、共有リンク経由でアクセスする外部の人である。                                                                                                                   |
| 共有リンク       | 文書を特定の利用者集合へ公開するためのトークン付きURLである。認証必須型(restricted)と認証不要型(anyone)の2種類を扱う。                                                                                   |
| MVP              | Minimum Viable Productの略。本プロジェクトの第一弾リリース範囲を指し、機能と非機能の確定値は本書および要件定義書 第8章で定める。リリース判定基準は、後続の要件定義書で定める品質ゲートの全通過とする。   |
| Vite+            | VoidZeroが提供する商用のVite拡張エコシステムである。素のViteと互換しつつフォーマッタとlinterとテストランナーを統合する。本書および後続文書では、特記なく「Vite+」と表記したものはVoidZeroのVite+を指す。 |

## 10. 添付資料の読み取りと取り込み

提示された図(ファイル名 `GoogleDocsOperationalTransformation.png`、リポジトリの `docs/requirements/assets/` 配下に格納予定、原本は利用者から提供された)から、本要求分析へ反映した内容を以下に列挙する。図中の主要要素を網羅する。図そのものは付録Aへ参照リンクを置く。

1. 同時編集の競合という基本課題と、ロックなしで競合を後から解決するという解決方針の前提を取り込んだ。
2. 操作(意図)を送信し、テキスト全体を送らないという原則を、機能要求と前提条件に取り込んだ。
3. サーバーが操作の全体順序を確立するという原則を、後続の基本設計で扱う設計方針として明示する前提を取り込んだ。
4. サーバーが後続の操作を変換し、最新の文書状態で有効になるようにするというOTの中核処理を、機能要求の中核として取り込んだ。
5. ブロードキャストにより全クライアントへ操作を送信する流れを、機能要求の中核として取り込んだ。
6. 全クライアントが同じ順序で適用することで同じ最終文書状態に収束するという保証を、非機能要求の整合性として取り込んだ。
7. 得られる効果として図中に示された五項目(更新の取りこぼしなし、上書きなし、全員が同じ文書を表示、リアルタイムで動作、大規模でもスケール可能)を、非機能要求の性能と整合性として取り込んだ。
8. 挿入操作のOTルールとして、A = 挿入(pa, la)、B = 挿入(pb, lb)のとき、pbがpa未満なら変換後のpbはpbのまま、pbがpa同等の場合は `(user_id, client_seq)` の辞書順で小さい方をA扱いとして大きい方の操作Bの位置をla分シフト、pbがpaより大きいなら変換後のpbはpb + laとなるという規則を、OTの中核ルールとして取り込んだ。pbとpaが同値の場合(同位置同時挿入)のtie-break規則は要件定義書で定める。図の具体例(位置6に「beautiful」(la=9)を挿入し、後続のBが位置6に「amazing」を挿入する場合)については、要件定義書のtie-break規則に従って位置6または位置15のいずれかへ変換される。
9. 削除には範囲ベースのルール(削除A=(pa, la)と挿入B=(pb, lb)が交差する場合、B側のpbをAの削除範囲外の値へ補正する)があるという補足情報を、要件定義での詳細化の対象として取り込んだ。
10. OTがGoogle Docs、Figma、Quip、ShareDBなどで採用されている事実を、技術選定の妥当性根拠として取り込んだ。
11. 代替アプローチとしてCRDT(サーバー不要)も存在することを認識しつつ、本プロジェクトでは中央サーバー前提のユースケース適合性と、ShareDB(OT実装の運用実績が確認できる主要OSS)の存在を踏まえてOTを採用する判断を、制約条件として明示した。
12. 操作は小さく効率的で文書サイズに依存しないという特性を、性能要件の根拠として取り込んだ。

## 11. リスクと未解決の課題

8. 監査ログの保存期間と可視性の方針が未確定である。
9. 画像や添付ファイルの保存先(DB内、オブジェクトストレージ、ローカルファイル)の方針が未確定である。
10. 提案モードの粒度(1操作単位か、まとめて1提案か)が未確定である。
