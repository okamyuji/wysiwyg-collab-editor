# 要件定義書

本書は要求分析書を入力として、システムが満たすべき機能要件と非機能要件を確定させる。後続の基本設計書と詳細実装指示書の入力として、本書の内容を確定版とする。

## 1. 文書の目的と範囲

本書は、複数人で同時にリアルタイム編集できるWYSIWYGエディタの要件を定める。MVPに含めるべき内容と、将来拡張へ振り分ける内容を明確に区別する。

本書で扱う機能の範囲は、リアルタイム同期(FR-01)、装飾付きエディタ(FR-02)、プレゼンス表示(FR-03)、コメント(FR-04)、提案モード(FR-05)、版履歴(FR-06)、共有と権限管理(FR-07)、認証(FR-08)、オフライン閲覧(FR-09)、エクスポート(FR-10)、画像と添付(FR-11)、文書管理(FR-12)、サポート運用(FR-13)の全13カテゴリである。要求分析書 第5章の17カテゴリとの対応関係および将来拡張への振り分けは第3章序文のマップ表と§3.14将来拡張への振り分けで確定する。

## 2. システム概要

システムは、ブラウザで動作するシングルページアプリケーションと、Node.jsで動作するアプリケーションサーバー、ShareDBによるOT処理エンジン、PostgreSQLによる永続化基盤、Redisによる複数サーバー間のPub/Sub基盤、S3互換オブジェクトストレージによる画像格納基盤、メールパスワード認証によるセッションCookie認証基盤から構成される。

利用者はブラウザでURLを開き、認証を経たうえで文書編集画面に到達する。編集中の操作は逐次サーバーへ送られ、ShareDBが順序付けと変換を行ったうえで、接続中の全クライアントへブロードキャストする。クライアントはサーバーから受け取った操作を自端末のエディタへ適用する。すべての編集者の最終的な文書状態は完全に一致する。

### 3.1 リアルタイム共同編集機能

ID FR-01。本機能は、操作変換(Operational Transformation)によって複数の編集者の同時編集を競合なく統合する。

要件の詳細は以下のとおりである。

1. クライアントはエディタへの入力を、Quill.jsのDelta形式の操作として表現し、サーバーへ送信する。
2. サーバーはShareDBを介して受信した操作に順序を割り当て、必要な変換を施したうえで、接続中の全クライアントへブロードキャストする。
3. クライアントは、サーバーから受信した操作を自端末のエディタに適用し、他者の編集を即時に反映する。
4. ネットワーク切断後の再接続では、未送信の操作を保持し、再接続後にサーバーへ再送する。サーバーは重複した操作を冪等に扱う。
5. 入力から他クライアントへの反映までの目標遅延は、社内ネットワーク上で平均150ミリ秒以内とする。

挿入操作の変換規則は、A = 挿入(pa, la)、B = 挿入(pb, lb)のとき、pb < paなら変換後のpbはそのまま、pb == paの場合は決定論的tie-break規則として `(user_id, client_seq)` の辞書順で小さい方のクライアントの挿入位置を先に確定する(=「user_id_a < user_id_b、または同一user_idならclient_seqが小さい方をA扱い」とする)。pb > paなら変換後のpbはpb + laとする。削除操作と属性変更操作の変換規則は、Quill.jsのDeltaライブラリが提供するtransform関数に準拠する。

### 3.2 装飾付きエディタ機能

ID FR-02。本機能は、文書の装飾と構造化要素を扱う。

サポートする装飾は以下のとおりである。

| 装飾           | 内容                                                                                                                                                                                        |
| -------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| 太字           | テキストを太字で表示する                                                                                                                                                                    |
| 斜体           | テキストを斜体で表示する                                                                                                                                                                    |
| 下線           | テキストに下線を引く                                                                                                                                                                        |
| 取り消し線     | テキストに取り消し線を引く                                                                                                                                                                  |
| 文字色         | テキストの色を、基本設計第3.4a節で確定する16色のカラーパレットから選択して指定する。任意HEXは受け付けない。理由はNFR-04第10項のCSP `style-src-attr` ハッシュallowlistを有限に保つためである |
| 背景色         | テキストの背景色を、基本設計第3.4a節で確定する16色のカラーパレットから選択して指定する。任意HEXは受け付けない。理由は同上                                                                   |
| 見出し         | 段落を1から6までの階層化された見出しに変換する                                                                                                                                              |
| 箇条書き       | 段落を階層化された箇条書きに変換する                                                                                                                                                        |
| 番号付きリスト | 段落を階層化された番号付きリストに変換する                                                                                                                                                  |
| 引用           | 段落を引用ブロックとして表示する                                                                                                                                                            |
| コードブロック | 段落を等幅フォントのコードブロックとして表示する                                                                                                                                            |
| 整列           | 段落を左寄せ、中央寄せ、右寄せ、両端揃えのいずれかに整列する                                                                                                                                |

サポートする構造化要素は以下のとおりである。

| 要素           | 内容                                                                      |
| -------------- | ------------------------------------------------------------------------- |
| 画像           | 画像をアップロードして文書に挿入し、サイズと配置を変更する                |
| 表             | 後述するMVP範囲に限定した行と列の追加削除と基本装飾を行う                 |
| ハイパーリンク | URLとアンカーテキストを指定する。外部リンクと内部リンクを区別して表示する |

表機能のMVP範囲は以下のように限定する。OT変換、PDFエクスポート、DOCXエクスポートでは装飾と構造を完全保持する範囲のみを扱う。Markdownエクスポートのみは仕様の表現力の制約から劣化変換を許容し、利用者にその旨を明示する。

1. 行と列の追加削除のみを扱う。セル結合と分割はMVPの対象外とし、将来拡張とする。
2. セル内には平文テキスト、太字、斜体、下線、文字色、背景色、ハイパーリンクの装飾のみを許可する。見出し、箇条書き、画像、入れ子の表はセル内に配置できない。
3. 表全体の幅は内容幅に追従し、列幅の手動調整はMVPの対象外とする。
4. Quill.jsの拡張モジュールとしては `quill-better-table` を採用する前提とし、Delta表現はBlot単位の `table`、`table-row`、`table-cell` の固定スキーマで保持する。
5. PDFとDOCXエクスポートでは、本MVP範囲のセル装飾(文字色、背景色を含む)をすべて保持する。
6. MarkdownエクスポートはGitHub Flavored Markdownのパイプテーブルにマップする劣化変換である。セル背景色と文字色はMarkdownでは表現できないため失われる。表セル内の装飾(太字、斜体、ハイパーリンクなど)はMarkdown構文へ変換するが、ネストや改行を含むセル内容は警告を表示し、近い表現に置き換える。利用者にはエクスポート前に「Markdownエクスポートでは表のセル装飾の一部が失われる」旨を表示する。

### 3.3 プレゼンス表示機能

ID FR-03。本機能は、文書を編集中の利用者の存在と位置を可視化する。

要件の詳細は以下のとおりである。

1. 文書を開いている全利用者の表示名と固有色を、画面右上のリストに表示する。
2. 各利用者のカーソル位置と選択範囲を、固有色のキャレットと半透明のハイライトでエディタ上に重ねて表示する。
3. 利用者の表示名と固有色は、利用者プロフィールに紐付ける。
4. 操作のない接続は、30秒の間隔でハートビートを送ることで生存確認する。

### 3.4 コメント機能

ID FR-04。本機能は、文書の任意の範囲に対するコメントとスレッドを管理する。

要件の詳細は以下のとおりである。

1. 任意のテキスト範囲を選択した状態で、コメントを開始できる。
2. コメントは一意のIDを持ち、文書内のアンカー(開始位置と終了位置)と関連付けられる。
3. コメントには返信を追加できる。返信はスレッドとして時系列に表示する。
4. コメントは未解決と解決の2状態を持つ。解決済みコメントは折り畳まれて表示する。
5. 文書のテキストが編集されたとき、アンカーの位置はDeltaの変換規則に従って自動追従する。
6. アンカー範囲が削除された場合、コメントは孤立状態となり、画面側面の一覧にのみ残る。
7. コメントの作成者は、認証済み利用者または共有リンクFR-07経由のゲストセッション(commenter権限)のいずれかである。ゲストセッションが作成したコメントは、表示名として「ゲスト+ゲストセッションIDの短縮表記」を用いる。
8. ゲストが作成したコメントの解決操作は、コメント権限(commenter)以上の認証済み利用者のみが行える。
9. 監査ログには、コメントの追加、返信、解決、削除のすべてを、作成者IDまたはゲストセッションIDと共に記録する。

### 3.5 提案モード機能

ID FR-05。本機能は、編集を本文に直接反映せず、提案として保留する。

要件の詳細は以下のとおりである。

1. 利用者は提案モードと通常編集モードを切り替えできる。
2. 提案モード中の操作は、提案集合に蓄積され、本文の状態へは即時反映しない。
3. 提案は、編集者ごとに固有色で表示し、追加と削除を見え消し表現で示す。
4. 文書の編集権限を持つ利用者は、各提案を受諾または却下できる。受諾した提案はその時点の本文状態へ反映される。
5. 提案の粒度は、利用者が連続して入力した一連の操作を1提案として束ねる。空白を挟む連続入力の境界は、5秒の無操作で切る。本値は第8章 項番10で確定済みであり、本書承認をもって決定値として扱う。
6. 提案集合は文書とは独立した版で管理し、本文のOT処理に影響を与えない。
7. 提案は、作成時点のShareDB版番号を `base_version` として必ず保持する。これは受諾時に本文との位置整合を取るための必須属性である。提案作成は認証済みの編集権限保持者(editor以上)に限定する。ゲストセッションは提案を作成できない。
8. 提案受諾時のサーバー処理は以下の手順で行う。
   1. 当該提案レコードのoptimistic_versionを読み込む(リード値V1)。
   2. base_version以降に本文へ確定した操作集合(opsの差分)を版番号昇順で順番に取得する。
   3. 提案Deltaを各opに対してQuill.jsのDelta `transform` でリベースする。
   4. リベース過程でアンカー範囲が削除されるなど追従不能な状態が検出された場合は、提案を `stale` に遷移させHTTP 409 Conflictで応答する。
   5. 受諾は二段CAS方式で原子性を確保する。詳細は基本設計第5.6節を正本とする。要点は以下の順序である。第一CAS(claim)で `UPDATE suggestions SET status='accepting', status_changed_at=now(), status_changed_by=$user, optimistic_version=optimistic_version+1 WHERE id=$2 AND optimistic_version=V1 AND status='pending'` を実行し、0行更新ならHTTP 409。第一CAS成功後にのみリベース済みDeltaをShareDBへsubmitOpし、submitOp失敗時は `status='pending'` への補償CASを実行してHTTP 503。submitOp成功後に第二CAS(confirm)で `UPDATE suggestions SET status='accepted', applied_version=$1, status_changed_at=now(), optimistic_version=optimistic_version+1 WHERE id=$2 AND status='accepting'` を実行して確定する。本フローにより本文submitOpと提案状態の中間整合性が保たれ、リトライ時の二重適用が不可能となる。提案状態enumには `accepting` を含める(第6章 提案エンティティ定義および基本設計第3.2節suggestions行と整合)。
   6. 利用者は警告を受けた場合、手動編集で確定するか、提案を却下する。
9. 提案には状態として `pending`、`accepting`、`accepted`、`rejected`、`stale`、`expired` の6種類を持たせる。`accepting` は二段CASの第一CASで占有された一時状態である。staleとexpiredの判定基準は同一(transform追従不能)であり、検出契機が異なるのみである。両者を分けるのは利用者向けUIの差別化のためであり、staleは「受諾を試みた直後の判定」として警告ダイアログで通知、expiredは「バックグラウンド検出」として通知センターで通知する。状態遷移は以下のとおりである。本表は基本設計第3.2節suggestions DDLおよび基本設計第5.6節と整合する。

| 元状態            | 遷移先    | 検出契機                                                                  | 通知UI                                     | 不可逆                               |
| ----------------- | --------- | ------------------------------------------------------------------------- | ------------------------------------------ | ------------------------------------ |
| pending           | accepting | 受諾の第一CAS成功(submitOp実行前の占有)                                   | (内部状態、利用者には可視化しない)         | 可逆(補償CASでpendingへ戻る場合あり) |
| accepting         | accepted  | submitOp成功後の第二CAS確定                                               | 受諾完了                                   | 不可逆(以後変更不可)                 |
| accepting         | pending   | submitOp失敗時の補償CAS(リトライ救済のため可逆)                           | (内部状態、HTTP 503応答後にUIで再試行案内) | 可逆                                 |
| pending           | rejected  | 編集権限保持者の却下API                                                   | 却下完了                                   | 不可逆                               |
| pending           | stale     | 受諾時にリベース不能と判定(アンカー範囲が削除済みまたはtransform結果が空) | 受諾ダイアログで警告                       | 不可逆                               |
| pending           | expired   | 10分間隔の自動バッチでbase_versionから現在版までtransform追従不能と判定   | 通知センターで通知                         | 不可逆                               |
| 他状態 → 上記以外 | 不可      | 通常遷移以外は禁止                                                        | -                                          | 状態の直接遷移は禁ずる               |

`accepted` のときのみ `applied_version` をNOT NULLとする。それ以外の状態ではNULLである。CHECK制約で強制する。`stale` および `expired` からの再活性化は、利用者が明示操作で新規 `pending` 提案として複製して再提示する形のみとする。

### 3.6 版履歴機能

ID FR-06。本機能は、文書の過去状態を保存し、復元と差分閲覧を可能にする。

要件の詳細は以下のとおりである。

1. 文書の状態を、明示的なスナップショットおよび自動スナップショットによって版として保存する。
2. 自動スナップショットは、最後の版作成から1時間以上経過し、かつ100件以上の操作が積み上がったときに作成する。
3. 利用者は版の一覧から任意の版を開き、その時点の文書を閲覧できる。
4. 利用者は任意の版を現在の文書状態として復元できる。復元は新たな操作として記録され、過去の版は失われない。
5. 利用者は任意の二つの版の差分を表示できる。差分は装飾を保持した見え消し表現で示す。
6. 版の保存上限は、自動スナップショットのみ文書1件あたり1000件とする。明示スナップショットは無制限保持とし、削除しない。上限到達時(1001件目作成時点)は、最古の自動版区間1件分のみを対象に以下を同一トランザクション内で実行する。第1段階で当該1区間のopsを `archived_ops` テーブルへ退避(圧縮)する。第2段階で対応する `revisions.snapshot_payload` を退避境界の `v_end` 時点のスナップショットで埋める(Revisionの到達可能性不変条件を保つため必須)。第3段階で当該1区間の自動版レコードのみを `revisions` から物理削除する。本処理は1上限超過につき1区間のみとし、複数件の一括削除は行わない(処理粒度を最小化し、復元の到達可能性連続を保つため)。

### 3.7 共有と権限管理機能

ID FR-07。本機能は、文書ごとのアクセス制御と共有リンク(社内向けの認証必須型と社外向けの認証不要型の両方)の発行を扱う。

要件の詳細は以下のとおりである。

1. 文書は、所有(owner)、編集(editor)、コメント(commenter)、閲覧(viewer)の4種類の権限を持つ。本書を通じて英日対応は「日本語表記(英表記)」の順で統一する。
2. 所有権限は、編集権限の操作に加え、権限の付与と剥奪、文書の削除を行える。
3. 編集権限は、本文の編集、画像と表の挿入、版の作成、提案の受諾と却下を行える。
4. コメント権限(commenter)は、本文の閲覧とコメントの追加返信解決を行える。
5. 閲覧権限は、本文と版履歴の閲覧のみを行える。
6. 利用者は、文書ごとに個別のメールアドレスへ権限を付与できる。
7. 共有リンクは、以下の2種類のいずれかとして発行する。発行時に必ず種別を選択する。
   1. 認証必須型(restricted)。リンクを開いた閲覧者は本システムへログインしている必要がある。ログインしていない場合はログイン画面へ誘導し、ログイン後にリンク先文書へ自動遷移する。
   2. 認証不要型(anyone)。リンクを開いた閲覧者は本システムへログインせずアクセスできる。アクセス時は匿名ゲストとして扱い、本文への監査追跡用にゲストセッションID(UUID v7、本書ではゲストセッションIDと同義であり、以下「ゲストセッションID」で統一する)を発行して監査ログに記録する。
8. 共有リンクで付与できる権限の上限は閲覧(viewer)とコメント(commenter)の2種類のみとする。共有リンク経由で編集権限(editor)以上を付与することは禁ずる。編集権限の付与は必ず認証済み利用者へのメールアドレス単位での個別付与に限定する。
9. 共有リンクには有効期限を必ず設定する。有効期限の上限はMVPで30日とする。
10. 共有リンクは作成者または所有者により失効できる。失効後は即時にアクセス不能とし、失効後にトークンでアクセスがあった場合は監査ログへ記録のうえHTTP 410(Gone)で応答する。
11. 共有リンクは推測困難なランダムトークンを含む。トークンは `crypto.randomBytes(32).toString('base64url')` で生成し、長さは43文字となる。
12. 共有リンクで閲覧した利用者の操作も、ゲストセッションIDで監査ログに記録する。
13. ローカルキャッシュ(ServiceWorker、ブラウザストレージ)の取扱は、本機能の権限剥奪と共有リンク失効に同期して、対象文書のキャッシュエントリを必ず無効化する。詳細はFR-09に定める。

### 3.8 認証機能

ID FR-08。本機能は、利用者の身元確認と認可の前提となるセッションを管理する。

要件の詳細は以下のとおりである。

1. 認証方式は、メールアドレスとパスワードによるセッションCookie認証とする。
2. パスワードのハッシュ方式、レートリミット、CSPなどの実装条件はNFR-04に集約する。FR-08側では参照に留める。
3. セッションCookieはHttpOnly属性、Secure属性、SameSite=Lax属性を持つ。
4. セッションの有効期限は、最終アクセスから2週間とする。
5. 利用者はパスワードリセットを、メール経由のワンタイムリンクで実施できる。リンクの有効期限は発行から1時間とする。
6. 利用者は自身のアカウントの表示名、固有色、アイコンを設定できる。

### 3.9 オフライン閲覧機能

ID FR-09。本機能は、ネットワーク切断時の最低限の文書アクセスを提供する。

要件の詳細は以下のとおりである。

1. 直前に閲覧した文書のスナップショットをServiceWorker経由でブラウザストレージにキャッシュする。
2. オフライン状態では、キャッシュされた文書の閲覧のみを許可する。編集は無効化する。
3. ネットワーク復帰時には、サーバーから最新状態を取得して画面を更新する。
4. オフライン編集はMVPの対象外とする。
5. ログアウト時には、当該利用者がローカルに保持する文書スナップショット、画像、UIシェル以外のキャッシュエントリをすべて削除する。
6. オンライン復帰直後の最初のサーバー応答で、当該利用者の当該文書への現在権限を確認する。権限が剥奪済み、もしくは文書が削除済みであった場合、ローカルキャッシュから当該文書を即時に削除し、画面を権限なしの表示に切り替える。
7. オンライン中の権限剥奪または共有リンク失効はWebSocketからの即時プッシュ通知で反映する(目標反映遅延500ミリ秒以内)。プッシュ通知が届かないオフラインクライアントはオンライン復帰時の権限確認で反映され、目標反映遅延は復帰検知から1秒以内とする。
8. 共有リンク経由のゲストアクセスでは、ServiceWorkerによる文書キャッシュを行わない。共有リンクの失効をローカル側で確実に反映するためである。
9. キャッシュ整合性のため、サーバーは文書ごとに `cache_etag` を発行する。`cache_etag` は、本文最新版番号、メタデータ最終更新時刻、当該利用者の現在権限種別、共有リンクIDの4要素をHMAC-SHA-256で結合した固定長文字列とする。クライアントは `If-None-Match` ヘッダで照合し、変化した場合はキャッシュを破棄して最新を取得する。権限が変わった場合は `cache_etag` も自動的に変わるため、権限変更後のキャッシュ流出を防止できる。

### 3.10 エクスポート機能

ID FR-10。本機能は、文書を外部形式へ書き出す。

要件の詳細は以下のとおりである。

1. 文書を、PDF、DOCX、Markdownのいずれかの形式でダウンロードできる。
2. PDFエクスポートでは、装飾、画像、表、見出し階層を保持する。改ページは見出し位置で揃える。
3. DOCXエクスポートでは、装飾と段落構造をWord形式の対応要素に変換する。
4. Markdownエクスポートでは、装飾と構造をCommonMarkとGitHub Flavored Markdownの範囲で表現する。Markdownで表現できない装飾は警告を表示し、近い表現に置き換える。
5. エクスポート処理はサーバー側で非同期に実行する。API契約は以下とする。
   1. `POST /api/documents/:id/exports` は形式(pdf/docx/markdown)を指定して呼び出す。サーバーは即時にジョブID、`status=queued`、ポーリングURL、生成完了時に発火するWebSocketチャネル名を返す(202 Accepted)。本APIは生成完了を待たない。
   2. クライアントは `GET /api/exports/:job_id` でジョブ状態を取得できる(`queued`/`running`/`succeeded`/`failed`/`cancelled`)。
   3. `status=succeeded` のときレスポンスにダウンロードリンク(署名付きURL、有効期限24時間)を含める。
   4. 進捗のプッシュは既存WebSocket上の `export:<job_id>` チャネルで配信する。クライアントはポーリングとプッシュのいずれかを選択できる。
6. エクスポートの所要時間目標はA4換算50ページの文書で30秒以内とする(ジョブ受領時刻から `status=succeeded` までで計測)。

### 3.11 画像と添付機能

ID FR-11。本機能は、画像のアップロードと表示を扱う。

要件の詳細は以下のとおりである。

1. 画像はS3互換オブジェクトストレージへ保存する。MVPの実装ではMinIOを利用する。
2. アップロード可能な画像形式は、PNG、JPEG、GIF、WebPの4種類とする。
3. 画像1枚あたりの上限サイズは10メガバイトとする。
4. アップロードされた画像は、サーバー側で1280ピクセル、640ピクセル、320ピクセルの3サイズのサムネイルを生成する。エディタ表示と印刷出力で最適なサイズを使い分ける。
5. 画像のメタデータ(Exif情報)はサーバー側で除去する。

### 3.12 文書管理機能

ID FR-12。本機能は、文書のライフサイクル管理を扱う(要求分析カテゴリ11に対応)。

要件の詳細は以下のとおりである。

1. 認証済み利用者は新規文書を作成できる。作成者が所有者となる。
2. 所有者は文書のタイトル変更、削除(論理削除)、復元(論理削除済みからの戻し)、所有者変更を行える。
3. 利用者は自分が所有者または編集権限または閲覧権限を持つ文書の一覧を、最終更新時刻降順で表示できる。
4. 削除済み文書は30日のグレース期間で物理削除し、グレース期間中はサポート運用(FR-13)の代理復元の対象となる。
5. 一覧表示は1ページ50件、サーバー側のcursor pagination、所有者単位のフィルタを提供する。本MVPでは全文検索は対象外(§3.14将来拡張 第1項)。

### 3.13 サポート運用機能

ID FR-13。本機能は、運用管理者、情報セキュリティ担当、カスタマーサポート(CS)担当の業務操作を扱う(要求分析カテゴリ17に対応)。

要件の詳細は以下のとおりである。

1. ユーザーロール基盤として `user_roles` を持ち、`standard`、`operations_admin`、`security_admin`、`cs_admin` の4種別を付与または失効できる。組織ロールの付与と失効は運用管理者のみが行える。
2. 監査ログ閲覧の権限と範囲は以下に固定する。運用管理者と情報セキュリティ担当は全件閲覧可。CS担当(`cs_admin`)は「(a)依頼元の利用者IDまたは(b)問い合わせ対象の文書ID」のいずれかを必須クエリパラメータとして指定する場合に限り、当該範囲のみ閲覧可とする。任意全文検索や全件閲覧は許可しない。一般利用者は自身の所有文書の監査ログのみ閲覧できる(NFR-05第6項と整合)。本範囲限定は要求分析カテゴリ17「当該文書範囲閲覧」と整合し、CS権限の過大付与を防ぐ。
3. CS担当は、利用者の依頼に基づき共有リンクの代理失効を行える。代理失効の操作は監査ログにaction=`share_link_revoke_by_cs` で記録され、payloadに依頼元の利用者IDと依頼の問い合わせIDを保持する。
4. CS担当は、論理削除済み文書のグレース期間内に限り、所有者からの依頼に基づき文書を代理復元できる。代理復元は監査ログにaction=`document_restore_by_cs` で記録する。
5. 情報セキュリティ担当はハッシュチェーン検証バッチ(NFR-05第5項)の検証結果閲覧、CSP違反レポート(NFR-05第9項)の閲覧を行える。
6. 運用管理者は秘密管理基盤の世代管理(secret_versions、NFR-05第5項)を操作できる。

### 3.14 将来拡張への振り分け

本節は要求分析17カテゴリのうちMVP対象外として将来拡張に振り分けたカテゴリを保持する。各項目はMVPリリース後の運用評価で優先順位を確定し、新規ADRを起票して取り込む。

1. 文書検索(要求分析カテゴリ12)。MVP外。文書一覧のページングと所有者フィルタはFR-12で提供するが、文書本文の全文検索(pg_trgm部分一致または別途検索エンジン)は本MVPで実装しない。

4. 外部認証連携(OIDC等)。NFR-04第5節の外部連携にHIBP・SMTP・S3互換のみとし、認証連携は本MVPで実装しない。
5. ペネトレーションテストの自動化。NFR-04第14項は手動年1回。

## 4. 非機能要件

### 4.1 性能

ID NFR-01。性能要件は以下のとおりである。

1. 入力から他クライアントへの反映までの目標遅延は、計測区間をクライアントの入力検知からブロードキャスト受信までとし、社内ネットワーク(RTT 10ミリ秒前提)で平均150ミリ秒以内、95パーセンタイル400ミリ秒以内とする。
2. 1文書あたりの同時編集者は最大30名を想定する。31名目以降のWebSocket購読要求は、サーバーがHTTP 503 Service Unavailable(WebSocket発展時はWebSocketメッセージで `error code=4503`)を返して拒否する。クライアントは「同時編集者数上限に達しています。しばらく待ってください」を画面に表示し、利用者は手動再試行できる。本上限はサーバー側で文書購読カウンタ(`document:<id>:subscribers` Redisキー、TTL接続生存中)で実装する。
3. システム全体の同時接続数は5000を上限とする。5001接続目以降のWebSocketアップグレード要求は、リバースプロキシ層でHTTP 503を返して拒否する。プロキシは指数バックオフの `Retry-After` ヘッダ(初回5秒、上限60秒)を付与し、クライアントは自動再試行する。サーバー側でも `verifyClient` フックで同接続数を確認し、Redis上の `system:active_connections` カウンタが5000を超えていればHTTP 503で拒否する。プロキシ層と二重防御とする。
4. 文書1件のサイズは1メガバイトを目安とし、超過時には警告を表示する。

### 4.2 可用性

ID NFR-02。可用性要件は以下のとおりである。

1. 月間稼働率は99.5パーセント以上を目標とする。
2. 計画停止の時間帯は事前告知のうえ、平日業務時間外に実施する。
3. アプリケーションサーバーは2台以上の冗長構成とする。
4. ShareDBの永続化先となるPostgreSQLは、フェイルオーバー可能なレプリカ構成とする。

### 4.3 整合性

ID NFR-03。整合性要件は以下のとおりである。

1. 全クライアントの最終的な文書状態は完全に一致する。
2. 順序が逆転した操作はサーバー側で正しい順序へ変換し、結果として収束する。
3. 通信切断と再接続を伴う場合でも、サーバーが受領済みの操作は失われない。サーバーが受領した時点で永続化層(`ops`、`snapshots`)に書き込まれているためである。未送信の編集はMVPでは扱わない(FR-09第4項により、切断中の新規入力は受け付けない)。冪等性判定情報(ack_seq)は再接続時に必須であり、その消失時の救済手順は項5に従う。
4. 同一クライアントから連続して送信された操作は、サーバー側で受信順を維持する。
5. ack_seqは再接続時の冪等性判定に必須である。ack_seqはRedis(短期参照、TTL 7日)とPostgreSQL `ack_seq_snapshots` テーブル(長期保持、30日)に二段で永続化する。再接続時にRedisから読めない場合はPostgreSQLから復元する。両方から読めない場合のみサーバーは `force_reload` を発令し、クライアント側の未確定編集はクリップボードへ退避してから最新スナップショットを再取得する(設計上の救済手順は基本設計書第5.3.1節に従う)。

### 4.4 安全性

ID NFR-04。安全性要件は以下のとおりである。

1. すべての通信はTLS 1.3で暗号化する。TLS 1.2は古いCBC暗号スイートを無効化したうえで互換のみ許容する。SSLv3とTLS 1.0と1.1は禁ずる。
2. セッションCookieはHttpOnly属性、Secure属性、SameSite=Lax属性を必ず付与する。
3. パスワードはArgon2idでハッシュ化し、OWASP Password Storage Cheat Sheetの推奨パラメータ(メモリ19MiB、反復回数2、並列度1、出力長32バイト、ソルト長16バイト)を最低値とする。
4. パスワードは、長さ12文字以上、3種類以上の文字種(英大文字、英小文字、数字、記号)を含む。利用者が選んだパスワードはHave I Been Pwned API(k-Anonymity形式)で漏えい辞書照合する。
5. 入力されたHTMLはサーバー側およびクライアント側の双方でサニタイズする。サーバー側はisomorphic-dompurifyを用いる。
6. CSRFトークンはcsrf-csrfミドルウェアのDouble Submit Cookie + HMAC方式で発行する。状態変更系のリクエストにX-CSRF-Tokenヘッダを必須化する。GETリクエストとSafeメソッドは適用対象外とする。WebSocketは同一オリジン要求(Originヘッダ照合)を必須とする。例外: ブラウザ自動送信であるCSP違反レポート受信エンドポイント `POST /api/csp-report` はX-CSRF-Tokenを付与できないため、本要件のCSRF検証対象外とする。代替として基本設計第4.1節で定める5項目(Origin検証、Refererドメイン一致検証、IPあたり1分10件のレートリミット、payload 8KiB上限、同一blocked_uri+violated_directive組合せの1分窓集約)で偽報告を抑制する。本例外を設けないと、CSPレポートが全件拒否され、NFR-05第1項の `csp_violation` 監査の発火源が消失して、silent rejection防止意図に反するため必須の例外である。
7. ファイルアップロードでは、Content-Typeとファイル拡張子と先頭バイト(マジックバイト)の三重チェックを行う。
8. 共有リンクのトークンは、暗号論的に安全な乱数で生成し、base64urlで43文字固定とする(crypto.randomBytes(32)由来)。
9. 文書アクセスの判定は、API層とWebSocket層の双方で行う。WebSocketの判定点は以下の三分類で定める。
   - 認可必須(7点): connect、read、fetch、submit、apply、sendPresence、receivePresence。これらは判定に失敗した時点でクライアントへエラーを返し、操作を拒否する。
   - 認可必須(追加2点): verifyClient(HTTPアップグレード前段のOriginと認証Cookie照合)、commit(submitの後段、確定直前の権限再確認)。これらは7点と同じく拒否権限を持つ。
   - 事後監査のみ(1点): afterWrite(永続化後)。本フックは権限判定の対象外であり、監査ログ書き込みと改ざん検出チェーン追記の用途に限定する。
     合計10点で、うち9点が認可必須、1点が事後監査である。各点の正本は基本設計書 第4.2節および第6.2節とADR-0010の表とする。
10. Content-Security-Policy(CSP)はnonce方式で配信する。`script-src` は `'self'` とnonceのみ許可し、`'unsafe-inline'` を禁ずる。`style-src` も `'self'` とnonceのみ許可し、`'unsafe-inline'` を禁ずる。Quill.jsが動的に挿入する `<style>` 要素にはサーバー発行のnonceを同期付与する。Quill 2系の色、背景色、字下げ、整列は要素の `style="..."` 属性で表現されるため、CSP3仕様により `style-src` のnonceは属性値に効かない。属性値の許可は `style-src-attr` ディレクティブで個別制御し、`'unsafe-hashes'` とビルド時に算出したSHA-256ハッシュallowlistで運用する(`'unsafe-inline'` の `style-src-attr` は禁ずる)。文字色と背景色はあらかじめ定義されたカラーパレットからの選択に限定し、任意HEXは受け付けないことで属性値ハッシュ集合を有限に保つ(FR-02および基本設計第3.4a節のカラーパレット16色固定仕様と整合)。`img-src` は `'self'` とdata: とMinIO配信ドメインを許可、`connect-src` は `'self'` とwss://、`frame-ancestors` は `'none'`、`object-src` は `'none'`、`base-uri` は `'self'` を設定する。詳細はADR-0017の確定値表に従う。
11. HTTPレスポンスにHSTS(max-age=63072000; includeSubDomains; preload)、X-Content-Type-Options、X-Frame-Options(DENY)、Referrer-Policy(strict-origin-when-cross-origin)、Permissions-Policyを必ず付与する。実装はhelmetに集約する。
12. 認証関連のレートリミットは以下のとおりである。ログインはアカウント単位(メールアドレス)で10分以内に5回失敗で10分間ブロック、IPアドレス単位で15分以内に20回失敗で30分間ブロック。パスワードリセット要求はメールアドレス単位で1時間以内に3回まで。共有リンクトークンへのアクセスは(リンクトークン単位)で1時間以内に60回まで、かつ(IPアドレス単位)で1時間以内に300回までとする。Redisキーは `rate:login:<email>`、`rate:login:ip:<ip_hash>`、`rate:pwreset:<email>`、`rate:share:<token>`、`rate:share:ip:<ip_hash>` の形で保持し、失敗回数とTTLを管理する。
13. ゲストセッションは、ブラウザCookie +短命WebSocketトークン の二段で管理する。Cookieは `Path=/share/<token>` 限定の `guest_session` でゲストセッションIDをブラウザに保持し、共有リンクページ内でのみ送信されるようにする。WebSocketの `/sharedb` パスへCookieが届かない問題を回避するため、共有リンクページのロード時にサーバーは当該ゲストセッション用の短命WebSocketトークン(有効期限15分、HMAC署名、ペイロードにguest_session_idとshare_link_idと権限種別)を発行し、クライアントはWebSocket接続時のサブプロトコル `Sec-WebSocket-Protocol: guest-token.<token>` でサーバーへ送信する(プロトコル名と値はドット連結の単一プロトコル識別子として送信する。カンマ区切りで「プロトコル名と値を別トークンとして送る」は採用しない。本書式は基本設計第5.9節と同一)。サーバーは `verifyClient` フックで本トークンを検証し、有効ならゲストセッションをWebSocketセッションに紐付ける。WebSocketサブプロトコル方式はCookieに依存せず、CSRF耐性も保つ(Originヘッダ照合と併用)。本MVPでは認証済み利用者はセッションCookie、ゲスト利用者は短命WebSocketトークンの二系統で `verifyClient` を判定する。
14. ペネトレーションテストは、本番リリース前と主要機能追加時に外部委託で実施する。年1回の定期実施を必須とし、本番リリースは以下の重要度別対応で判定する。

| High     | リリース停止             | 検出時点から30日以内に是正   |
| Medium   | リリース可、是正計画必須 | 検出時点から90日以内に是正   |
| Low      | リリース可、累積管理     | 次回ペネトレ実施まで         |

### 4.5 監査性

ID NFR-05。監査性要件は以下のとおりである。

1. 監査ログの対象操作は本項に列挙したaction識別子の集合とし、第9項のpayload表と1対1で対応する。本MVPで記録するactionは以下の通り。認証(`auth_login`、`auth_logout`、`password_reset_request`、`password_reset_confirm`)、文書作成(`document_create`)、文書削除(`document_delete`)、文書通常復元(`document_restore`)、文書代理復元(`document_restore_by_cs`、FR-13由来)、文書物理削除(`document_purge`、FR-12第4項の30日グレース期間満了)、文書タイトル変更(`document_title_change`、FR-12由来)、文書所有者変更(`document_owner_change`、FR-12由来)、権限変更(`permission_change`)、共有リンク発行(`share_link_issue`)、共有リンク失効(`share_link_revoke`)、共有リンク代理失効(`share_link_revoke_by_cs`、FR-13由来)、コメント(`comment_add`、`comment_reply`、`comment_resolve`、`comment_delete`)、提案(`suggestion_create`、`suggestion_accept`、`suggestion_reject`)、版作成(`revision_create`)、版復元(`revision_restore`)、エクスポート実行(`export_request`、`export_succeed`、`export_fail`、`export_cancel`、`export_worker_handoff`)、版圧縮(`ops_compaction`)、強制再ロード(`force_reload`)、ServiceWorkerキャッシュ無効化(`cache_invalidate`)、ユーザーロール付与(`user_role_grant`、FR-13由来)、ユーザーロール失効(`user_role_revoke`、FR-13由来)、監査ログ閲覧(`audit_log_view`、FR-13由来)、秘密管理基盤の世代操作(`secret_version_rotate`、FR-13由来)、ペネトレ実施結果(`pentest_result`)、容量上限超過拒否(`capacity_reject_subscribe`はNFR-01第2項の31名目以降の文書購読拒否、`capacity_reject_connect`はNFR-01第3項のWebSocketアップグレード拒否のうちアプリ層検出分、`capacity_reject_proxy`はプロキシ層検出分)、CSP違反レポート受信(`csp_violation`、ブラウザの `report-uri=/api/csp-report` 経由)、版到達可能性不変条件違反検出(`reachability_violation`、archived_ops purge保険チェック)、HIBP連携失敗(`hibp_unavailable`、NFR-04第4項のローカル要件のみ暫定許可時)、DR訓練結果(`dr_drill_result`、NFR-07第6項)、システムが自動実行するバッチ処理(提案の自動expired検出は`suggestion_expire_batch`、ハッシュチェーン日次検査は`audit_chain_verify`、エクスポートワーカーの孤児ジョブ復旧は`export_reaper_recover`)、認証されていない利用者と認可されていない利用者の操作試行(401、403を返したリクエストは`unauthorized_attempt`)。actor_kind=systemはバッチおよびシステム由来の拒否が発火源となる。リバースプロキシ層での503拒否は、プロキシのアクセスログをFluent Bit経由で集約し、アプリケーションサーバーは1分間隔で集約済みカウンタを `capacity_reject_proxy` として監査ログへバッチ追記する。本項のaction集合は第9項payload表と完全に1対1で対応する(突合はQUALITY_RUBRICの観点で機械的に検査する)。
2. 監査ログは1年間保持する(第8章で確定)。
3. 監査ログには、actor_kind(user、guest、systemのいずれか)、利用者ID(actor_kind=userのときNOT NULL、それ以外はNULL)、ゲストセッションID(actor_kind=guestのときNOT NULL、それ以外はNULL)、対象リソース種別、対象リソースID、操作種別、時刻、IPアドレスのHMAC-SHA-256ハッシュ(個情法対応のため平文を保存しない)、UserAgent、構造化ペイロード(payload JSONB)を必須項目として含める。
4. 監査ログは追記専用とする。レコード更新と削除はバッチ的な期限超過削除のみを許可する。
5. 監査ログには改ざん防止のため、各レコードに直前レコードのprev_hashとレコード自身のentry_hashを保持するハッシュチェーンを実装する。正規化はRFC 8785 JSON Canonicalization Scheme(JCS)に従い、Unicode正規化はNFCを適用する。数値は整数とJSONNumber表現を区別せずJCS規約に従う。ハッシュ対象はpayloadだけでなくレコード全体の改ざん検出可能な全フィールドとする。正規化対象に含めるキーは以下の固定集合とする(辞書順、JCS規則に従う): `id`、`seq`、`actor_kind`、`user_id`、`guest_session_id`、`target_kind`、`target_id`、`action`、`created_at`(マイクロ秒精度のISO 8601文字列、UTC)、`ip_address_hash`(16進文字列)、`user_agent`、`payload`、`salt_version`。これにより、actor、action、target、timestamp、IP hashを含む任意フィールドの改ざんを検出する。entry_hashの計算式は `entry_hash = HMAC-SHA-256(key=AUDIT_HASH_SALT_v{salt_version}, message=prev_hash ‖ canonicalize_jcs(record_canonical))` とし、`record_canonical` は上記キー集合のみで構成したJSONオブジェクトである。salt_versionレコード列に使用したソルトバージョンを記録する。最初のレコード(genesis)のprev_hashは固定値 `0000...0000` (256bit全ゼロ)とする。日次でチェーンを検査し、不整合を検出した場合の運用は以下のとおりとする。

| 段階     | 担当                 | 対応                                                                                               |
| -------- | -------------------- | -------------------------------------------------------------------------------------------------- |
| 検出     | 自動バッチ           | 運用管理者と情報セキュリティ担当へアラート発火、検出時点以降の改ざん疑いレコードを別テーブルへ隔離 |
| 一次調査 | 情報セキュリティ担当 | 不整合の発生時点とレコード範囲を特定、原因(障害かインシデントか)を判定                             |
| リカバリ | 運用管理者           | 直近のバックアップから当該レコード範囲を復元、復元範囲のentry_hashを再計算してチェーンを再構築     |
| 切替     | 情報セキュリティ担当 | インシデント判定の場合はAUDIT_HASH_SALTをローテーション、関係者通報                                |

AUDIT_HASH_SALTローテーションの整合性は以下で確保する。

- 各監査ログレコードに `salt_version` カラム(SMALLINT)を追加し、ハッシュ計算に用いたソルトバージョンを記録する。
- ローテーション後は新ソルト(salt_version+1)で新規レコードを記録する。旧ソルトは、それを用いて記録された監査ログがすべて第2項の保持期間(1年)を満了するまで秘密管理基盤(AWS Secrets Manager相当または `secret_versions` テーブル)で保持する。各ソルトの保持期間は「当該salt_versionで追記された最後の監査ログの `created_at`(=`secret_versions.last_used_at`)+1年+30日のグレース期間」を下限とする。発行日基準ではなく最終使用日基準とすることで、四半期ごとの能動ローテーションでも旧ソルトで追記された最後のレコードから保持期間中はハッシュ検証が常に可能となる。
- ソルト破棄は「該当 `salt_version` の最も新しいレコードの `created_at` から1年+30日経過した日次バッチ」が条件を満たしたソルトのみを削除する。破棄前後の検証可能性は、当該ソルトで記録された全レコードが既に保持期間満了で物理削除済みであることを前提とし、削除順序はバッチで保証する(レコード削除→当該salt_versionの参照ゼロ確認→ソルト破棄、の順)。
- 通常運用ではソルトを四半期ごとに能動ローテーションし、過去4世代を上記の保持期間で並行保持する。インシデント発生時の緊急ローテーションは追加世代として扱い、同じ保持規則に従う。

6. 監査ログ閲覧の権限は以下の4区分とし、FR-13第2項と完全同一の正本とする(本第6項とFR-13第2項のいずれを正本としても同じ結論となる)。(a)一般利用者(認証済み)は自身が所有者である文書の監査ログのみ閲覧可。(b)運用管理者(`operations_admin`)は全件閲覧可。(c)情報セキュリティ担当(`security_admin`)は全件閲覧可。(d)CS担当(`cs_admin`)は「依頼元の利用者ID」または「問い合わせ対象の文書ID」のいずれかをクエリパラメータで指定した場合に限り、当該範囲のみ閲覧可で問い合わせIDの記録を必須とする(任意全文検索や全件閲覧は許可しない)。
7. 監査ログのエクスポートをJSON Linesで提供する。
8. 認証されていない利用者と認可されていない利用者の操作試行(401、403を返したリクエスト)も監査ログに記録する。

### 4.6 拡張性

ID NFR-06。拡張性要件は以下のとおりである。

### 4.7 運用容易性

ID NFR-07。運用容易性要件は以下のとおりである。

1. バックアップは、PostgreSQLの論理バックアップを日次で取得する。
2. オブジェクトストレージのバックアップはバージョニング機能を有効化する。
3. アプリケーションログ、アクセスログ、監査ログ、ShareDB操作ログを集約する。
4. ヘルスチェックエンドポイントを提供し、外形監視に対応する。
5. デプロイは、Dockerコンテナ単位で扱い、コンテナイメージの差し替えで更新する。
6. 災害復旧(DR)のRPO(Recovery Point Objective)は24時間、RTO(Recovery Time Objective)は4時間を上限とする。本数値は要求分析書 第6章 第2項およびADR-0020と整合する。日次の論理バックアップで上記RPOを達成し、Helmチャートからのコールドスタンバイ環境への切替手順で上記RTOを達成する。年1回のDR訓練で本目標の充足を検証し、結果を監査ログ(action=`dr_drill_result`、actor_kind=system)へ記録する。

### 4.8 アクセシビリティ

ID NFR-08。アクセシビリティ要件は以下のとおりである。

1. 主要操作はキーボードのみで完結する。
2. 配色は背景と前景のコントラスト比4.5対1以上を保つ。
3. スクリーンリーダー対応はWCAG 2.1のレベルAA相当を目標とする。
4. プレゼンスのキャレット表示は、色覚多様性に配慮し、固有色だけでなく利用者名のラベルを併記する。

### 4.9 国際化

ID NFR-09。国際化要件は以下のとおりである。

1. UIは日本語と英語の2言語に対応する。
2. 文書本文は任意の文字種を受け付ける。Unicodeの全範囲を扱う。
3. IME入力中の確定前テキストは、OT操作として送信しない。確定時点でまとめて1操作として送信する。

### 4.10 品質ゲート(テスト戦略)

ID NFR-10。CIパイプライン上で必達とする品質ゲートは以下のとおりである。いずれの失敗もデプロイ禁止とする。

1. gitleaksにより、コミットに秘密情報が含まれていないことを検査する。
2. Vite+のフォーマッタ(素のViteではなく、VoidZeroが提供するVite+のフォーマッタを指す)を全変更ファイルに適用し、差分がないことを確認する。
3. Vite+のlinterを実行し、警告とエラーの双方をゼロにする。
4. Vite+標準のテストランナーによるユニットテストを実行し、ステートメントカバレッジ80パーセント以上を達成する。テストはAAA(Arrange-Act-Assert)パターンに従い、各テストの `Assert` 句が当該機能の意味ある振る舞い(入出力契約、副作用、エラーパス、不変条件)を検証する内容であることを必須とする。形だけのカバレッジ達成(例えばコードを呼び出すだけで結果を検証しないテスト)は禁ずる。カバレッジ閾値とAAA厳密性の双方を必達とする。カバレッジ閾値は必達とする。
5. PlaywrightによるE2Eテストを実行し、後述のUX主要導線をすべて成功させる。
6. マイグレーションSQLの構文と適用順序を `pnpm migrate:dry`(ADR-0025およびADR-0009で定めるnode-pg-migrate実行)で検証する。構文エラーまたは未適用順序の検出時はブロッキングとする。本ゲートはgitleaks成功後に直列実行し、後続のフォーマッタとlinterとテストを遮断する(本MVPでは「4種類の必達+補助1種類(migrate-dryrun)」の合計5種類をブロッキング条件として扱う)。

E2EテストでカバーするUX主要導線は以下のとおりである。

1. 利用者登録、ログイン、ログアウトの一連の動線。
2. 新規文書の作成と表示。
3. 文書本文への入力と装飾の適用と保存の確認。
4. 別のブラウザセッションで同一文書を開き、同時編集の反映を確認する動線。
5. 画像のアップロードと表示。
6. コメントの追加、返信、解決。
7. 提案モードの切り替え、提案の追加、受諾、却下。
8. 版履歴の保存と復元と差分閲覧。
9. 共有リンクの発行、共有先での閲覧、共有リンクの失効。
10. 文書のエクスポート(PDF、DOCX、Markdownの3形式すべて)。
11. ネットワーク切断時のオフライン閲覧と、復帰時の権限/ETag再検証およびキャッシュ破棄(FR-09に従い切断中の編集は対象外であり、本導線は閲覧のみ。復帰時の検証で権限剥奪または文書削除が検出された場合にローカルキャッシュが正しく破棄されることを必達とする)。

## 5. 外部インターフェース要件

### 5.1 利用者インターフェース

利用者インターフェースは、近代的なブラウザで動作するシングルページアプリケーションとして提供する。サポート対象は、Chrome、Firefox、Safari、Edgeの各最新版および1つ前の版とする。画面の最小幅は360ピクセル、推奨幅は1280ピクセル以上とする。

### 5.2 通信インターフェース

クライアントとサーバー間の通信は、認証とリソース取得にHTTPS、リアルタイム同期にWebSocket (wss)を用いる。WebSocketのプロトコルは、ShareDBが規定するJSON-RPC様のメッセージ形式に従う。

### 5.3 外部システム連携

MVPでは外部システムとの連携を以下の3点に限定する。

1. メール送信(SMTP)。
2. 画像保存(S3互換APIによるオブジェクトストレージ)。
3. Have I Been Pwned(以下HIBP)API。NFR-04第4項のパスワード漏えい辞書照合に用いる。k-Anonymity形式で、SHA-1ハッシュ先頭5文字のみを `https://api.pwnedpasswords.com/range/<prefix>` へGET送信し、完全なパスワードおよびメールアドレスは外部へ送出しない。HIBP APIが応答不能または5xxを連続して返した場合は、ユーザー登録とパスワード変更を「漏えい辞書照合を実施できなかった」として拒否せず、ローカルの長さ・文字種要件のみで暫定許可し、当該事象を監査ログ(actor_kind=system、action=`hibp_unavailable`)に記録する。タイムアウトは2秒とし、リトライは1回までとする。プライバシー説明(HIBPへのハッシュ送信、送出範囲、ローテーション基準)は登録画面とパスワード変更画面に明示する。

OIDCなどの外部認証連携は将来拡張で扱う。

## 7. 制約条件

本システムの制約条件は以下のとおりである。

1. バックエンドはNode.jsとTypeScriptで実装する。
2. リアルタイム同期はShareDBを採用する。
3. 永続化はPostgreSQL (sharedb-postgresアダプタ)を採用する。
4. 複数アプリケーションサーバー間のOT同期はRedis Pub/Subを採用する。
5. 画像保存はS3互換オブジェクトストレージ(MinIO)を採用する。
6. フロントエンドのエディタはQuill.jsを採用する。
7. フロントエンドのビルドと開発、フォーマッタ、linter、ユニットテスト、E2Eテスト基盤はVite+のエコシステムを採用する。
8. CIパイプライン上でgitleaks、Vite+のフォーマッタとlinter、Vite+標準のテストランナー(カバレッジ80パーセント以上)、Playwright (UX主要導線全件成功)の4つの品質ゲートと、補助の `pnpm migrate:dry`(マイグレーションSQL構文と適用順序の検証)の合計5つをブロッキング条件として扱う(NFR-10第6項およびADR-0009に従う)。

## 8. 既定値と確認事項

本章では、要求分析書 第11章の未解決事項を含めたすべての検討事項を、MECE(Mutually Exclusive, Collectively Exhaustive)に分解し、根拠付きで確定値として整理する。ADR側のステータスをAcceptedへ昇格できる前提を本書の確定で揃えるためである。

確定値の決定根拠は、本MVPで達成すべきビジネスゴール(要求分析書 第3章)、想定する利用規模(NFR-01)、開発体制(少人数)、運用容易性方針の四つを評価軸とした。各値は次の3条件を満たすことを必須とした。

1. 必要性: 本MVPの主要ユースケース(認証済み利用者間の同時編集、社外共有、装飾付き出力)を成立させるうえで欠かせない。
2. 十分性: 想定する規模で運用が破綻しない水準を満たす。
3. 排他性: 他の確定値との二重決定や重複を含まない。

## 9. 自己レビューと是正記録

本要件定義書を作成したのち、五つの観点から自己レビューを行い、検出した問題点と是正対応を以下に記録した。

### 9.1 レビュー観点1: 要求分析書とのトレーサビリティ

問題点として、初版では要求分析書で挙げた17カテゴリの機能要求と、本書の機能要件のカテゴリ対応関係が示されていなかった。是正として、機能要件にFR-01からFR-13までの一意のIDを付与し、第3章序文のマップ表で要求分析書17カテゴリと1対1または1対多で対応するよう整理した。MVP対象外のカテゴリ(検索、高度な通知、オフライン編集、外部認証連携、ペネトレ自動化)は§3.14将来拡張へ振り分けた。

### 9.2 レビュー観点2: 品質ゲートの取り込み漏れ

問題点として、初版では品質ゲートを非機能要件として明示しておらず、PlaywrightのUX導線の具体内容が不在であった。是正として、第4.10節を新設し、CIの必達条件と、E2EでカバーするUX主要導線を11件列挙した。

### 9.3 レビュー観点3: OT変換規則の記述不足

問題点として、初版では挿入操作の規則のみ言及していたが、削除と属性変更の規則が抜けていた。是正として、第3.1節に、削除と属性変更はQuill.jsのDelta transform関数に準拠する旨を追記した。

### 9.4 レビュー観点4: コメントと提案モードのデータ要件の不整合

問題点として、初版ではコメントと提案のデータ構造が部分的にしか定義されておらず、本文OTとの分離が明示されていなかった。是正として、第3.4節と第3.5節にアンカー追従と本文OT非干渉の方針を明記し、第6章のデータ要件にコメント(Comment)とコメント返信(CommentReply)、提案(Suggestion)を整理した。提案返信は本MVPの範囲外とする。

### 9.5 レビュー観点5: MECE確定値の整理

問題点として、初版では決定値が「確定」「既定値」「未承認」に分かれて曖昧であり、ADRがProposedで止まる原因となっていた。是正として、第8章でMECE分析を行い、データベース、認証方式、文書履歴、オフライン、エクスポート、国際化、アクセシビリティ、監査ログ、画像保存、提案モード粒度、表機能MVP範囲、共有リンク種別、共有リンク有効期限上限の13項目を根拠付きで確定値として整理した。承認後はすべてのADRがAcceptedとして成立する。

### 9.6 残存する確認事項

要件定義書承認後、第8章の確定値が前提となり、基本設計書と詳細実装指示書を進める。残存する未解決事項は存在しない。
