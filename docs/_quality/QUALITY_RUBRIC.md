# QUALITY_RUBRICバージョン1.5

本書は本プロジェクトの全技術文書に適用する品質ルーブリックである。レビュアー(人間およびAgent)は本ルーブリックの判定項目に該当する違反を正当な指摘として扱う。

ただし、本ルーブリック未収載でも以下の「技術的妥当性に関する特例カテゴリ」に該当する指摘は無条件に受理する。本MVPの存続を脅かす技術破綻を、ルーブリック整備の遅延で見落とさないためである。技術的妥当性はルーブリック準拠より上位に置く。

| カテゴリ           | 例                                                                           | 重要度   |
| ------------------ | ---------------------------------------------------------------------------- | -------- |
| データ整合性破綻   | 永続化矛盾、データ消失、トランザクション境界の破れ、復元不能                 | Critical |
| セキュリティ破綻   | 認証バイパス、認可漏れ、改ざん検出不能、秘密漏えい、暗号方式の不適切な選定   | Critical |
| サービス不能リスク | デッドロック、無制限リソース消費、再現性のない障害、非冪等な再試行           | Critical |
| MVP境界からの逸脱  | 要件側の境界と設計側の実装が衝突する、要件で対象外としたものが設計に紛れ込む | Critical |
| 文書間の決定的矛盾 | 同一事項について要件、設計、ADRが異なる結論を持つ                            | High     |
| 仕様破綻           | API契約と実装契約の不一致、状態機械の到達不能状態、不変条件の破れ            | High     |

特例カテゴリの指摘を受理した場合は、本ルーブリックに対応観点を起票し、次版で正式項目として取り込む(本ルーブリック第9章に従う)。これにより、ルーブリックは時間とともに完備に近づき、特例受理の発動回数は減少していく設計とする。

本ルーブリックは `STYLE_GUIDE.md` (バージョン1.0以降)を前提とする。

## 7. 重要度の定義

| 重要度   | 判定                                           |
| -------- | ---------------------------------------------- |
| Critical | 文書の主要主張を破壊する。即時修正必須         |
| High     | 不整合、誤情報、トレーサビリティ破綻。修正必須 |
| Medium   | 表現改善余地、規約の追加余地。次版で対応可     |
| Low      | 表記の好み。規約に取り込まないなら無視可       |

## 8. レビュー受理規約

レビュアー(人間およびAgent)は以下を守る。

1. 本ルーブリックの判定項目に該当する違反を指摘する。加えて、本ルーブリック序文の「技術的妥当性に関する特例カテゴリ」に該当する指摘は、ルーブリック未収載でも受理する(技術的妥当性 > 規約準拠)
2. 通常指摘には該当する判定項目の番号(例「2.6添付OT図取り込み」)を引用する。特例カテゴリの指摘には序文表のカテゴリ名を引用する
3. 指摘の重要度は本ルーブリック第7章の定義に厳密に従う。特例カテゴリは表の重要度欄を用いる
4. 主観的好み(語感、文体の流麗さ)は指摘の対象外とする。これらを規約に取り込みたい場合はSTYLE_GUIDEへの改訂提案として別途起票する
5. 特例カテゴリで受理した指摘は、本ルーブリック第9章の改訂手順で正式項目に取り込む。特例受理が発生しないことを最終ゴールとする
