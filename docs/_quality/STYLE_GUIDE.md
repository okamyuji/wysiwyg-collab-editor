# STYLE_GUIDEバージョン1.2

本書は本プロジェクトの全技術文書(要求分析書、要件定義書、基本設計書、ADR、付随README)に適用する文体と用語と表記の規約である。Agentまたはレビュアーは本規約に照合した違反を指摘する。主観的指摘は受理しないが、`QUALITY_RUBRIC.md` 序文の「技術的妥当性に関する特例カテゴリ」に該当する技術破綻指摘は規約未収載でも受理する(規約準拠より技術的妥当性を上位に置く)。本規約の変更は新規バージョンの起票で行い、過去バージョンとの差分を履歴に残す。

## 1. 適用範囲

本規約は `docs/` 配下のすべての `.md` ファイルに適用する。ただし、規約自体を記述するファイル(`docs/_quality/` 配下のすべての `.md`)は自己違反検査の対象外とする。これは規約表が禁止語と用語例を直接列挙する必要があるためであり、これらの記述を自己違反と判定するとlintが収束不能となるためである。lintの実装(`tools/lint_docs.py`)も `_quality` ディレクトリを一律スキップする。

また、本規約は以下の文脈にも適用しない。

| 適用除外                                                                  | 理由                                           |
| ------------------------------------------------------------------------- | ---------------------------------------------- |
| コードブロック内                                                          | コードとして読まれるべきためである             |
| URLとファイルパス                                                         | 機械可読の識別子であり改変不能                 |
| 英数字のみの引用                                                          | 引用元の正確性を優先                           |
| 表のセル内の英語固有名詞                                                  | 製品名やAPI名の改変は誤情報となる              |
| 規約改訂候補や禁止語列を直接列挙するMarkdown表(規約文書 `_quality/` 配下) | 規約表自体は規約の対象ではなく規約の本体である |

## 2. 言語と表現

### 2.1 自然な単文の日本語

主述が一対一の単文を基本とする。読点で連結する複文は1文あたり最大3つまでとする。

### 2.2 禁止表現

以下を禁止表現リストとして固定する。Agentはこのリストに含まれる語の検出のみを指摘してよい。

| 禁止表現                                 | 理由       | 代替表現の例                        |
| ---------------------------------------- | ---------- | ----------------------------------- |
| 「業界での実績ある標準」                 | 主観的賞賛 | 「運用実績が確認できる」            |
| 「最小ライン」                           | AI典型語彙 | 「水準を満たす」「下限である」      |
| 「最小集合」                             | AI典型語彙 | 「網羅する2種類」「過不足ない範囲」 |
| 「現行安定版」                           | 抽象すぎる | 具体バージョン番号を併記            |
| 「事実上の標準」                         | 抽象すぎる | 「業界での採用事例が確認できる」    |
| 「直感的」                               | AI典型語彙 | 「学習コストが低い」                |
| 「シンプル」「クリーン」「堅牢」「柔軟」 | AI典型語彙 | 具体特性で記述                      |
| 「網羅的に」                             | 自己賞賛   | 「以下の項目を列挙する」            |
| 「漏れがないように」                     | 自己賞賛   | 削除する                            |
| 「逸脱を許さない」                       | 過剰強調   | 「必須とする」                      |

### 2.3 表記ルール

| 項目         | ルール                                                                                                                                                                             |
| ------------ | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| 文末         | 「である」体で統一する。本文の文末を「:」で終えることを禁ずる                                                                                                                      |
| 強調         | Markdownの `**` による太字強調を禁ずる。重要事項は表または箇条書きで強調する                                                                                                       |
| 体言止め     | 本文の文末で体言止めを禁ずる。見出しは体言止めまたは動詞文の自由を許容する                                                                                                         |
| 絵文字       | 禁止                                                                                                                                                                               |
| 一人称       | 「私」「私たち」を禁ずる。書き手主体を示す場合は「本書」「本ADR」とする                                                                                                            |
| 半角スペース | 日本語と英数字の境界での半角スペースを禁ずる。例「OTを採用」(正)。例外として、H1見出しのADR識別子直後(`# ADR-NNNN `形式)とMarkdownリストおよび見出しの先頭インデント空白は許容する |

### 2.5 数値と単位

数値はアラビア数字を用いる。単位は日本語または記号のいずれかで統一し、本書では以下を採用する。

| 単位   | 採用表記                                          |
| ------ | ------------------------------------------------- |
| 時間   | ミリ秒、秒、分、時間、日                          |
| サイズ | バイト、KB、MB、GB(本MVPでは「メガバイト」を維持) |
| 割合   | パーセント                                        |

### 2.6 引用と参照形式

文書内引用は「第N章」「第N.N節」「FR-NN」「NFR-NN」「ADR-NNNN」の形式で統一する。

ファイルパス引用は ` でクォートし、リポジトリルートからの相対パスとする。

URL引用は `[テキスト](URL)` 形式とする。

## 3. 構造規約

### 3.1 見出しレベル

| レベル | 用途                       |
| ------ | -------------------------- |
| #      | 文書タイトル(1ファイル1個) |
| ##     | 章                         |
| ###    | 節                         |
| ####   | 小節(必要な場合のみ)       |

### 3.2 箇条書きと番号付きリスト

順序が意味を持つ場合は番号付きリスト、順序が意味を持たない場合は箇条書きを用いる。番号付きリストは「1.」から始め、連番を維持する。

### 3.3 表

3列以上の表は必ずヘッダ行と区切り行を含める。表のセル内では文末「である」を省略してよい(セル内は体言止め許容)。

## 4. ADR固有規約

ADRは以下の8節構成を必須とする。

1. ステータス(Accepted、Proposed、Deprecated、Supersededのいずれかと日付)
2. 文脈
3. 検討した選択肢(3案以上、各案の長所と短所)
4. 決定
5. 根拠(箇条書き、3項以上)
6. 影響(ポジティブ、ネガティブ、リスクと緩和策の3小節)
7. 関連ADR
8. 自己レビュー記録(3観点以上)

ADRファイル名は `ADR-NNNN-スラッグ.md` の形式とし、`NNNN` は4桁の連番、`スラッグ` は日本語または英語の簡潔なタイトルとする。

H1タイトルは `# ADR-NNNNタイトル` の形式とし、NNNNとタイトルの間に半角スペースを1個入れる(例: `# ADR-0001同時編集の整合性方式としてOTを採用する`)。これは本規約2.3「日本語と英数字の境界での半角スペース禁止」のADR識別子直後の例外として明示している箇所である。

## 5. トレーサビリティ規約

要件、設計、ADRは相互参照で連結する。

| 関係              | 表記方法                      |
| ----------------- | ----------------------------- |
| 要求分析→要件定義 | 第13章後工程確定状況表        |
| 要件定義→ADR      | 第8章ADR充足表                |
| ADR→要件定義      | 関連ADR節と影響節で言及       |
| 基本設計→要件定義 | 各機能フローの冒頭にFR-NN参照 |
| 基本設計→ADR      | 第11.5節ADRトレーサビリティ表 |

## 6. 規約の改訂

本規約の改訂は以下の手順で行う。

1. 改訂提案を本ファイルの末尾に「改訂候補」として記録する
2. ユーザー承認をもって本ファイルのバージョン番号を上げ、改訂内容を反映する
3. 既存文書はバージョン2.0以降の規約に対し追従改修を行う
