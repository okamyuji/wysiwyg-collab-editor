# ADR-0013 エクスポート形式をPDF、DOCX、Markdownの3種類に固定する

## ステータス

Accepted (2026-06-12)

## 文脈

要件定義書FR-10により、文書を外部形式へエクスポートする機能を提供する。MVP段階の対象形式は、利用者指定によりPDF、DOCX、Markdownの3種類である。それぞれの形式に対し、どのライブラリで生成するか、サーバー側で実行するかクライアント側で実行するか、生成物の保存先と配布方法を確定させる必要がある。

## 検討した選択肢

### 選択肢A: 全形式をサーバー側で生成し、署名付きURLで配布する

PDF、DOCX、Markdownの生成をすべてサーバー側で実行する。生成物はオブジェクトストレージ(ADR-0007)へ保存し、ブラウザは署名付きURLでダウンロードする。

長所として、ブラウザ側の処理負荷を抑え、ライブラリ依存をサーバー側に集約できる点、フォントや表のレンダリングが安定する点が挙げられる。短所として、サーバー側のCPU負荷が増え、長時間処理がアプリケーションサーバーのスレッドを占有する可能性がある点が課題である。

### 選択肢B: 全形式をクライアント側で生成する

Quill.jsのDeltaからクライアント側で各形式へ変換する。

長所として、サーバー負荷を増やさずに済む点が挙げられる。短所として、PDF生成のフォント埋め込みや表のレイアウトが、ブラウザ環境では安定しない点と、巨大文書の変換でブラウザがフリーズする可能性がある点が課題である。

### 選択肢C: 形式別に最適な側で生成する(ハイブリッド)

PDFはサーバー(Playwright)、DOCXとMarkdownはクライアント側で生成する案である。

長所として、形式ごとに最適化できる点が挙げられる。短所として、生成パスが複数になり、テストとデバッグが複雑化する点が課題である。

## 決定

選択肢Aを採用する。形式別の生成方法は以下のとおりである。

1. PDFはPlaywrightでヘッドレスChromiumを起動し、文書のレンダリング結果をPDF化する。
2. DOCXは `docx` ライブラリでDelta構造を変換する。
3. MarkdownはunifiedとremarkベースでDelta構造をCommonMarkとGitHub Flavored Markdownの範囲に変換する。Markdownで表現できない装飾は警告を表示し、近い表現に置き換える。
4. 生成物はオブジェクトストレージのdocs-exportsバケットへ保存し、24時間有効の署名付きURLを返す。MinIOのライフサイクルポリシーで24時間後に自動削除する。

エクスポート処理は、アプリケーションサーバーとは独立した `export-worker` 専用コンテナで動作する(ADR-0028と整合)。本文編集のWebSocket応答性をプロセス境界で物理的に分離するためであり、Playwright依存OS(debian-slim)と共通サーバーOS(Distroless)が異なる事実が別コンテナ採用の決定的根拠である。状態機械とコンテナ分離は必須とする。具体仕様は基本設計書 第5.10節の正式記述に従う。要点は以下のとおりである。

## 根拠

採用根拠は以下のとおりである。

1. PDFのフォントと表のレンダリングはサーバー側のChromiumが最も信頼できる。
2. DOCXとMarkdownのライブラリはNode.js環境で安定動作するため、サーバー側で実行する利点が大きい。
3. 全形式をサーバー側で生成することで、生成パスが1系統に統一され、保守とテストが単純化される。
4. オブジェクトストレージへ保存することで、巨大ファイルでもアプリケーションサーバーのメモリを圧迫しない。
5. 署名付きURLによる配布は、ADR-0007と整合する。

## 影響

### ポジティブな影響

1. 形式ごとの実装が独立しており、それぞれの形式の改善が他に影響しない。
2. 生成物がオブジェクトストレージに保存されるため、エクスポートのダウンロードがアプリケーションサーバーを経由しない。
3. PDF生成にChromiumを使うことで、Webと同等のレイアウトと装飾が再現できる。

### ネガティブな影響

1. Playwrightが必要とするChromiumバイナリと依存パッケージのデプロイサイズが大きくなる。
2. PDF生成がCPU負荷の大きい処理であり、ワーカープロセス分離を将来検討する必要がある。
3. DOCXとMarkdownの形式制約により、装飾の一部が完全には保持されない場面がある。

### リスクと緩和策

1. リスクとして、巨大文書のPDF生成でメモリ不足を起こす可能性がある。緩和策として、文書サイズに上限(目安1メガバイトのテキスト相当、要件定義書NFR-01)を設け、超過時には警告を表示する。Playwrightは1リクエストごとに新規プロセスを起動し、終了時に確実にクローズする。
2. リスクとして、Markdownで表現できない装飾(背景色、複雑な表など)が無断で消失すると利用者が混乱する。緩和策として、変換時に警告を画面表示し、変換結果に欠落要素のサマリーをコメントとして含める。
3. リスクとして、DOCX生成中にエラーが起きるとアプリケーションサーバーのスレッドが長時間ブロックされる。緩和策として、タイムアウト(60秒)を設け、超過時はジョブを失敗として処理する。
4. リスクとして、エクスポート機能の利用が集中するとアプリケーションサーバーの応答性が低下する。緩和策として、利用者単位のエクスポート頻度に制限を設け、超過時は待機メッセージを表示する。規模拡大時には専用ワーカープロセスとジョブキューに分離する。

## 自己レビュー記録

### レビュー観点1: 形式別生成方針の明確化

問題点として、初版では3形式の生成手段が一覧化されておらず、ハイブリッド案との違いが曖昧だった。是正として、決定の項で形式別に生成方法を列挙し、サーバー一本化の理由を根拠で示した。

### レビュー観点2: メモリとCPU負荷への配慮

問題点として、初版ではPlaywrightのリソース消費への対処が触れられていなかった。是正として、リクエスト単位のプロセス起動とタイムアウト、文書サイズ上限を緩和策に追記した。

### レビュー観点3: Markdownの欠落要素警告

問題点として、初版ではMarkdownで表現できない装飾の扱いが「警告を表示」とだけ書かれていた。是正として、変換結果コメントへの欠落サマリー追加を緩和策に明示した。

### レビュー観点4: ワーカー分離への将来余地

問題点として、初版ではエクスポートを将来ジョブキューへ切り出す方針が示されていなかった。是正として、決定とリスク緩和策に明示した。
