# ADR-0009 CI上の品質ゲートをgitleaks、migrate-dryrun、Vite+フォーマッタとlinter、Vite+標準のテストランナー、Playwrightの5種類に固定する

## ステータス

Accepted (2026-06-02)

本ADRのうちユニットテストのカバレッジ閾値の強制(ジョブ2cと2d)の適用はフェーズ2以降である(2026-07-05のユーザー決定)。フェーズ1(現行実装水準のMVP)の適用範囲はADR-0029を参照。

## 文脈

要件定義書NFR-10により、CIパイプライン上で実行する品質ゲートは必達であり、いずれの失敗もデプロイ禁止として扱う。利用者から明示された方針は以下の4種類である。

1. gitleaksによる秘密情報スキャン
2. Vite+(素のViteではなく、VoidZeroが提供するVite+)のフォーマッタとlinter
3. Vite+標準のテストランナーによるユニットテスト(ステートメントカバレッジ80パーセント以上)
4. PlaywrightによるE2Eテスト(UX主要導線を全件成功)

これらをCIにどう組み込み、どの順序で実行し、どの粒度で失敗を扱うかを確定させる必要がある。

## 検討した選択肢

### 選択肢A: 利用者指定の4種類のゲートをCIに直列と並列を組み合わせて配置する

gitleaksを最初に実行し、それが成功した場合のみ後続を実行する。フォーマッタ、linter、テストは並列実行し、E2Eは前段全てが成功した場合のみ実行する。

長所として、利用者方針に厳密に沿い、無駄なリソース消費を抑えつつ、フィードバックが速い点が挙げられる。

### 選択肢B: 全ゲートを並列に実行する

全ゲートを最初から並列で実行し、いずれかが失敗した時点でジョブを終了する。

長所として、最短のフィードバック時間を得られる点が挙げられる。短所として、gitleaksの失敗時にも他ジョブが実行され、CIリソースが無駄になる点がある。

### 選択肢C: 全ゲートを直列に実行する

ゲートを順番に実行し、前のゲートが成功した場合にのみ次を実行する。

長所として、設定が単純である点が挙げられる。短所として、フィードバックが遅く、開発速度に影響する点が課題である。

## 決定

選択肢Aの組み合わせ実行を採用する。Vite+ CLI(ADR-0008)に揃えた具体的なジョブ構成は以下のとおりである。

| 順序 | ジョブ名         | 実行コマンド                                                                                                                                                                   | 内容                                                                                                                                                                                                                                                                |
| ---- | ---------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| 1    | scan-secrets     | `gitleaks detect --source . --redact`                                                                                                                                          | リポジトリ全体の秘密情報スキャン                                                                                                                                                                                                                                    |
| 1b   | migrate-dryrun   | `pnpm migrate:dry`                                                                                                                                                             | マイグレーションSQLの構文と適用順序を検証する補助チェック(ADR-0025で定めるnode-pg-migrate実行)。本ジョブは失敗時にブロッキング(ジョブ2以降を中止)する5番目の必達ゲートとして扱う。本MVPでは「4種類固定」の表現を「4種類の必達+補助1種類(migrate-dryrun)」へ拡張する |
| 2a   | check-web        | `cd apps/web && vp check`                                                                                                                                                      | フロントエンドのフォーマッタとlinterを一括実行                                                                                                                                                                                                                      |
| 2b   | check-server     | `cd apps/server && vp check`                                                                                                                                                   | サーバーのフォーマッタとlinterを一括実行                                                                                                                                                                                                                            |
| 2c   | test-unit-web    | `cd apps/web && vp test --coverage --coverage.thresholds.lines=80 --coverage.thresholds.statements=80 --coverage.thresholds.branches=70 --coverage.thresholds.functions=80`    | フロントエンドのユニットテスト、カバレッジ閾値を `vitest.config.ts` と本コマンドの双方で強制                                                                                                                                                                        |
| 2d   | test-unit-server | `cd apps/server && vp test --coverage --coverage.thresholds.lines=80 --coverage.thresholds.statements=80 --coverage.thresholds.branches=70 --coverage.thresholds.functions=80` | サーバーのユニットテスト、同上のカバレッジ強制                                                                                                                                                                                                                      |
| 2e   | build-web        | `cd apps/web && vp build`                                                                                                                                                      | 本番ビルドが成立することの確認(E2Eに必要)                                                                                                                                                                                                                           |
| 2f   | build-server     | `cd apps/server && pnpm tsc --noEmit`                                                                                                                                          | サーバーの型検査                                                                                                                                                                                                                                                    |
| 3    | test-e2e         | `cd e2e && pnpm exec playwright test --reporter=html,junit,json --output=test-results`                                                                                         | UX主要導線のE2Eテスト。JSON/JUnitレポータをCI集約に必須化し、HTMLレポートと失敗時のスクリーンショット、ビデオ、トレースをアーティファクトとして保存する                                                                                                             |
| 4    | summary          | -                                                                                                                                                                              | 全ジョブの結果を集約してデプロイトリガを発行                                                                                                                                                                                                                        |

ジョブ間の依存関係は以下のとおりである。

1. ジョブ1 (scan-secrets)は単独で実行する。失敗時は後続(ジョブ1b、ジョブ2、ジョブ3、ジョブ4)をすべて中止する。
   1b. ジョブ1b (migrate-dryrun)はジョブ1成功後に直列実行する。失敗時はブロッキングとし、ジョブ2以降をすべて中止する(マイグレーション構文エラーの早期検出)。
2. ジョブ2a〜2fはジョブ1bの成功後に並列実行する。いずれかが失敗した場合は他は続行しつつ、最終的にジョブ3とジョブ4は実行しない。
3. ジョブ3 (test-e2e)はジョブ1とジョブ2のすべてが成功した場合のみ実行する。PlaywrightのHTMLレポートと失敗時のスクリーンショット、ビデオ、トレースを成果物として保存する。
4. ジョブ4 (summary)はすべての前段ジョブの結果を集約する。すべて成功時のみ本番デプロイをトリガする。

## 根拠

採用根拠は以下のとおりである。

1. gitleaksの失敗は秘密情報の混入を意味するため、後続ジョブの実行は無駄であるだけでなく、漏洩を拡大する可能性がある。最初に直列で配置する。
2. フォーマッタとlinterとユニットテストは互いに依存しないため、並列実行で開発者のフィードバック時間を短縮できる。
3. E2Eは実行時間が長く、リソース消費が大きいため、前段ジョブの成功を確認したうえで実行する設計が経済的である。
4. ユニットテストのカバレッジ閾値はVite+標準のテストランナーの `--coverage.thresholds.statements=80 --coverage.thresholds.lines=80 --coverage.thresholds.branches=70 --coverage.thresholds.functions=80` 相当で強制する。閾値は `vitest.config.ts` でも宣言し、CIコマンドとの二重定義により事故を防ぐ。
5. PlaywrightはCIランナー上でブラウザを起動するため、ジョブ単位での実行とし、HTMLレポートを成果物として残すことで失敗解析を容易にする。

## 影響

### ポジティブな影響

1. 秘密情報の混入が早期に検知され、漏洩リスクが抑制される。
2. フォーマッタとlinterとテストが並列実行されるため、開発者のフィードバックが速い。
3. E2Eが前段成功時のみ実行されるため、CIリソースの消費が無駄にならない。
4. デプロイ条件が明確であり、品質ゲートを通過した変更のみが本番へ流れる。

### ネガティブな影響

1. CI設定が4ジョブ以上に分かれ、初期構築の手間が増える。
2. E2Eが前段の完了を待つため、E2Eまでの所要時間は順次実行時の合計時間となる。
3. PlaywrightのCI環境(ブラウザバイナリ、依存システムパッケージ)の管理が必要になる。

### リスクと緩和策

1. リスクとして、ユニットテストのカバレッジ80パーセントを満たすために形だけのテストが書かれ、品質が下がる可能性がある。緩和策として、カバレッジ閾値はCI上でブロッキング条件として強制し、AAA(Arrange-Act-Assert)パターンに従わないテストはコードレビューで差し戻す慣行を整える。要件定義書NFR-10は「カバレッジを必達としつつ、AAAパターンで意味のある振る舞いを検証する」方針へ更新済みである。
2. リスクとして、E2Eテストが間欠的に失敗する(フレーキー)場合、CIが不安定化する。緩和策として、Playwrightのretries設定を1回に限定し、3回連続で同じテストがフレーキーになった場合はquarantineタグでスキップし、別途修正の専用Issueを起票する運用とする。
3. リスクとして、gitleaksの誤検知が頻発する場合、開発者がbypassフラグを乱用する可能性がある。緩和策として、`.gitleaks.toml` で誤検知パターンの除外を整備し、bypassの権限はリポジトリ管理者に限定する。
4. リスクとして、Vite+のフォーマッタとlinterの新版でルールが変わり、既存コードが大量に失敗する可能性がある。緩和策として、依存バージョンを固定し、更新時には専用ブランチで影響範囲を確認する手順を定める。

## 自己レビュー記録

### レビュー観点1: ジョブ間の依存関係の明示

問題点として、初版ではジョブの順序と並列性が曖昧だった。是正として、決定の項にジョブ単位の依存関係と並列実行可能性を具体的に整理した。

### レビュー観点2: カバレッジ閾値の運用

問題点として、初版ではカバレッジ80パーセントの強制方法が示されていなかった。是正として、Vite+標準のテストランナーのCLIオプションで強制する方針を根拠に明記した。

### レビュー観点3: E2Eフレーキー対策

問題点として、初版ではE2Eの間欠失敗への対処方針が触れられていなかった。是正として、retries回数、quarantineタグ、専用Issue起票の運用を緩和策に追加した。

### レビュー観点4: gitleaksのbypass運用

問題点として、初版ではgitleaks誤検知時の対処が未定義だった。是正として、`.gitleaks.toml` での除外パターン整備とbypass権限の管理を緩和策として追加した。
