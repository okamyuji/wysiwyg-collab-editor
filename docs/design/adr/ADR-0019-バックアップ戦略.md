# ADR-0019 バックアップ戦略

## ステータス

Accepted (2026-06-23)

## 文脈

PostgreSQL、MinIO、Redisの3層に対しバックアップ戦略を確定させる。

## 検討した選択肢

### 選択肢A: PostgreSQL日次論理+ MinIOバージョニング(本案)

復旧手順が単純

### 選択肢B: PostgreSQLストリーミングレプリカ+ MinIO増分バックアップ

RPOは短縮されるが運用負担が増加する

### 選択肢C: バックアップなし(MinIOのバージョニングのみ)

PostgreSQLの障害復旧ができないため不採用

## 決定

選択肢Aを採用する。具体仕様は要件定義書および基本設計書の該当節に置き、本ADRは決定の所在と根拠を保持する。

### 確定値

| 対象       | 方式                      | 頻度 | 世代                 |
| ---------- | ------------------------- | ---- | -------------------- |
| PostgreSQL | 論理バックアップ(pg_dump) | 日次 | 日次7、週次4、月次12 |
| MinIO      | バージョニング機能        | 連続 | バージョンごと無期限 |
| Redis      | バックアップ対象外        | -    | -                    |

## 根拠

採用根拠は以下のとおりである。

1. 日次論理バックアップでRPO 24時間を達成する(ADR-0020整合)。
2. MinIOの内蔵バージョニング機能は誤上書きと誤削除に対する保護を提供する。
3. Redisのセッションデータは失われても再ログインで復旧可能なため、バックアップ対象外とする。
4. バックアップは7日間世代を保持し、過去30日分は週次バックアップで保持する。

## 影響

### ポジティブな影響

1. 主要設計決定のMECEな整理に寄与する。
2. 要件定義書および基本設計書とのトレーサビリティが明確になる。
3. 当該分野の確定値を1箇所に集約することで、将来の差し替えADR起票時の起点が明確になる。

### ネガティブな影響

1. 詳細仕様を要件定義書と基本設計書に委ねるため、ADR単体での自己完結性は限定的である。
2. 関連文書の更新時には本ADRも見直す必要がある。

### リスクと緩和策

1. 日次バックアップ取得時の性能影響に対しては、業務時間外(深夜帯)に実行することで緩和する。
2. バックアップ世代数が運用コストに与える影響を継続的に評価する。
3. MinIOバージョニングのストレージ消費は四半期に1度監視する。

## 関連ADR

1. ADR-0004永続化基盤としてPostgreSQLに統一する
2. ADR-0007画像と添付ファイルの保存にS3互換オブジェクトストレージ(MinIO)を採用する
3. ADR-0020災害復旧の目標値

## 自己レビュー記録

### レビュー観点1 (ADR-0019): RPO達成

日次論理バックアップでRPO 24時間を達成する。

### レビュー観点2 (ADR-0019): 世代設計

7日+ 4週+ 12月の3層構成で運用する。

### レビュー観点3 (ADR-0019): リストア訓練

半年に1回の頻度で実際にリストアを行う訓練を実施する。
