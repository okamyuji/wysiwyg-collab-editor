# ADR-0028 デプロイ単位とコンテナ基盤

## ステータス

Accepted (2026-06-27)

## 文脈

デプロイ単位、コンテナランタイム、ベースイメージを確定させる。

## 検討した選択肢

### 選択肢A: Docker(OCI互換)、Distrolessまたはdebian-slim(本案)

長所: 業界標準のコンテナランタイムであり、CI/CD連携、レジストリ、運用ツールが揃っている。Distrolessで攻撃面を最小化しつつ、Playwright依存のエクスポートワーカーはdebian-slimで分離できる。
短所: イメージ二系統(Distrolessとdebian-slim)の管理が必要であり、ベースOSの脆弱性対応と更新を二系統で追う必要がある。

### 選択肢B: podman

長所: ルートレス実行が標準であり、Dockerデーモン不要でセキュリティ表面が小さい。Docker互換CLI。
短所: CI/CDのDockerプラグインとレジストリ連携で別途設定とテストが必要であり、社内ツーリングがDocker前提のため新規習熟コストが発生する。本MVP規模ではこのコストに見合わない。

### 選択肢C: Kubernetes(Helm chart)パッケージ単独

長所: Helmで一括デプロイができ、運用標準化が容易である。スケール時の移行コストが低い。
短所: 本MVPは小規模であり、Kubernetes運用知識と監視基盤を前提とする本選択肢は過剰投資となる。MVP第一リリース後のスケール期に再評価する価値がある。

### 選択肢D: VM (Packer image)

長所: OSとミドルウェアを丸ごとイメージ化でき、互換性問題が起きにくい。
短所: コンテナ化のメリット(軽量、起動高速、レイヤキャッシュ、デプロイ単位の細粒度)を失う。デプロイサイクルが分単位になり、本MVPのリリース速度目標に整合しない。

## 決定

選択肢Aを採用する。具体仕様は要件定義書および基本設計書の該当節に置き、本ADRは決定の所在と根拠を保持する。

### 確定値

| 項目                         | 値                                   |
| ---------------------------- | ------------------------------------ |
| ランタイム                   | Docker(OCI互換)                      |
| 共通サーバーイメージ         | Distrolessベース                     |
| エクスポートワーカーイメージ | debian-slimベース(Playwright依存)    |
| Alpine                       | 不採用(muslベースでPlaywright非互換) |
| デプロイ単位                 | Dockerコンテナ単位                   |
| CI/CD                        | GitHub Actions + Docker BuildKit     |

## 根拠

採用根拠は以下のとおりである。

1. Dockerは業界標準のコンテナランタイムである。
2. Distrolessは最小フットプリントでセキュリティが高い。
3. Playwrightを含むエクスポートワーカーはdebian-slim、その他はDistrolessを使い分ける。
4. AlpineはmuslベースでPlaywrightの動作に問題があるため不採用。

## 影響

### ポジティブな影響

1. 主要設計決定のMECEな整理に寄与する。
2. 要件定義書および基本設計書とのトレーサビリティが明確になる。
3. 当該分野の確定値を1箇所に集約することで、将来の差し替えADR起票時の起点が明確になる。

### ネガティブな影響

1. 詳細仕様を要件定義書と基本設計書に委ねるため、ADR単体での自己完結性は限定的である。
2. 関連文書の更新時には本ADRも見直す必要がある。

### リスクと緩和策

1. Distrolessはデバッグ用シェルがないため、トラブル時に専用デバッグイメージを別途準備する。
2. 複数ベースイメージの並行運用ではセキュリティパッチの追従が二重作業となるが、CIで自動的に最新イメージを取り込む。
3. 将来コンテナランタイムをpodmanへ移行する場合は差し替えADRを起票する。
