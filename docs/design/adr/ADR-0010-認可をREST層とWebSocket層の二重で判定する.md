# ADR-0010 認可判定をREST層とWebSocket層の二重で実施する

## ステータス

Accepted (2026-06-05)

## 文脈

本システムは、REST APIとWebSocket(ShareDB)の2系統でクライアントと通信する。文書アクセス権限の判定は両系統で同等に行う必要があり、片方の判定漏れは情報漏洩や権限昇格に直結する。

ShareDBはミドルウェアフックを提供しており、特定の操作(connect、submit、receive、apply、presenceなど)の直前後に処理を差し込める。REST APIはExpressのミドルウェアで認可判定を行うのが標準的である。両者は別の実装で動作するため、判定ロジックの一貫性を担保するための設計が必要である。

## 検討した選択肢

### 選択肢A: REST層とWebSocket層で同じ認可判定関数を共有しつつ、両層に判定を入れる

権限判定の中核ロジックを `permissions/check.ts` のような共通モジュールに集約し、REST層のミドルウェアとShareDBのミドルウェアフックから同じ関数を呼ぶ。

長所として、判定ロジックの一貫性を担保しつつ、二層で判定するため片方を漏らした場合の防御として機能する点が挙げられる。短所として、二層で判定するためのテスト網が必要になる点がある。

### 選択肢B: REST層のみで判定し、WebSocket層は信頼する

REST層でセッション確立時に判定し、その結果をセッションに記録して、ShareDBはセッション情報のみを信頼する。

長所として、実装が単純である点が挙げられる。短所として、セッション中に権限が変化した場合(共有が剥奪、文書が削除など)の追従が遅れる点と、ShareDBへ直接WebSocket接続される攻撃経路でREST層を経由しない可能性が残る点が課題である。

### 選択肢C: WebSocket層のみで判定し、REST層を最小限化する

REST APIはCRUDのみとし、認可はShareDB層に集約する。

長所として、認可ロジックが1箇所に集約される点が挙げられる。短所として、REST APIで権限が必要なエンドポイント(共有リンク発行、コメント、提案など)が判定漏れになる点が課題である。

## 決定

選択肢Aを採用する。共通モジュール `permissions/check.ts` を定義し、REST層のExpressミドルウェアとShareDBの複数のミドルウェアフックの全てから当該関数を呼び出す。判定結果はリクエスト単位で取得し直し、キャッシュしない。

ShareDBのライフサイクルにおける認可点と拒否時挙動を以下の表のとおりに固定する。本表は基本設計書 第4.2節と同一の内容であり、両文書の正式記述として扱う。差異が生じた場合は基本設計書を優先する。

| --------------------------- | ----------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ | ------------------------------------------------------- | ------------------------------------------------------------------ |
| WebSocketアップグレード前段 | Express層 `verifyClient`(wsオプション)                            | セッションCookie(認証済み利用者用)または `Sec-WebSocket-Protocol: guest-token.<token>` サブプロトコル(ゲスト用)を検証。Cookie `Path=/share/<token>` は `/sharedb` へ届かないため、ゲスト判定は本サブプロトコルのHMAC検証で行う | リソース未指定のため認可なし                            | HTTP 401、監査ログ記録(基本設計第5.9節と整合)                      |

`apply` フックで再確認する理由は、セッション継続中に権限剥奪または共有リンク失効が発生した場合、`submit` で判定済みでも `apply` 直前で再判定することで漏洩を防ぐためである。

## 根拠

採用根拠は以下のとおりである。

1. ShareDBへの直接WebSocket接続でも、REST層の判定と同じロジックが必ず動作するため、判定漏れによる権限昇格を防止できる。
2. 共通モジュールに判定を集約することで、ロジックの一貫性が保たれる。
3. 判定結果をキャッシュしないことで、権限変化(共有剥奪、文書削除)が即座に反映される。

## 影響

### ポジティブな影響

1. REST層とWebSocket層の判定が同一ロジックで動作するため、判定の食い違いが起きない。
2. 二層で判定するため、片方を実装漏れしても他方が防御層として機能する。
3. 権限変化が即座に反映されるため、共有剥奪と文書削除の運用が安全になる。

### ネガティブな影響

1. 毎回判定するため、PostgreSQLへの権限テーブル参照が頻発する。
2. テスト網がREST層とShareDB層の両方で必要になる。
3. ShareDBミドルウェアフックの仕様を理解する必要がある。

### リスクと緩和策

1. リスクとして、権限判定の頻度が高くなり、PostgreSQLの負荷が増す可能性がある。緩和策として、`document_permissions` テーブルに `(user_id, document_id)` の主キー兼インデックスを設定し、判定クエリを単一のPK検索で完結させる。本ADRは権限変化の即時反映を優先するためRedis短期キャッシュは導入しない。将来導入する場合は差し替えADRを起票し、キャッシュ無効化(共有リンク失効、権限剥奪)の伝播経路を明示したうえで実施する。
2. リスクとして、共通モジュールの仕様変更で、片方の層だけ更新を忘れる可能性がある。緩和策として、ユニットテストでREST層のミドルウェアとShareDB層のフック双方の振る舞いを検証する。
3. リスクとして、ShareDBのフック仕様の変更で、`apply` 以外のフックでの判定が必要になる可能性がある。緩和策として、ShareDBのリリースノートをアップグレード前に確認する手順を運用に組み込む。

## 自己レビュー記録

### レビュー観点1: 判定漏れの具体的シナリオ

問題点として、初版では二層判定の理由が抽象的だった。是正として、ShareDBへの直接WebSocket接続という具体的な攻撃経路を文脈と根拠に明示した。

### レビュー観点2: キャッシュ可否の判断

問題点として、初版では「キャッシュしない」の理由が示されていなかった。是正として、権限変化の即時反映を根拠とし、将来のキャッシュ導入は慎重に判断する旨をリスク緩和策で扱った。

### レビュー観点3: 性能影響への対処

問題点として、初版ではPostgreSQLへの負荷増のリスクが触れられていなかった。是正として、`(user_id, document_id)` の主キー設計とPK検索化を緩和策に追加した。

### レビュー観点4: ShareDBフックの粒度

問題点として、初版では `apply` フックだけに言及していた。是正として、`presence` フックと `submit` フックでも同等の判定を行う方針を決定の項に追記した。
