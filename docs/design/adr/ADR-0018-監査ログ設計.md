# ADR-0018 監査ログの保存期間と改ざん防止

## ステータス

Accepted (2026-06-22)

## 文脈

本MVPは内部統制と外部監査の対象となる業務利用を想定する。改ざん検出と保持期間を確定させる。

## 検討した選択肢

### 選択肢A: 1年保持+ハッシュチェーン(本案)

保持コストとリスクの均衡が取れる

### 選択肢B: 3年または7年保持

法令要求が発生してから差し替えADRで延長する余地を残すため本MVPでは不採用

### 選択肢C: WORM専用ストレージ採用

改ざん検出の最強選択肢だが導入コストが高い

### 選択肢D: 署名のみ(チェーンなし)

個別レコードの改ざんは検出できるが順序入れ替えを検出できないため不採用

## 決定

選択肢Aを採用する。具体仕様は要件定義書および基本設計書の該当節に置き、本ADRは決定の所在と根拠を保持する。

### 確定値

| 項目                 | 値                                                                                                                                                                                                                                                                                                                                                                                                                                             |
| -------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| 保持期間             | 1年間                                                                                                                                                                                                                                                                                                                                                                                                                                          |
| ハッシュアルゴリズム | HMAC-SHA-256                                                                                                                                                                                                                                                                                                                                                                                                                                   |
| 正規化               | RFC 8785 JCS + Unicode NFC                                                                                                                                                                                                                                                                                                                                                                                                                     |
| salt_version運用     | 各ソルトは「最終使用日(`secret_versions.last_used_at`)+監査ログ保持期間1年+30日グレース」以上を秘密管理基盤で保持する。四半期ごとの能動ローテーションと過去4世代以上の並行保持を標準運用とする。本基準は要件定義書NFR-05第5項を一次正本とし、本ADRと基本設計はその参照とする(旧版では「発行日基準」と書いていたが、検証ギャップを防ぐため「最終使用日基準」へ統一した。発行日基準では旧salt_versionで追記が続いた場合に検証ギャップが発生する) |
| 検証バッチ           | 日次0時、所要時間目安1万件/秒                                                                                                                                                                                                                                                                                                                                                                                                                  |

## 根拠

採用根拠は以下のとおりである。

1. ハッシュチェーンは順序入れ替えと個別改ざんの両方を検出できる。
2. 1年保持は内部統制の最短水準として一般的である。
3. salt_versionローテーションで鍵漏えい時の再起動性を確保する。
4. PostgreSQL内で完結する設計のため、本MVPの運用範囲に収まる。

## 影響

### ポジティブな影響

1. 主要設計決定のMECEな整理に寄与する。
2. 要件定義書および基本設計書とのトレーサビリティが明確になる。
3. 当該分野の確定値を1箇所に集約することで、将来の差し替えADR起票時の起点が明確になる。

### ネガティブな影響

1. 詳細仕様を要件定義書と基本設計書に委ねるため、ADR単体での自己完結性は限定的である。
2. 関連文書の更新時には本ADRも見直す必要がある。

### リスクと緩和策

1. ハッシュチェーンの並行追記でレースが起きる可能性がある。緩和策として、PostgreSQLのadvisory lockで追記をシリアライズする(基本設計書 第10.1節)。
2. AUDIT_HASH_SALTローテーションで過去レコード検証が一時的に複雑化する可能性がある。緩和策として、salt_versionカラムと「監査ログ保持1年+30日グレース」を満たすソルト保持運用で吸収する(旧版は90日並行運用としていたが、監査ログ1年保持中の検証可能性を失う矛盾があり、保持期間延長へ改訂した)。
3. 保持期間延長要求が発生した場合のストレージ膨張に対しては、パーティショニングで対応する余地を残す。

## 自己レビュー記録

### レビュー観点1 (ADR-0018): payload正規化

json-canonicalize(RFC 8785)を採用しキー順序の非決定性を排除する。

### レビュー観点2 (ADR-0018): ハッシュチェーン検証バッチ周期

日次。所要時間は1万件/秒程度を想定する。

### レビュー観点3 (ADR-0018): 改ざん検出時の運用

別テーブル隔離、バックアップ復元、salt_versionローテーションの3段階で対応する。
