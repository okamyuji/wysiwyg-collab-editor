# ADR-0024 画像処理ライブラリにSharpを採用する

## ステータス

Accepted (2026-06-25)

## 文脈

サムネイル生成とExif除去のための画像処理ライブラリを選定する。

## 検討した選択肢

### 選択肢A: Sharp(本案)

libvipsベースで高速かつ低メモリ

### 選択肢B: ImageMagick(node-imagemagick経由)

高機能だが起動オーバーヘッドが大きく、本MVPの規模では過剰

### 選択肢C: jimp

純JS実装で遅く、本MVPの想定アップロード頻度に対し性能不足

## 決定

選択肢Aを採用する。具体仕様は要件定義書および基本設計書の該当節に置き、本ADRは決定の所在と根拠を保持する。

### 確定値

| 項目       | 値                                       |
| ---------- | ---------------------------------------- |
| ライブラリ | Sharp 0.33以降                           |
| 依存       | libvips                                  |
| Exif除去   | アップロード受信時に必須化、テストで検証 |
| サムネイル | 1280、640、320の3サイズ                  |
| 対応形式   | PNG、JPEG、GIF、WebP                     |

## 根拠

採用根拠は以下のとおりである。

1. Sharpは大量のサムネイル生成で高速である。
2. libvipsの省メモリ特性は単一プロセスの共存(ADR-0015)と相性が良い。
3. Exif除去とフォーマット変換が同一APIで完結する。
4. Node.js環境で最も採用されている。

## 影響

### ポジティブな影響

1. 主要設計決定のMECEな整理に寄与する。
2. 要件定義書および基本設計書とのトレーサビリティが明確になる。
3. 当該分野の確定値を1箇所に集約することで、将来の差し替えADR起票時の起点が明確になる。

### ネガティブな影響

1. 詳細仕様を要件定義書と基本設計書に委ねるため、ADR単体での自己完結性は限定的である。
2. 関連文書の更新時には本ADRも見直す必要がある。

### リスクと緩和策

1. libvipsのネイティブ依存が一部Linuxディストリで問題化する可能性に対しては、Distrolessまたはdebian-slimのコンテナイメージで吸収する(ADR-0028)。
2. メジャーバージョンアップで破壊的変更が入る可能性に対しては、依存バージョンを固定し、リリース毎に動作確認する。
3. 画像形式の新規対応が必要となった場合は、Sharpがlibvipsの新規対応を取り込むまで遅延する可能性がある。

## 関連ADR

1. ADR-0007画像と添付ファイルの保存にS3互換オブジェクトストレージ(MinIO)を採用する

## 自己レビュー記録

### レビュー観点1 (ADR-0024): 依存バージョン

Sharp 0.33以降を採用する。

### レビュー観点2 (ADR-0024): Exif除去テスト

Exif除去後のメタデータが空であることを必ずユニットテストで検証する。

### レビュー観点3 (ADR-0024): 将来差し替え

libvipsの保守状況に問題が出た場合は差し替えADRを起票する。
