# Architecture Decision Records

本ディレクトリには、WYSIWYG共同編集エディタの主要な設計決定を記録したADRを格納する。ADRはMichael Nygardが提案した形式を基礎に、本プロジェクトの文脈に合わせて項目を拡張したものである。

## 採用するテンプレート

各ADRは以下の節構成で記述する。

1. ステータス(Proposed、Accepted、Deprecated、Supersededのいずれかと日付)
2. 文脈(なぜこの決定が必要になったか)
3. 検討した選択肢(採用候補、それぞれの長所と短所)
4. 決定(最終的に何を選んだか)
5. 根拠(なぜその選択肢を選んだか)
6. 影響(ポジティブな影響、ネガティブな影響、リスクと緩和策)
7. 関連ADR (依存するADR、影響を与えるADR)
8. 自己レビュー記録(作成後に行ったレビューと是正対応)

## 採番ルール

ADRは採番順に連番を振る。一度割り当てた番号は再利用しない。決定が破棄された場合は、対象ADRのステータスをDeprecatedまたはSupersededへ変更し、後続のADRで経緯を記述する。

## 履歴と経緯

旧版でADR-0003(Quill.js採用)、ADR-0011(提案モード)、ADR-0016(版履歴上限と圧縮)の3件を一時的にProposedとしていたが、要件定義書 第8章でMECE分析により残存していた既定値と未承認事項が消滅した時点で、依拠する要件の不確定性が無くなったためAcceptedへ昇格した。本MVPでは「ADR本体に判断根拠を保持する」を方針とするため、各ADRの本文「## ステータス」節と「## 自己レビュー記録」節に昇格根拠を直接記載しているので、本READMEは案内のみとし詳細はそちらを参照する。コミット履歴は補助情報として残るが、ADR本体が一次正本である。

## ステータス変更と改訂の手順

ADRは以下の2モデルを併用する(本MVPでは「Accepted前の改訂は本文書き換え、Accepted後の方針変更は新規ADRでSupersede」を正本とする)。

1. Accepted前の改訂(本文書き換え許可)。ADRがProposedまたはAcceptedとなる前の合意形成段階では、本文の改訂を許可する。改訂時は本文末尾の「## 改訂履歴」節(必須節、ADRルーブリックで判定)に「版、日付、改訂者、概要」を1行追記する。本モデルにより、合意過程の微修正でADR番号が乱発されることを避ける。
2. Accepted後の方針変更(Supersedeまたは本文末尾の改訂履歴節への追記)。Accepted後に方針そのものが変わる場合は、原則として新規ADRを起票し、旧ADRのステータスをSupersededへ変更したうえで、旧ADRの冒頭に後継ADRの番号を追記する。一方、合意の方向性は維持しつつ実装詳細(例: 別ADRの確定によりデプロイ単位が変わるなど)が変わる場合に限り、本文を改訂しつつ「## 改訂履歴」節に記録する形を許容する。決定の取り下げはDeprecatedとし、再度の参照を抑制する。

本2モデルの境界判定は、影響範囲が「同ADRの定義範囲内で完結するか」(改訂可)、「別ADRや要件の決定を変更しうるか」(Supersede)で行う。判定が曖昧な場合は新規ADRでSupersedeを選択する。本2モデル併用はADR-0015(`worker_threads` から `export-worker` 別コンテナへ実装詳細を整合させた変更)など、過去の改訂例を含めて整合させるための統治規約である。
