# ADR-0021 監視と可観測性の方針

## ステータス

Accepted (2026-06-24)

## 文脈

本MVPの可観測性(メトリクス、ログ、トレース)の中核選定を確定させる。

## 検討した選択肢

### 選択肢A: prom-client + Pino(本案)

Node.jsエコシステムの主流選定で本MVPの規模に十分

### 選択肢B: OpenTelemetry SDK

メトリクス、ログ、トレースを統一でき将来性が高いが、本MVP規模では運用負担が増加するため見送る

### 選択肢C: Datadog等のSaaS

自社管理下方針と不整合

## 決定

選択肢Aを採用する。具体仕様は要件定義書および基本設計書の該当節に置き、本ADRは決定の所在と根拠を保持する。

### 確定値

| 項目                 | 採用                                     |
| -------------------- | ---------------------------------------- |
| メトリクスライブラリ | prom-client(Prometheus形式)              |
| ログライブラリ       | Pino 9系(構造化JSON)                     |
| メトリクス露出       | `/metrics` エンドポイント                |
| トレース             | MVPでは未採用(OpenTelemetryは将来)       |
| アラート閾値設計     | バーンレート方式、SLO 99.9パーセント基準 |

## 根拠

採用根拠は以下のとおりである。

1. prom-clientはOpenMetrics形式の出力を提供する。
2. Pinoの構造化JSONログはFluent Bitとの連携が容易である。
3. 本MVP規模ではOpenTelemetryの全機能は過剰である。
4. 将来OpenTelemetry移行時の差し替えADRで対応する。

## 影響

### ポジティブな影響

1. 主要設計決定のMECEな整理に寄与する。
2. 要件定義書および基本設計書とのトレーサビリティが明確になる。
3. 当該分野の確定値を1箇所に集約することで、将来の差し替えADR起票時の起点が明確になる。

### ネガティブな影響

1. 詳細仕様を要件定義書と基本設計書に委ねるため、ADR単体での自己完結性は限定的である。
2. 関連文書の更新時には本ADRも見直す必要がある。

### リスクと緩和策

1. メトリクス収集による性能影響に対しては、メトリクス更新間隔を5秒以上に設定して緩和する。
2. ログ集約システム(別文書のインフラ設計)の選定遅延が継続課題となる。
3. 可観測性ツールの陳腐化に対しては、年1回の選定見直しを運用に組み込む。

## 関連ADR

1. ADR-0015アプリケーションサーバーをHTTPとWebSocketの単一プロセスで構成する

## 自己レビュー記録

### レビュー観点1 (ADR-0021): メトリクス露出形式

Prometheus形式、`/metrics` エンドポイント。

### レビュー観点2 (ADR-0021): ログ構造

JSON、利用者ID、リクエストID、ログレベルを必須キーとする。

### レビュー観点3 (ADR-0021): 将来トレース対応

OpenTelemetry移行時に差し替えADRを起票する。
