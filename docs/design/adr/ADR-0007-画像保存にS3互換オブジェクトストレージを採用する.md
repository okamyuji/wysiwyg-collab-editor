# ADR-0007 画像と添付ファイルの保存にS3互換オブジェクトストレージ(MinIO)を採用する

## ステータス

Accepted (2026-05-27)

本決定の適用はフェーズ2以降である(2026-07-05のユーザー決定)。フェーズ1(現行実装水準のMVP)の適用範囲はADR-0029を参照。

## 文脈

要件定義書FR-11により、エディタは画像のアップロードと表示をサポートする。画像のサイズは1枚あたり最大10メガバイト、サムネイル3サイズ(1280、640、320)を生成して保存する。文書1件あたりの画像枚数や、システム全体の総画像枚数は明確に上限を定めない。

画像と添付ファイルの保存先は、以下の3案が考えられる。

1. オブジェクトストレージ(S3互換、MinIOやAWS S3、Google Cloud Storageなど)
2. データベースにBLOBとして保存
3. サーバーローカルのファイルシステム

## 検討した選択肢

### 選択肢A: S3互換オブジェクトストレージ(MinIO)を採用する

S3互換のオブジェクトストレージを採用し、MVPでは自前運用のMinIOを使う。アプリケーションサーバーはS3 APIで読み書きする。

長所として、ファイル容量が増えてもDB負荷に影響しない点、横スケールが容易、署名付きURLでブラウザから直接ダウンロード可能、バージョニング機能を有効化することでバックアップが組み込み可能な点が挙げられる。短所として、運用するミドルウェアが1種類増える点と、認証認可をストレージ層でも考慮する必要がある点がある。

### 選択肢B: PostgreSQLにBLOBとして保存する

画像バイナリをPostgreSQLのlarge object (lo)もしくはBYTEAカラムに格納する。

長所として、データベースのバックアップに画像も自動的に含まれ、整合性が単純な点が挙げられる。短所として、DBサイズと書き込み負荷が急増する点、配信に必ずアプリケーションサーバーを経由する点、PostgreSQLのVACUUMやWAL量に影響する点が課題である。

### 選択肢C: サーバーローカルのファイルシステムに保存する

アプリケーションサーバーの共有ボリュームに保存する。

長所として、構成が単純で運用ミドルウェアが増えない点が挙げられる。短所として、アプリケーションサーバーの水平スケール時には共有ストレージ(NFSなど)が必要となり、結局運用負荷が増える点、バックアップとバージョニングを手動で構築する必要がある点が課題である。

## 決定

選択肢AのS3互換オブジェクトストレージを採用する。MVPの実装ではMinIOをセルフホストする。バケット構成は、画像と添付の格納用にdocs-images、エクスポート生成物の一時保存用にdocs-exportsを用意する。docs-exportsには24時間で自動削除するライフサイクルを設定する。アクセス制御は、ブラウザからの直接ダウンロードに署名付きURL(有効期限付き)を用いる。

## 根拠

採用根拠は以下のとおりである。

1. PostgreSQLの負荷をデータベース業務(関係性、集計、整合性)に集中させ、バイナリ保存の負荷を分離できる。
2. 配信パスをアプリケーションサーバーから切り離せるため、画像配信がDBやアプリの可用性に影響しない。
3. バージョニング機能を有効化することで、ファイルの誤上書きと誤削除に対する保護が組み込みで得られる。
4. 本番運用ではマネージドのS3互換サービス(AWS S3、GCSなど)へ容易に移行できる。MinIOはS3 API互換である。
5. ADR-0010の認可と組み合わせて、署名付きURLによる短期アクセス権限を発行できる。

## 影響

### ポジティブな影響

1. データベースサイズと書き込み負荷が抑制され、DBのバックアップ時間も短く保てる。
2. オブジェクトストレージのバージョニングにより、誤上書きと誤削除からの復旧が組み込みで得られる。
3. 将来クラウドS3への移行が小さな変更で可能である。

### ネガティブな影響

1. 運用するミドルウェアが1種類増える。
2. ストレージ層の認可(バケットポリシー、IAMなど)を別途設計する必要がある。
3. アプリケーションサーバーがダウンロードURLを発行する責務を持つため、署名付きURLの有効期限と鍵管理を厳密に行う必要がある。

### リスクと緩和策

1. リスクとして、署名付きURLの有効期限が長すぎる場合、URLが流出すると認可なしで画像へ到達できる。緩和策として、署名付きURLの有効期限を短く(数十分程度)し、再アクセスの度に新規発行する設計とする。
2. リスクとして、MinIOの自前運用で容量見積もりを誤ると枯渇する。緩和策として、ディスク使用率のメトリクスを必ず監視対象とし、80パーセント超過でアラートを上げる。
3. リスクとして、画像の認可ロジックがREST層とストレージ層で齟齬を起こす可能性がある。緩和策として、REST層で文書の閲覧権限を確認したうえで署名付きURLを発行し、ストレージ層では署名付きURLの正当性のみを検証する役割分担を明確化する。
4. リスクとして、Exif情報の除去漏れによる位置情報や個人情報の漏洩が起こりうる。緩和策として、アップロード受信時にSharp(libvipsベースの高速画像処理ライブラリ、Node.js環境では業界で広く採用される基盤選定)で必ずExifを除去する処理を必須化し、テストで除去後のメタデータが空であることを確認する。本ADRはストレージ層に焦点を当てる目的のため、画像処理ライブラリ単独のADRは将来必要に応じて分離する。

## 自己レビュー記録

### レビュー観点1: 配信経路の独立性

問題点として、初版では「アプリケーションサーバー負荷の軽減」を抽象的に述べただけだった。是正として、署名付きURLにより配信がアプリの可用性に影響しない点を明記した。

### レビュー観点2: Exif除去の必須化

問題点として、初版ではExif情報の除去がリスクとして識別されていなかった。是正として、リスクと緩和策の項に追加した。

### レビュー観点3: 署名付きURLの有効期限

問題点として、初版では署名付きURLの管理運用方針が触れられていなかった。是正として、有効期限を短く設定する方針をリスク緩和策に追加した。

### レビュー観点4: 将来クラウド移行への余地

問題点として、初版ではMinIOから将来クラウドのS3互換サービスへの移行容易性が触れられていなかった。是正として、根拠と影響の項に明記した。
