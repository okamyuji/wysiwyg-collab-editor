# ADR-0006 認証方式としてメールパスワードによるセッションCookieを採用する

## ステータス

Accepted (2026-05-25)

## 文脈

本システムの認証認可は、HTTP REST APIとWebSocket(ShareDB)の両系統で動作する必要がある。要件定義書FR-08により、利用者の認証はメールアドレスとパスワードを基本とする。MVPの段階では外部IdPとの連携(OIDC、SAML)は対象外である。

認証情報をクライアントとサーバー間で受け渡す方式は、大別して以下の3系統がある。

1. セッションCookie方式(サーバー側のセッションストアにキーで紐付け)
2. JWT(JSON Web Token)をローカルストレージかCookieで保持する方式
3. セッションCookieとJWTのハイブリッド

## 検討した選択肢

### 選択肢A: セッションCookie方式を採用する

express-sessionとconnect-redisにより、サーバー側のRedisにセッションを保存する。クライアントはセッションIDをCookieで受け取り、HttpOnly、Secure、SameSite=Lax属性で送出する。

長所として、伝統的なWeb認証であり、CSRF対策、Cookie属性、サーバー側での失効(ログアウト、強制ログアウト)が単純である点、WebSocketのHTTPアップグレード時に同じCookieで認証できる点が挙げられる。短所として、ステートフルでありセッションストアが必要だが、Redis(ADR-0005)を既に運用するためコストは小さい。

### 選択肢B: JWT方式を採用する

サーバーがJWTを署名発行し、クライアントはBearerトークンとしてAPIへ送信する。

長所として、ステートレスでスケールしやすい点が挙げられる。短所として、失効(ログアウト)を厳密に行うには結局サーバー側にブラックリストが必要であり、ステートレスの利点が薄まる。ローカルストレージに保存する場合はXSSの影響が大きく、Cookieに保存するならセッションCookieと事実上同じ運用形態になる。WebSocketのアップグレード時に追加のヘッダ処理が必要となり、設計が複雑化する。

### 選択肢C: セッションCookieとJWTのハイブリッド

ブラウザ向けにはセッションCookie、外部API利用者向けにはJWTを発行する併用案である。

長所として、用途別に最適化できる点が挙げられる。短所として、MVPでは外部API利用者の要件がないため過剰である。

## 決定

選択肢AのセッションCookie方式を採用する。実装にはexpress-sessionとconnect-redisを使う。本ADRの決定事項は以下のとおりである。

1. ハッシュアルゴリズムはArgon2idを採用する(OWASP推奨方式)。具体パラメータ(メモリ、反復、並列、出力長、ソルト長)は[詳細設計03認証セッションCSRF](../detailed/03-auth-session-csrf.md)で確定する。
2. Cookie属性はHttpOnly+Secure+SameSite=Laxを採用する。Max-Ageとストアの詳細は詳細設計03で確定する。
3. パスワードポリシーは最小長12文字以上、3種類以上の文字種、Have I Been Pwned辞書照合を必須とする。
4. ログイン試行制限はアカウント単位とIP単位の二段で実施する。窓と上限とブロック時間は詳細設計03で確定する。
5. メール検証とパスワードリセットはメール経由ワンタイムリンクで実施する。有効期限と再利用防止の機構は詳細設計03で確定する。
6. セッション有効期限は最終アクセスから延長する方式を採用する。具体秒数は詳細設計03で確定する。
7. CSRFはcsrf-csrfミドルウェアでDouble Submit Cookie + HMAC方式とし、状態変更APIでX-CSRF-Tokenを必須化する。WebSocketはOrigin照合とする。
8. ゲスト用WebSocket認証は短命トークンとサブプロトコルで実施する。バイト形式と有効期限は詳細設計03および[詳細設計09共有リンクとゲストセッション](../detailed/09-share-link-and-guest.md)で確定する。

## 根拠

採用根拠は以下のとおりである。

1. WebSocket(ShareDB)のアップグレード要求でも、認証済み利用者の場合は同じセッションCookieが自動付与され、HTTPとWebSocketで認証経路を統一できる。ゲスト利用者(共有リンク認証不要型)については、Cookie `Path=/share/<token>` の制約により `/sharedb` WebSocketエンドポイントへCookieが届かないため、短命WebSocketトークンとサブプロトコル `Sec-WebSocket-Protocol: guest-token.<token>` で認証する(基本設計第5.9節および要件定義書NFR-04第13項と整合)。本ADRは認証済み利用者のセッションCookie方式を主とし、ゲスト利用者のWS認証は本ADRの「ゲストセッション」節で別途定める。
2. ログアウト、強制ログアウト、セッション失効が単純である。
3. CSRF対策にはcsrf-csrfを採用する。csurfは2022年に公式アーカイブとなり保守が停止しているため不採用とする。csrf-csrfはDouble Submit Cookie + HMAC方式で、ステートレスかつExpress 5互換である。
4. Redisを既に運用するため(ADR-0005)、セッションストアの追加コストが小さい。
5. MVPでは外部API利用者がいないため、JWTのステートレス性の利点は不要である。

## ゲストセッション

本ADRは認証済み利用者のセッションを主対象とするが、共有リンク認証不要型(anyone)を通じて発行されるゲストセッションも本ADRの枠内で扱う。

ゲストセッションの決定事項は以下のとおりである。

1. UUID v7で識別し、guest_sessionsテーブルで永続化する。
2. Cookie名は `guest_session`、Path属性は `/share/<token>` 限定とし、共有リンクページのHTTP応答にのみ送信する。Cookie `Path=/share/<token>` は `/sharedb` には届かない設計とする。
3. WebSocket認証はゲスト用にCookie経路ではなく短命トークン+サブプロトコル方式を正本とする。HMAC署名、ペイロード、有効期限、サブプロトコル形式の具体は詳細設計03および詳細設計09で確定する。
4. セッション有効期限は共有リンクの有効期限と同じ(短命WSトークンの15分は別途独立)。
5. 認可は共有リンクの権限種別(viewerまたはcommenter)に従う。
6. 監査ログはactor_kind='guest'でguest_session_idを記録する。

## 影響

### ポジティブな影響

1. HTTPとWebSocketで同じ認証経路を共有でき、認可ミドルウェアの実装が単純になる。
2. ログアウト時に当該セッションをRedisから削除するだけで失効が完了する。
3. SameSite=Lax属性により、基本的なCSRF防御がブラウザ側で得られる。

### ネガティブな影響

1. ステートフルであり、セッションストアの可用性が認証可用性に直結する。
2. SameSite=Strictを採用しないため、共有リンクなど他オリジンからの遷移でもセッションCookieが送出される。クロスサイトPOST攻撃に対してはcsrf-csrfトークン検証で防御する。共有リンクの被影響面はガードされている。
3. クロスサイトAPIアクセス(将来モバイルアプリやサードパーティ連携)を実装する際は、別途トークン認証への拡張が必要になる。

### リスクと緩和策

1. リスクとして、CSRFトークンの実装漏れによる状態変更APIの不正実行が起こりうる。緩和策として、すべての状態変更APIに対するE2Eテストでcsrf-csrfトークン無しの呼び出しが拒否されることを必ず検証する。
2. リスクとして、Cookie盗難によるセッションハイジャックが起こりうる。緩和策として、Cookieの属性を厳格化し、IPアドレス変動と異常なUserAgent変動を監査ログから検出する仕組みを設ける。
3. リスクとして、Redis障害でセッション認証が一斉に無効化される可能性がある。緩和策として、Redisをフェイルオーバー可能な構成にし、無効化された場合は再ログインで復旧できるようUIを設計する。

## 自己レビュー記録

### レビュー観点1: WebSocket認証経路の整合

問題点として、初版ではWebSocketでの認証手順が明示されておらず、ゲストCookieのPath制約と `/sharedb` パスの不整合により実装不能な仕様となっていた。是正として、認証済み利用者は同じセッションCookieが自動付与される経路を根拠と影響に明記し、ゲスト利用者はCookieの `Path=/share/<token>` 制約により `/sharedb` にCookieが届かないため、短命WebSocketトークンとサブプロトコル `Sec-WebSocket-Protocol: guest-token.<token>` で認証する経路をゲストセッション節に明示した。本対応により認証済み利用者とゲスト利用者の両方でWebSocket認証が成立するようになった。

### レビュー観点2: CSRF対策の漏れ

問題点として、初版ではCSRF対策が概念的にしか触れられていなかった。是正として、決定の項でcsrf-csrfとX-CSRF-Tokenヘッダの仕様を明示し、E2Eテストでの検証もリスク緩和策に加えた。

### レビュー観点3: JWT併用の必要性検討

問題点として、初版ではJWT併用の選択肢が雑に却下されていた。是正として、選択肢Cでハイブリッド案を整理し、MVPの範囲では不要である根拠を整理した。

### レビュー観点4: セッションストア可用性

問題点として、初版ではセッションストアであるRedis障害時の影響が触れられていなかった。是正として、リスクと緩和策に組み込んだ。
