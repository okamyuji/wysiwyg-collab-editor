# ADR-0027 レートリミットの実装と適用範囲

## ステータス

Accepted (2026-06-26)

## 文脈

本MVPのレートリミットの実装ライブラリと具体的な適用範囲を確定させる。

## 検討した選択肢

### 選択肢A: express-rate-limit + rate-limit-redis(本案)

Express標準的選定で運用知見が豊富

### 選択肢B: rate-limiter-flexible

高機能だが本MVPには過剰

### 選択肢C: 自前実装

実装と検証コストが高く不採用

## 決定

選択肢Aを採用する。具体仕様は要件定義書および基本設計書の該当節に置き、本ADRは決定の所在と根拠を保持する。

### 確定値

| 項目           | 値                                                                             |
| -------------- | ------------------------------------------------------------------------------ |
| ライブラリ     | express-rate-limit + rate-limit-redis                                          |
| 適用範囲       | 認証API、共有リンクアクセス、ShareDB接続、エクスポート要求、パスワードリセット |
| 具体閾値       | 要件定義書NFR-04 12項に列挙                                                    |
| Redis共用      | ADR-0005のRedisと共用、DB番号で分離                                            |
| フォールバック | Redis障害時はプロセスローカルカウンタで継続                                    |

## 根拠

採用根拠は以下のとおりである。

1. express-rate-limitはExpress 5互換である。
2. rate-limit-redisで複数アプリケーションサーバー間の状態共有ができる。
3. 具体的な閾値は要件定義書NFR-04 12項で定める。
4. Redis共用(ADR-0005)と整合する。

## 影響

### ポジティブな影響

1. 主要設計決定のMECEな整理に寄与する。
2. 要件定義書および基本設計書とのトレーサビリティが明確になる。
3. 当該分野の確定値を1箇所に集約することで、将来の差し替えADR起票時の起点が明確になる。

### ネガティブな影響

1. 詳細仕様を要件定義書と基本設計書に委ねるため、ADR単体での自己完結性は限定的である。
2. 関連文書の更新時には本ADRも見直す必要がある。

### リスクと緩和策

1. Redis障害時のレートリミットの挙動に対しては、フォールバック(プロセスローカルカウンタ)を実装する。
2. ホワイトリストIP(社内静的IP)の管理に対しては、別途の運用手順書に集約する。
3. DoS攻撃対策としては、リバースプロキシ層のレートリミットを併用する。

## 関連ADR

1. ADR-0006認証方式としてメールパスワードによるセッションCookieを採用する
