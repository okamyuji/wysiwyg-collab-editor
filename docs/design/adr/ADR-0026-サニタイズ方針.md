# ADR-0026 入力サニタイズの方針

## ステータス

Accepted (2026-06-26)

## 文脈

XSS耐性のための入力サニタイズの実装方針を確定させる。

## 検討した選択肢

### 選択肢A: isomorphic-dompurify双方サニタイズ(本案)

サーバーとクライアントの両方で同一APIを利用

### 選択肢B: サーバー側のみのサニタイズ

クライアントのリッチペースト時の防御が不十分

### 選択肢C: クライアント側のみのサニタイズ

サーバー保存前の最終ガードを欠く

## 決定

選択肢Aを採用する。具体仕様は要件定義書および基本設計書の該当節に置き、本ADRは決定の所在と根拠を保持する。

### 確定値

| 項目           | 値                                                        |
| -------------- | --------------------------------------------------------- |
| ライブラリ     | isomorphic-dompurify 2系                                  |
| 適用箇所       | サーバー保存前、クライアント表示前、ペースト直後          |
| 許可タグ・属性 | Quill 2系の標準フォーマットに準拠、共通定数ファイルへ集約 |
| 互換性検証     | 新バージョン採用時にE2Eで回帰確認                         |

## 根拠

採用根拠は以下のとおりである。

1. サーバー側はjsdomで完全なDOM操作を行える。
2. クライアント側はQuill clipboardモジュールでペースト時に即時サニタイズできる。
3. 両層で同一のサニタイズルールを採用するため、混乱を避けられる。
4. ライブラリの更新時に両層を同期できる。

## 影響

### ポジティブな影響

1. 主要設計決定のMECEな整理に寄与する。
2. 要件定義書および基本設計書とのトレーサビリティが明確になる。
3. 当該分野の確定値を1箇所に集約することで、将来の差し替えADR起票時の起点が明確になる。

### ネガティブな影響

1. 詳細仕様を要件定義書と基本設計書に委ねるため、ADR単体での自己完結性は限定的である。
2. 関連文書の更新時には本ADRも見直す必要がある。

### リスクと緩和策

1. isomorphic-dompurifyのアップグレードで挙動変化が起きる可能性に対しては、CHANGELOGを確認し回帰テストで検出する。
2. カスタム属性の許可リストとQuillのフォーマット定義の同期に関しては、設定を共通定数ファイルへ集約する。
3. 複雑な装飾(例えばカスタムBlot)が増加した場合のサニタイズルール拡張は差し替えADRで対応する。

## 関連ADR

1. ADR-0003 WYSIWYGエディタとしてQuill.jsを採用する
2. ADR-0010認可判定をREST層とWebSocket層の二重で実施する
