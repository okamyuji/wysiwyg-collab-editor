# ADR-0005 複数アプリケーションサーバー間のOT同期にRedis Pub/Subを採用する

## ステータス

Accepted (2026-05-22)

本決定の適用はフェーズ2以降である(2026-07-05のユーザー決定)。フェーズ1(現行実装水準のMVP)の適用範囲はADR-0029を参照。

## 文脈

要件定義書NFR-01により、本システムは1文書同時編集者最大30名、システム全体5000同時接続を上限として想定する。可用性要件NFR-02により、アプリケーションサーバーは2台以上の冗長構成を取る。

ShareDBが複数プロセスへ分散された場合、ある編集者の操作が他のプロセスに接続している編集者へ届くためには、プロセス間で操作を伝播する仕組みが必要である。ShareDBはこの目的のためにPub/Subアダプタ機構を提供しており、Redis Pub/Sub、ZooKeeper、メモリ内Pub/Sub(単一プロセス向け)などのアダプタが用意されている。

## 検討した選択肢

### 選択肢A: Redis Pub/Subを採用する

Redis Pub/Subはアプリケーションサーバー間の操作伝播チャネルとして使う。sharedb-redis-pubsubアダプタが公式に提供されている。

長所として、設定が単純、軽量、レイテンシが低い、運用知見が広く存在する、セッションストアとレートリミットのカウンタにも同じRedisを併用できる点が挙げられる。短所として、Redis Pub/Subは購読者がいない場合や購読接続が切れている間のメッセージを保持しないfire-and-forget配信であり、メッセージロストが理論上ありうる点がある。

### 選択肢B: ZooKeeper、NATS、Kafkaなどの分散メッセージングを採用する

より厳密な配信保証や順序保証を持つメッセージングを使う案である。

長所として、配信保証や順序保証が強化される点が挙げられる。短所として、運用コストが高く、本システムの規模(5000同時接続)に対しては過剰である点が決定的である。

### 選択肢C: メモリ内Pub/Sub(単一プロセス構成のまま)

アプリケーションサーバーを単一プロセスに限定し、Pub/Subはメモリ内で完結させる案である。

長所として、構成が最も単純である点が挙げられる。短所として、可用性要件NFR-02を満たせない点と、水平スケールができない点が課題である。

## 決定

選択肢AのRedis Pub/Subを採用する。アダプタはsharedb-redis-pubsubを採用する。Redisインスタンスは、ShareDBのPub/Sub、express-sessionのストア、レートリミットのカウンタの3用途で共用する。

## 根拠

採用根拠は以下のとおりである。

1. ShareDB公式のPub/Subアダプタとして実績がある。
2. 5000同時接続の規模に対してRedis Pub/Subは十分な性能を持つ。
3. セッションストアとレートリミットでもRedisを利用するため、運用対象のミドルウェアを増やさずに済む。
4. ShareDBはサーバーローカルにスナップショットを保持し、Pub/Sub経由で他サーバーの操作を受け取って自身のスナップショットを更新する仕組みである。万一Pub/Subメッセージのロスが発生しても、永続化層のops(ADR-0004)を再読込することで状態が再構築できる。
5. RedisのフェイルオーバーはSentinelもしくはマネージドサービスで担保できる。

## 影響

### ポジティブな影響

1. アプリケーションサーバーの水平スケールが可能になる。
2. 軽量で運用コストが低く、ミドルウェア種別を最小限に抑えられる。
3. セッションストアとレートリミットのカウンタにも同じRedisを利用するため、運用対象が増えない。

### ネガティブな影響

1. Pub/SubはRedisの障害時にメッセージ伝播が止まり、ピア間の同期が遅延する可能性がある。
2. メッセージロストが理論上ありうるため、状態整合性は永続化層の再読込で担保する必要がある。
3. Redisがクラスタ構成の場合、Pub/Subチャネルのシャーディング設計が必要になる。

### リスクと緩和策

1. リスクとして、Redis障害でPub/Subが停止し、複数サーバー間の同期が止まる可能性がある。緩和策として、Sentinelによるフェイルオーバーを構築し、復旧時はShareDBがopsの再読込で状態を再同期する設計とする。
2. リスクとして、Pub/Subメッセージロストにより、ローカルスナップショットが古いまま残る可能性がある。緩和策として、基本設計書 第5.3.1節「切断と再接続のレジリエンスプロトコル」で定める仕組みを必ず実装する。具体的には、クライアント再接続時に最後の受信版番号を `since` パラメータで渡し、サーバーはShareDB.opsから漏れているopsを取得して再送する。サーバー側のローカルスナップショットがopsと不整合な場合は破棄して永続化層から再構築する。
3. リスクとして、Redisが他用途(セッション、レートリミット)と帯域を奪い合う可能性がある。緩和策として、本番運用では用途別にDB番号を分けることでネームスペースを分離する。負荷増大時にはRedisインスタンスを用途別に分離する余地を残す。
4. リスクとして、未ACK操作を保持するクライアントが切断した場合、再接続後の再送で重複適用や順序逆転が起こる可能性がある。緩和策として、クライアント発操作には `client_seq` を付与し、サーバー側で当該文書当該利用者の最終ACK済み `client_seq` 以下の操作は冪等に無視する。
5. リスクとして、サーバー再起動時に当該プロセスへ接続中のクライアントが一斉に再接続を試み、雪崩を引き起こす可能性がある。緩和策として、クライアントの再接続戦略をエクスポネンシャルバックオフ(初回1秒、上限30秒、最大10回)に固定し、ジッタを加える。

## 自己レビュー記録

### レビュー観点1: 配信保証の弱さへの言及

問題点として、初版ではRedis Pub/Subのat-most-once配信保証への言及が浅かった。是正として、決定の項と影響の項に明示し、永続化層の再読込で整合性を担保する設計方針を根拠に組み込んだ。

### レビュー観点2: Redis共用の影響

問題点として、初版ではRedisを多用途で共用する設計の影響が整理されていなかった。是正として、リスクと緩和策にDB番号分離と将来分離可能性を加えた。

### レビュー観点3: 規模感の根拠

問題点として、初版ではRedis Pub/Subが本システム規模に十分という主張が裏付けに乏しかった。是正として、要件定義書NFR-01の同時接続数5000を根拠として明記した。

### レビュー観点4: 障害復旧の流れ

問題点として、初版ではRedis障害時の復旧手順が触れられていなかった。是正として、Sentinelによるフェイルオーバーとopsからの再同期という二段構えを明示した。
