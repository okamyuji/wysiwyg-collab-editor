# ADR-0003 WYSIWYGエディタとしてQuill.jsを採用する

## ステータス

Accepted (2026-05-18)

表機能のMVP範囲は、要件定義書 第8章(項番11)においてMECE分析の根拠付きで確定した(quill-better-table採用、行列の追加削除と基本装飾のみ、セル結合と分割は将来拡張)。本ADRはこの確定値を前提とし、Acceptedとして確定する。

## 文脈

本システムでは、装飾付きのWYSIWYG編集体験をブラウザ上で提供する。WYSIWYGエディタはブラウザの `contenteditable` 機構の癖を抽象化し、装飾と構造化要素の編集を扱う必要がある。`contenteditable` を素のまま使うと、ブラウザ間の挙動差異とDOMの予測困難な変化により実装が破綻するため、既存のエディタライブラリの採用が必須である。

エディタの選定は、ADR-0002で採用したShareDBとの相性、装飾の豊富さ、保守状況、学習コスト、フレームワーク独立性の5観点から検討する。

## 検討した選択肢

### 選択肢A: Quill.jsを採用する

Quill.jsは純粋なJavaScriptで実装されたWYSIWYGエディタである。文書差分の表現としてDelta形式を用い、ShareDBの `rich-text` 操作タイプと完全互換である。React、Vue、Svelteなどの個別フレームワークに依存しない。

長所として、ShareDBとの直接的な互換性、装飾の標準サポート、フレームワーク独立性、学習コストの低さが挙げられる。短所として、テーブルや複雑な配置などの構造化要素の表現は限定的で、独自Blotの実装が必要な場面がある。

### 選択肢B: ProseMirrorを採用する

ProseMirrorはAtlassianのConfluenceやNew York Timesなどの大規模文書系プロダクトで採用実績のある高機能WYSIWYGエディタである。スキーマベースの厳格な文書モデルを採用し、独自のOT互換変換(collabモジュール)を提供する。

長所として、複雑な装飾と構造化要素(テーブル、画像配置、脚注など)を厳密に扱える点が挙げられる。短所として、ShareDBの `rich-text` とは互換性がなく、別の操作モデルを採用しているため、ShareDBと組み合わせるには変換層の実装が必要である点が決定的である。

### 選択肢C: Tiptapを採用する

TiptapはProseMirrorのラッパーであり、ReactとVueで利用しやすいAPIを提供する。Notion風UIに近い体験を構築できる。

長所として、UI構築の容易さが挙げられる。短所として、選択肢Bと同じくShareDBとの直接的な互換性がない点が課題である。

### 選択肢D: contenteditableを素のまま用いて自前実装する

ブラウザ標準の `contenteditable` 属性のみで実装する選択肢である。

長所として、依存ライブラリが不要である。短所として、ブラウザ間の挙動差異の吸収、IME処理、装飾の整合性確保、Undo Redoの信頼性確保、コピーペーストの正規化など、エディタライブラリが既に解決している問題を自前で解決する必要があり、実装規模が膨大になる点が決定的である。

## 決定

選択肢AのQuill.jsを採用する。バージョンは現行の安定版を起点とし、定期的に追従する。

表機能のMVP範囲を以下に固定する。本範囲はOT変換、PDFエクスポート、DOCXエクスポート、Markdownエクスポートのすべてで完全互換となる範囲のみを採用するためである。

1. 表の挿入は `quill-better-table` 拡張モジュールを利用する。
2. Delta上の表現は `table`、`table-row`、`table-cell` の固定スキーマで保持する。
3. 行と列の追加削除のみを許可する。セル結合と分割はMVPの対象外とする。
4. セル内の装飾は、平文テキスト、太字、斜体、下線、文字色、背景色、ハイパーリンクのみを許可する。
5. 見出し、箇条書き、画像、入れ子の表をセル内に配置することは禁ずる。
6. 列幅の手動調整はMVPの対象外とし、内容幅に追従する。
7. Markdownエクスポートでは、GitHub Flavored Markdownのパイプテーブル表現のみに変換する。セル装飾のうちMarkdownに表現できないものは警告を表示する。

## 根拠

採用根拠は以下のとおりである。

1. ShareDBの `rich-text` 操作タイプと完全互換であり、フロントエンドとバックエンドで同一の操作モデルを共有できる。
2. Delta形式は挿入、削除、保持の3種類の操作で構成され、OTの変換規則と直接対応する。
3. フレームワーク独立性が高く、Vite+で構築するシングルページアプリケーションへの組み込みが容易である。
4. 太字、斜体、下線、見出し、リスト、引用、コードブロック、画像、リンクを標準でサポートし、要件定義書FR-02の主要装飾をほぼ追加実装なしで実現できる。
5. 表の実装は標準では限定的だが、独自Blotの追加で対応可能である。

## 影響

### ポジティブな影響

1. ShareDBとの統合が単純になり、操作の送受信に専用の変換層が不要となる。
2. 学習コストが低く、エディタの拡張ポイント(Blot、Format、Module)を把握すれば多くの要件を満たせる。
3. フレームワーク独立性のおかげで、将来UIシェルを差し替えてもエディタ実装を再利用できる。

### ネガティブな影響

1. 表の機能を高度化する場合、独自Blotの実装が必要となり、変換規則の整合性確認が追加で必要になる。
2. ProseMirrorに比べて、文書モデルの厳格性は弱く、不正な装飾の組み合わせを禁じる仕組みはアプリケーション側で実装する必要がある。
3. Quill.jsのリリースペースは緩やかであり、新興フォーマットへの対応が遅れる可能性がある。

### リスクと緩和策

1. リスクとして、Quill.jsが将来メンテナンスされなくなる可能性がある。緩和策として、Delta形式は仕様が公開されており、最悪の場合は別エディタ実装に切り替えられる余地を残すため、フォーマット拡張は標準APIの範囲に留める。
2. リスクとして、独自Blotを多用すると変換規則の整合性確認が困難になる。緩和策として、表の機能拡張は段階的に行い、テストケースをBlotごとに整備する。
3. リスクとして、装飾の組み合わせを禁じるアプリケーション側の検査を怠ると、不正な状態に陥る可能性がある。緩和策として、Quill.jsの `clipboard` モジュールでのサニタイズと、サーバー側でのisomorphic-dompurifyによるサニタイズの二重防御で担保する。

## 自己レビュー記録

### レビュー観点1: 装飾の網羅性

問題点として、初版では「装飾の豊富さ」を抽象的に述べただけで、要件定義書FR-02の装飾とQuill.jsの対応関係が示されていなかった。是正として、根拠の項に標準装飾の網羅状況を具体的に追記した。

### レビュー観点2: 表機能の限界の扱い

問題点として、初版では「表は限定的」とだけ記述して終わっていた。是正として、影響の項とリスクの項に、独自Blotの追加と整合性確認の必要性を明示した。

### レビュー観点3: ProseMirrorとTiptapの不採用理由

問題点として、初版では「ShareDBと互換でない」とだけ述べて整理が浅かった。是正として、変換層の自前実装コストと、結局ShareDBの利点を相殺する点を具体的に補足した。

### レビュー観点4: セキュリティとサニタイズ

問題点として、初版ではエディタの装飾サニタイズについて触れていなかった。是正として、クライアント側のclipboardと、サーバー側のisomorphic-dompurifyによる二重防御を緩和策として追記した。
