# ADR-0008 フロントエンドの開発と本番ビルドの基盤としてVite+を採用する

## ステータス

Accepted (2026-06-01)

## 文脈

フロントエンドはTypeScriptで実装するシングルページアプリケーションである。ビルド、開発サーバー、フォーマッタ、linter、ユニットテストランナーをどの基盤に揃えるかが、開発体験と保守性に大きく影響する。

本プロジェクトのフロントエンド基盤は、素のViteではなく、VoidZeroが提供するVite+のエコシステムを採用する方針である。Vite+は素のViteと完全互換のうえ、Vitest、Rolldown、Oxc(高速TypeScriptパーサ)、フォーマッタ、linterを統合的に扱える商用配布物である。少人数開発体制でビルド、テスト、フォーマッタ、linterの設定を分散させずに単一基盤に揃えられる点が本MVPの開発体制と整合する。CLI仕様はVoidZero公式文書を一次出典として参照する。

## 検討した選択肢

### 選択肢A: Vite+を採用する

利用者指定の選択肢である。Vite+のエコシステムを採用し、開発、ビルド、テスト、フォーマット、lintを単一の基盤で扱う。

長所として、フォーマッタとlinterまでが一貫した思想で統合される点、高速なビルドとテストが得られる点、利用者が既に習熟している点が挙げられる。短所として、Vite+は比較的新しい商用製品であり、ライセンスや長期メンテナンスについて検討が必要である点がある。

### 選択肢B: 素のVite + Vitest + ESLint + Prettierを採用する

OSSの個別ツールを組み合わせる構成である。

長所として、各ツールの情報が豊富で、設定の自由度が高い点が挙げられる。短所として、設定ファイルが分散して保守負荷が増える点、フォーマッタとlinterのルール調整に時間がかかる点、テストランナーとビルドの統合度が低い点がある。さらに利用者がVite+の採用を方針として明示しているため、本案を選ぶ合理性はない。

### 選択肢C: webpack + jest + ESLint + Prettierを採用する

伝統的なフロントエンド構成である。

長所として、情報が圧倒的に多い点が挙げられる。短所として、開発サーバーの起動とテスト実行が遅く、現代の標準的な開発体験から離れる点、利用者の方針と整合しない点が決定的である。

## 決定

選択肢AのVite+を採用する。具体的に採用する範囲とCLIマッピングは以下のとおりである。

| 用途                   | CLIコマンド                                                       | 説明                                                                                                                                                                                         |
| ---------------------- | ----------------------------------------------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| 初期セットアップ       | `setup-vp` (出典: VoidZero公式文書https://voidzero.dev/vite-plus) | プロジェクトディレクトリに対しVite+環境を一括導入する。出典URLは契約時に提供される顧客ポータルの恒久リンクを当てる。CLI名が将来 `vp init` などに改称された場合、本ADRを差し替えADRで更新する |
| 開発サーバー           | `vp dev`                                                          | ローカル開発時のHMR付き開発サーバー                                                                                                                                                          |
| 本番ビルド             | `vp build`                                                        | 本番アーティファクトのバンドリングと最適化                                                                                                                                                   |
| フォーマット確認とlint | `vp check`                                                        | Vite+のフォーマッタとlinterを一括実行する集約コマンド。差分があれば非ゼロ終了する                                                                                                            |
| フォーマット適用       | `vp check --fix`                                                  | フォーマット違反を自動修正する                                                                                                                                                               |
| テスト実行             | `vp test`                                                         | Vite+標準のテストランナーによるユニットテストとカバレッジ計測を実行する                                                                                                                      |
| プレビュー             | `vp preview`                                                      | ビルド後のアーティファクトをローカルで起動して確認する                                                                                                                                       |

CIパイプラインへの統合はADR-0009に従う。

## 根拠

採用根拠は以下のとおりである。

1. 利用者が明示的に方針として指定している。
2. ビルド、テスト、フォーマット、lintが単一の基盤で統合され、設定ファイルの分散が抑制される。
3. 高速なバンドリングとテスト実行により、開発体験が良くなる。
4. ADR-0009の品質ゲートをCIで強制する際、フォーマッタとlinterとテストが一貫した思想で動作するため、CIの設定が単純になる。

## 影響

### ポジティブな影響

1. 設定が一本化され、新規参画者の立ち上がりが速い。
2. ビルドとテストの待ち時間が短くなり、開発速度が向上する。
3. フォーマッタとlinterのルール衝突が起きにくい。

### ネガティブな影響

1. Vite+は比較的新しい商用製品であり、ライセンスコストの予算化が必要である。
2. OSSコミュニティの情報がVite+特化の範囲ではまだ少なく、トラブル時に本家サポートへの依存が出る場面がありうる。
3. Vite+独自の機能に深く依存すると、別基盤への移行コストが高くなる。

### リスクと緩和策

1. リスクとして、Vite+のライセンス体系が将来変更される可能性がある。緩和策として、Vite+独自の拡張機能の利用は最小限に留め、素のViteへ移行できる範囲で構成する。
2. リスクとして、Vite+のフォーマッタとlinterのルールセットが将来更新され、CIが落ちる可能性がある。緩和策として、依存バージョンを固定し、定期的なバージョンアップを計画タスクとして扱う。
3. リスクとして、Vite+の文書が商用ゆえに公開範囲が狭く、トラブル解決に時間がかかる可能性がある。緩和策として、Vite+の購入契約と同時にサポート窓口を確保する。

## 関連ADR

1. ADR-0009 CI上の品質ゲートをgitleaks、Vite+フォーマッタとlinter、Vite+標準のテストランナー、Playwrightの4種類に固定する

## 自己レビュー記録

### レビュー観点1: 採用理由の利用者方針への帰着

問題点として、初版では採用理由が技術的な観点に偏り、利用者指定が根拠の中心であることが弱く示されていた。是正として、決定と根拠の項で「利用者が明示的に指定」を最上位の根拠として明記した。

### レビュー観点2: ライセンスとコスト

問題点として、初版ではVite+の商用ライセンスとコスト面の影響が触れられていなかった。是正として、影響とリスクの項に追加した。

### レビュー観点3: 範囲の明確化

問題点として、初版ではVite+のどの機能まで採用するかが曖昧だった。是正として、決定の項に採用する5つの範囲を箇条書きで明示した。

### レビュー観点4: 移行可能性

問題点として、初版ではVite+独自機能への依存リスクが触れられていなかった。是正として、素のViteへの移行余地を残す方針をリスク緩和策に加えた。
