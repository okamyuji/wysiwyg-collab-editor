# ADR-0015 アプリケーションサーバーをHTTPとWebSocketの単一プロセスで構成する

## ステータス

Accepted (2026-06-15)

## 文脈

アプリケーションサーバーが扱う通信は、HTTPによるREST API、HTTPアップグレードを経たWebSocketによるShareDB通信の2系統である。両者を単一プロセスで扱うか、別プロセスに分離するかを選定する必要がある。

両者は認可ロジック(ADR-0010)を共通化する必要があり、ShareDBはRedis Pub/Sub(ADR-0005)で複数サーバー間の同期を取る。冗長化は2台以上の構成で行う(NFR-02)。

## 検討した選択肢

### 選択肢A: HTTPとWebSocketを単一プロセスで提供する

Node.jsの単一プロセス内でExpressとShareDB(WebSocket)を起動し、同じHTTPサーバーにマウントする。

長所として、認可ロジックの共通モジュール(ADR-0010)を直接共有できる点、HTTPとWebSocketの単一プロセス化で運用が単純になる点、プロセス間通信のオーバーヘッドがない点、セッションCookieが同じプロセスで読める点が挙げられる。短所として、CPU負荷の大きいエクスポート処理(Playwright)を本プロセスに同居させるとWebSocketの応答性が低下するため、本ADRの最終確定ではエクスポートは別コンテナ(ADR-0028の `export-worker`)へ完全分離する。

### 選択肢B: HTTPとWebSocketを別プロセスに分離する

REST APIサーバーとShareDB専用サーバーをそれぞれ別のプロセスで起動し、リバースプロキシで振り分ける。

長所として、片方の負荷が他方に影響しにくい点が挙げられる。短所として、認可判定の共通化に共有ライブラリのパッケージ化が必要になる点、セッションの参照が両プロセスで重複する点、デプロイ単位が2つに増える点が課題である。

### 選択肢C: マイクロサービス的に細分化する

REST API、ShareDB、エクスポート、認証などをそれぞれ別サービスとして起動する。

長所として、責務の分離が明確になる点が挙げられる。短所として、本システムの規模(5000同時接続程度)に対して過剰な設計である点と、運用負荷の増加が決定的である点がある。

## 決定

選択肢Aを採用する。アプリケーションサーバーは単一プロセスで、ExpressとShareDB(WebSocket)を同じHTTPサーバーにマウントする。エクスポート処理はアプリケーションサーバー内では行わず、`export-worker` 専用コンテナ(別Deployment、debian-slimベース、Playwright同梱)が担当する(ADR-0028と整合)。アプリケーションサーバーから見るとexport_jobsテーブルにジョブを挿入するだけで、メインプロセスのイベントループはエクスポート計算で占有されない。Playwrightの大きなメモリと不安定挙動はコンテナ境界で隔離される。Playwright依存OS(debian-slim)とアプリケーションサーバーOS(Distroless)が異なる事実が、別コンテナ採用の決定的根拠である。旧版では `worker_threads` 採用としていたが、デプロイ単位の二重化を解消するため本ADRも `export-worker` コンテナに統一した。ADR-0013もこれに揃えてある。

## 根拠

採用根拠は以下のとおりである。

1. 認可の共通モジュール(ADR-0010)を直接共有でき、判定漏れリスクを抑えられる。
2. セッションCookieが同じプロセスで読めるため、WebSocket認証が単純化される。
3. アプリケーションサーバーのデプロイ単位が単一イメージ(Distrolessの `app-server`)となり、ロールアウト戦略が単純化される。エクスポート専用の `export-worker` コンテナは別Deploymentとして分離されるが、両者は独立に水平スケールできる(ADR-0028)。
4. Pub/Sub経由で複数サーバー間の同期は確保されるため、単一プロセス構成でも水平スケールできる。
5. Playwrightをアプリケーションサーバー外の `export-worker` コンテナへ完全分離することで、メインプロセスの応答性を保ちつつエクスポートを実行できる(子プロセス化はワーカーコンテナ内部で行う)。

## 影響

### ポジティブな影響

1. 共通モジュール(認可、ロガー、メトリクス)の共有が直接的で、コードの統一感が高い。
2. アプリケーションサーバー側のデプロイとロールアウトが単一イメージで完結する(エクスポートは別イメージ、ADR-0028)。
3. プロセス間通信のオーバーヘッドがない。

### ネガティブな影響

1. CPU負荷の大きい処理(エクスポート、画像処理)が同じプロセスで動くため、リクエスト全体の応答性へ影響する可能性がある。
2. プロセスクラッシュ時にHTTPとWebSocketの両方が同時に停止する。
3. Node.jsのシングルスレッド特性により、計算負荷の大きい処理を非同期で書く必要がある。

### リスクと緩和策

1. リスクとして、Playwrightの長時間実行がメインプロセスを占有し、WebSocketの応答性が低下する可能性があった。本ADRの最終確定ではPlaywrightをアプリケーションサーバーから完全に分離し、`export-worker` 専用コンテナ(別Deployment)へ移譲する(ADR-0028と整合)。本リスクは設計レベルで除去された。
2. リスクとして、プロセスクラッシュで全機能が停止する。緩和策として、PM2やKubernetesのreadinessProbeでヘルスチェックを行い、複数台で冗長化することで、1台障害時にも他台でサービス継続できるようにする(NFR-02)。
3. リスクとして、Node.jsのメモリリークがHTTPとWebSocketの双方に波及する。緩和策として、メモリ使用量を監視し、しきい値超過で自動再起動する仕組みを運用に組み込む。
4. リスクとして、HTTPルートとWebSocketルートのパスが衝突する可能性がある。緩和策として、WebSocketは `/sharedb` パスに固定し、REST APIは `/api/*` パスに固定する。

## 自己レビュー記録

### レビュー観点1: 単一プロセスの応答性リスク

問題点として、初版ではエクスポートのCPU負荷の影響が触れられていなかった。是正として、Playwrightを子プロセス化する方針を決定とリスク緩和策に追記した。

### レビュー観点2: パス衝突の予防

問題点として、初版ではHTTPとWebSocketのパス衝突への対策が示されていなかった。是正として、`/sharedb` と `/api/*` の固定方針をリスク緩和策に追加した。

### レビュー観点3: 冗長化との整合

問題点として、初版では単一プロセス構成と冗長化の関係が曖昧だった。是正として、複数台冗長化(NFR-02)による1台障害時の継続性を緩和策に明示した。

### レビュー観点4: メモリリークへの対応

問題点として、初版では長期運用時のメモリ管理が触れられていなかった。是正として、メモリ監視と自動再起動を緩和策として追加した。
