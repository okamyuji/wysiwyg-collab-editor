# ADR-0022 国際化の範囲

## ステータス

Accepted (2026-06-24)

## 文脈

本MVPの対応ロケールとIME処理方針を確定させる。

## 検討した選択肢

### 選択肢A: 日本語+英語のUI、本文はUnicode全範囲(本案)

自社主要利用言語と国際業務の必要十分な集合

### 選択肢B: 単一言語(日本語のみ)

国際業務に対応できないため不採用

### 選択肢C: 多言語(日本語、英語、中国語、韓国語など)

翻訳とテストコストが本MVP範囲を超過するため不採用

## 決定

選択肢Aを採用する。具体仕様は要件定義書および基本設計書の該当節に置き、本ADRは決定の所在と根拠を保持する。

### 確定値

| 項目           | 値                                 |
| -------------- | ---------------------------------- |
| UIロケール     | ja-JP、en-US                       |
| 本文文字種     | Unicode全範囲                      |
| 文字方向       | 左から右のみ(BiDiは将来)           |
| IME処理        | 確定前テキストはOT送信対象外       |
| i18nライブラリ | 採用候補はreact-intlまたはformatjs |

## 根拠

採用根拠は以下のとおりである。

1. 本MVPの主要利用者は社内の日本語話者であり、社外共有先で英語UIが必要となる。
2. 本文はUnicode全範囲を許容するため、任意言語の文書作成を妨げない。
3. IME確定前のテキストはOT送信対象外として、IME未確定状態のブロードキャストを防ぐ。
4. 右から左の文字方向(BiDi)はMVP対象外とし、将来拡張で対応する。

## 影響

### ポジティブな影響

1. 主要設計決定のMECEな整理に寄与する。
2. 要件定義書および基本設計書とのトレーサビリティが明確になる。
3. 当該分野の確定値を1箇所に集約することで、将来の差し替えADR起票時の起点が明確になる。

### ネガティブな影響

1. 詳細仕様を要件定義書と基本設計書に委ねるため、ADR単体での自己完結性は限定的である。
2. 関連文書の更新時には本ADRも見直す必要がある。

### リスクと緩和策

1. 英語UIの翻訳品質に対しては、ネイティブレビューを必須化する。
2. UI文言のハードコード混入を防ぐため、i18nライブラリ(react-intlまたはformatjs)の採用を必須化する。
3. 将来ロケール拡張時のUI文字列管理工数を見積もる必要がある。

## 関連ADR

## 自己レビュー記録

### レビュー観点1 (ADR-0022): 対応ロケール

日本語(ja-JP)、英語(en-US)の2言語。

### レビュー観点2 (ADR-0022): IME処理

IME未確定中はDelta操作を発行しない。

### レビュー観点3 (ADR-0022): 将来拡張

新規ロケール追加時はi18nリソース追加のみで対応できる構成を維持する。
