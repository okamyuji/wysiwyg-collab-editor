# ADR-0016 版履歴は1文書あたり1000版を上限とし自動圧縮を行う

## ステータス

Accepted (2026-06-17)

文書履歴の保存ポリシー(明示版無期限保持、自動版1000版上限)は、要件定義書 第8章(項番3)においてMECE分析の根拠付きで確定した。本ADRはこの確定値を前提とし、Acceptedとして確定する。

## 文脈

要件定義書FR-06により、文書の版履歴は、明示スナップショットと自動スナップショットの両方を扱う。版は復元と差分閲覧に用いられる。ShareDBは操作履歴(ops)を保持しており、版番号と操作の対応が取れる。長期運用では、版とopsの蓄積によりPostgreSQL(ADR-0004)のストレージが増大する。

設計上の論点は、版とopsの保持上限、自動スナップショットの作成基準、圧縮(古いopsの統合)の方針である。

## 検討した選択肢

### 選択肢A: 版を1文書あたり1000版上限とし、自動圧縮を行う

明示スナップショットは無期限保持、自動スナップショットは1000版を上限としLRU方式で削除する。古いopsは中間スナップショットに統合して圧縮する。

長所として、ストレージ消費を予測可能に保てる点、復元と差分閲覧の対象が明確である点、明示スナップショットは保護されるため利用者の意図が尊重される点が挙げられる。短所として、自動削除の対象選定ロジックの実装が必要になる点がある。

### 選択肢B: 版を無制限に保持する

すべての版とopsを永久保持する。

長所として、設計が単純である点が挙げられる。短所として、長期運用でストレージが線形に増大する点が課題である。

### 選択肢C: 期間ベースで保持する(1年以内のみ)

時刻ベースで1年以内の版とopsのみを保持する。

長所として、運用ポリシーが明確である点が挙げられる。短所として、編集頻度の高い文書では1年で大量の版が生まれる一方、低頻度文書ではほぼ全版が残るなど、文書ごとの偏りが大きい点が課題である。

## 決定

選択肢Aを採用する。具体的なポリシーは以下のとおりである。

1. 明示スナップショット(利用者が「版を保存」を実行)は無期限に保持する。利用者の意図が明確だからである。
2. 自動スナップショットは、最後の版作成から1時間以上経過し、かつ100件以上の操作が積み上がったときに作成する。
3. 自動スナップショットは1文書あたり最新1000版までを保持し、超過時は最も古い自動スナップショットから削除する。
4. opsの圧縮は、保持対象から外れる古い区間のopsを保管テーブルへ退避させる形で実行する。圧縮ジョブは日次バッチで実行する。

圧縮実行の手順は基本設計書 第5.7.1節で正式記述として定義する。本ADRで重要な前提を以下に重ねて記述する。

1. 圧縮の境界は必ず既存のスナップショット版番号で切る。任意のop境界では切らない。
2. 圧縮対象区間に明示版が含まれる場合、明示版の直前で区間を分割し、明示版以前のopsは保持する。
3. 圧縮実行前に、当該区間内の版番号を購読中のアクティブクライアントを最新版へ強制再ロードする(基本設計書 第5.3.1節 `force_reload`)。
4. オフラインクライアントの再接続に備え、圧縮されたopsは `archived_ops` テーブルへ72時間保管する(document_id、v_start、v_endで識別)。72時間経過後に物理削除する。archived_opsへの参照はADR-0010の認可判定を適用し、当該文書の閲覧権限保持者のみが再構築リクエスト経由でopsを取得できる。権限剥奪者には404を返し、内容を露呈しない。
5. 圧縮前後で当該文書の最新版のスナップショットがSHA-256ハッシュで一致することを必ず検証する。検証失敗時は圧縮を中止し、ロールバックする。
6. 圧縮された区間内の中間状態への差分閲覧と復元はできない。UIで利用者へ「圧縮済みのため閲覧不可」と通知する。明示版は圧縮対象外であるため、明示版間の差分閲覧と復元は常に可能である。

## 根拠

採用根拠は以下のとおりである。

1. ストレージ消費が文書あたり一定の上限内に収まり、運用予測が容易になる。
2. 明示スナップショットを保護することで、利用者が「重要な版」と判断した状態は失われない。
3. 自動スナップショットの作成基準(1時間または100操作)により、編集頻度に応じた粒度で版が残る。
4. opsの圧縮により、ShareDBの操作再生時の負荷を抑えられる。

## 影響

### ポジティブな影響

1. ストレージ消費が予測可能であり、運用と容量計画が立てやすい。
2. 明示スナップショットが保護されるため、利用者にとっての安心感が高い。
3. opsの圧縮により、長期運用での再生負荷が抑えられる。

### ネガティブな影響

1. 自動スナップショットを大量に上限まで使った文書では、古い自動スナップショットが復元できなくなる。
2. 圧縮ジョブの実装と運用が必要になる。
3. 利用者が「すべての版を残したい」と要望した場合、明示スナップショット運用を案内する必要がある。

### リスクと緩和策

1. リスクとして、圧縮ジョブのバグでopsとスナップショットの整合性が崩れる可能性がある。緩和策として、圧縮前後で文書状態がハッシュ一致することを検証してから旧版を削除する設計とし、テストでこの不変条件を必ず検査する。
2. リスクとして、自動スナップショット削除を利用者が把握せず、後から「あの時点に戻りたい」と言われる可能性がある。緩和策として、UIに自動版と明示版の区別、保持上限、自動削除の説明を表示し、重要な版は「版を保存」で明示版に変換するよう案内する。
3. リスクとして、圧縮ジョブが長時間化し、業務時間帯の負荷を高める可能性がある。緩和策として、圧縮ジョブは深夜帯のバッチとし、1ジョブで処理する文書数を制限する。
4. リスクとして、自動スナップショット数の上限1000が大規模文書では不足する可能性がある。緩和策として、運用開始後の実績を踏まえて上限値を見直す方針を明示する。

## 自己レビュー記録

### レビュー観点1: 明示版と自動版の区別

問題点として、初版では明示版と自動版の扱いが混在しており、明示版が保護されることが曖昧だった。是正として、決定の項に明示版は無期限保持、自動版は1000版上限と明確に分けて記述した。

### レビュー観点2: 圧縮ジョブの整合性検証

問題点として、初版では圧縮処理の正しさをどう担保するかが触れられていなかった。是正として、圧縮前後の文書状態ハッシュ一致を検証する手順をリスク緩和策に追加した。

### レビュー観点3: 利用者通知

問題点として、初版では自動版削除を利用者へどう伝えるかが示されていなかった。是正として、UIでの説明と明示版への変換案内をリスク緩和策に追加した。

### レビュー観点4: 上限値の見直し余地

問題点として、初版では1000という数字が固定的に書かれていた。是正として、運用実績を見て見直す方針を緩和策に明示した。
