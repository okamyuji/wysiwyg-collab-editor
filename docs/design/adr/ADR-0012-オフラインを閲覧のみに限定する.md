# ADR-0012 オフライン対応を閲覧のみに限定しServiceWorkerで実現する

## ステータス

Accepted (2026-06-10)

## 文脈

要件定義書FR-09により、本システムのオフライン対応はMVPの範囲で閲覧のみとする。ネットワーク切断時には、直前に開いていた文書を閲覧できることが要件である。オフライン編集はMVPの対象外とし、再接続時にサーバーから最新状態を取得して画面を更新する。

技術選択としては、ServiceWorkerによるキャッシュ管理が標準的である。実装には素のServiceWorker APIか、Workboxのような抽象化ライブラリのいずれかを選ぶ。

## 検討した選択肢

### 選択肢A: ServiceWorker + Workboxで閲覧のみオフライン対応

ServiceWorkerをWorkboxで実装し、文書の最新スナップショットと画像をCacheStorageへ保存する。オフライン時はキャッシュから返し、編集UIは無効化する。

長所として、Workboxによりキャッシュ戦略の記述が簡潔になる点、メンテナンスが容易な点、キャッシュ失効戦略のテンプレートが充実している点が挙げられる。短所として、Vite+ビルドへの組み込みに追加設定が必要である点がある。

### 選択肢B: 素のServiceWorker APIで閲覧のみオフライン対応

ServiceWorker APIを直接記述する。

長所として、追加依存が不要である点が挙げられる。短所として、CacheStorageの戦略実装、バージョン管理、不要キャッシュの削除を自前で書く必要があり、コード量が増える点が課題である。

### 選択肢C: オフライン編集まで対応する

CRDTを併用し、オフライン編集をローカルバッファに蓄積し、再接続時にOTへ統合する。

長所として、利用者体験が向上する点が挙げられる。短所として、本システムのMVP範囲を逸脱する点と、CRDT併用のための追加実装が必要になる点が決定的である。

## 決定

選択肢AのServiceWorker + Workboxを採用する。キャッシュ対象は、直近に開いた文書のスナップショット、文書に含まれる画像、UIシェル(HTML、CSS、JavaScriptの本体)とする。キャッシュの最大保持件数は文書5件、画像50枚、UIシェルは最新版1つとする。オフライン時には編集UIを無効化し、利用者へ「オフラインのため閲覧のみ可能」と明示する。

キャッシュの認可と無効化に関するルールを以下のとおりに固定する。本ルールは基本設計書 第5.11節の正式記述と同一の内容である。

1. 共有リンク経由のゲストアクセスでは、ServiceWorkerによるキャッシュを行わない。サーバーは当該レスポンスに `Cache-Control: no-store` を付与する。
2. ログアウト時、ログアウトAPI成功直後にクライアントからServiceWorkerへキャッシュクリア指示を送り、当該利用者がアクセスしていたすべての文書キャッシュ、画像キャッシュ、ゲストセッションCookieを削除する。UIシェルキャッシュは保持してよい。
3. 文書ごとに利用者文脈依存の `cache_etag` をサーバーが応答時に算出して発行する。算出入力は本文最新版番号・メタデータ更新時刻・利用者権限種別・共有リンクIDを含み、これにより権限剥奪後のキャッシュは正しく無効化される。クライアントは `If-None-Match` ヘッダで照合し、レスポンスには `Cache-Control: private` を併用する。具体的なHMACアルゴリズム、鍵管理、鍵ローテーション周期と並行運用期間、`CACHE_ETAG_SECRET` の世代管理は[詳細設計10オフライン閲覧](../detailed/10-offline-readonly.md)で確定する。
4. ネットワーク復帰時に、クライアントは当該利用者の現在権限と最新 `cache_etag` を必ず確認する。権限剥奪済みならHTTP 403と `X-Cache-Invalidate` ヘッダにより当該文書キャッシュを即時削除する。文書削除済みならHTTP 410で同様に削除する。
5. 共有リンク失効時、当該リンク経由でWebSocket接続中のゲストへ失効通知を送り、クライアントが即時に当該文書キャッシュとCookieを削除する。オフラインだった場合は次回復帰時の `cache_etag` 確認で削除される。
6. 権限剥奪時、WebSocketを4403で切断し、クライアントは切断時に当該文書キャッシュを削除する。オフラインだった場合は次回復帰時の `cache_etag` 確認で削除される。

## 根拠

採用根拠は以下のとおりである。

1. Workboxによりキャッシュ戦略の記述が簡潔になり、保守負荷を抑えられる。
2. MVP範囲のオフライン閲覧のみであれば、追加の同期ロジックが不要である。
3. ネットワーク復帰時には、サーバーから最新スナップショットを取得して画面を更新するだけで整合性が保たれる。
4. オフライン編集は要件定義書FR-09の対象外であり、将来CRDT併用などで拡張する選択肢を残す。

## 影響

### ポジティブな影響

1. 一時的なネットワーク切断時にも直前の文書が閲覧できる。
2. ServiceWorkerによりUIシェルもキャッシュされるため、低速回線時の起動が速くなる。
3. オフライン編集の複雑性を抱え込まずに済む。

### ネガティブな影響

1. ServiceWorkerのデバッグはブラウザ依存の挙動があり、開発時の検証が手間取る場合がある。
2. キャッシュの容量制限を超過した場合の利用者体験を考慮する必要がある。
3. UIシェルの更新が利用者に届くタイミングが、ServiceWorkerの更新サイクルに依存する。

### リスクと緩和策

1. リスクとして、古いキャッシュが残り、サーバー側のバージョンと食い違う可能性がある。緩和策として、ServiceWorkerのバージョン番号をデプロイ毎に変え、`skipWaiting` と `clientsClaim` を適切に組み合わせて新版を即座に有効化する。
2. リスクとして、オフライン中に編集UIが見えてしまうと利用者の誤解を招く。緩和策として、ネットワーク状態(オンラインまたはオフライン)をUIに明示し、オフライン時はツールバーをグレーアウト表示する。
3. リスクとして、キャッシュ容量がブラウザの上限を超過した場合、書き込みが失敗する。緩和策として、キャッシュ件数の上限を設定し、LRUで自動削除する。Workboxのストラテジーを利用する。
4. リスクとして、画像のキャッシュ件数が多すぎるとブラウザストレージを圧迫する。緩和策として、画像のキャッシュ件数を50枚に限定し、超過時は古いものから削除する。

## 自己レビュー記録

### レビュー観点1: キャッシュ件数の具体化

問題点として、初版ではキャッシュ件数が「適度」と曖昧だった。是正として、文書5件、画像50枚、UIシェル1つの上限を具体的に明示した。

### レビュー観点2: 編集UI無効化の明示

問題点として、初版ではオフライン時の編集UIの扱いが曖昧だった。是正として、ツールバーグレーアウトと利用者への通知をリスク緩和策に明示した。

### レビュー観点3: ServiceWorker更新タイミング

問題点として、初版ではServiceWorker新版の有効化タイミングが触れられていなかった。是正として、`skipWaiting` と `clientsClaim` の併用を緩和策として記述した。

### レビュー観点4: 将来のオフライン編集への余地

問題点として、初版では将来のCRDT併用拡張の余地が示されていなかった。是正として、根拠の項に明記した。
