# ADR-0014 業務テーブルの主キーにUUID v7を採用する

## ステータス

Accepted (2026-06-14)

## 文脈

PostgreSQL(ADR-0004)の業務テーブルの主キー方式を確定させる。候補は以下の3案がある。

1. BIGINT GENERATED ALWAYS AS IDENTITY(整数連番)
2. UUID v4(完全乱数)
3. UUID v7(時刻順序を保つUUID)

主キーは、APIで露出する識別子としても用いる。識別子の予測可能性、生成順序、インデックス性能、分散環境での衝突回避、フロントエンドとの相性、デバッグの容易さなどが選定基準となる。

## 検討した選択肢

### 選択肢A: UUID v7を採用する

UUID v7は先頭にミリ秒精度の時刻を埋め込んだUUIDであり、生成順序が単調増加に近い。PostgreSQL 17時点では `gen_random_uuid()` は依然としてv4を生成するため、UUID v7はNode.js側で生成し、PostgreSQLへはUUID型として渡す。複数アプリケーションサーバー間で時刻ずれが生じると単調性が崩れるため、運用上の前提としてNTPまたはchronyによる時刻同期(誤差100ミリ秒以内)を必須とする。PostgreSQL 18以降の `uuidv7()` 関数の登場後に切り替えを検討する。

長所として、識別子が予測困難で外部露出に向く点、生成順序がほぼ単調でBツリーインデックスの効率が良い点、分散生成で衝突しない点、UUIDの汎用性が高い点が挙げられる。短所として、UUID v7はRFC 9562で標準化されたばかりであり、生成関数を自前で書くか、利用するライブラリのバージョンを確認する必要がある点がある。

### 選択肢B: BIGINT IDENTITYを採用する

PostgreSQL標準の整数連番である。

長所として、サイズが小さく、インデックスが高速である点が挙げられる。短所として、識別子が予測可能で、URLに露出した際に他文書のIDを推測できる点と、分散環境やバルクインポートで衝突回避の追加設計が必要な点が課題である。

### 選択肢C: UUID v4を採用する

完全乱数のUUIDである。

長所として、識別子が予測困難である点と分散生成で衝突しない点が挙げられる。短所として、生成順序がランダムであり、Bツリーインデックスの挿入位置が分散して肥大化しやすい点と、ページ分割が頻発する点が課題である。

## 決定

選択肢AのUUID v7を採用する。生成はNode.js側で行い、PostgreSQLのカラム定義はUUID型とする。生成ライブラリは `uuid` パッケージ(v10以降。v9はv7サポートが限定的なため不採用)の `v7()` 関数を用いる。

## 根拠

採用根拠は以下のとおりである。

1. 識別子が予測困難であり、共有リンクや文書URLでの直接識別子露出に対しても安全性が高い。
2. 生成順序が時刻に基づくため、PostgreSQLのBツリーインデックスの挿入位置が局所化し、UUID v4より書き込み性能が良い。
3. 分散生成しても衝突しない。アプリケーションサーバーが2台以上になる(ADR-0015)構成と整合する。
4. UUIDは標準形式でフロントエンドとの相互運用が容易である。
5. 時刻情報を含むため、デバッグ時に大まかな生成時刻が分かる。

## 影響

### ポジティブな影響

1. 識別子の予測攻撃が成立しにくい。
2. インデックスの肥大化と挿入性能劣化がUUID v4と比べて軽減される。
3. 分散環境でも衝突を心配せずに生成できる。

### ネガティブな影響

1. 主キーが16バイトとなり、BIGINTの8バイトより倍のサイズを占める。
2. 人間の連番認識より検索負荷が高く、サポート対応時の検索性は劣る。
3. UUID v7は新しい標準であり、ライブラリやツールの対応状況を確認する必要がある。

### リスクと緩和策

1. リスクとして、UUID v7の生成ライブラリの実装に欠陥があり、時刻順序が崩れる可能性がある。緩和策として、生成ライブラリのテストと、定期的な生成結果のサンプリング検証を運用に組み込む。
2. リスクとして、主キーサイズが2倍になることで、インデックスとテーブルのストレージ消費が増える。緩和策として、子テーブルにおける外部キー参照のサイズも増えるため、初期からカラム設計でこの点を織り込む。
3. リスクとして、サポート対応時の検索性が劣る。緩和策として、業務上重要なエンティティ(文書、利用者)には別途人間可読な識別子(短いランダム文字列、シーケンシャル番号など)を補助カラムとして付与する設計を将来検討する。

## 自己レビュー記録

### レビュー観点1: UUID v7標準化の状況

問題点として、初版ではUUID v7の標準化状況が触れられていなかった。是正として、RFC 9562で標準化された旨を文脈に追記した。

### レビュー観点2: 生成側の選定

問題点として、初版では生成をPostgreSQL側で行うか、Node.js側で行うかが曖昧だった。是正として、Node.js側で生成しPostgreSQLはUUID型として受ける方針を決定の項に明記した。

### レビュー観点3: 検索性の劣化

問題点として、初版ではサポート対応時の検索性劣化に対する補助策が示されていなかった。是正として、補助カラムによる対応の将来検討をリスク緩和策に追加した。

### レビュー観点4: 外部キーのサイズ波及

問題点として、初版では主キーサイズ増による外部キーへの波及が触れられていなかった。是正として、影響の項に追記した。
