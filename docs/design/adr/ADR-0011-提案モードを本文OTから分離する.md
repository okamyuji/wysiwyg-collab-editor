# ADR-0011 提案モードを本文のOTから分離して管理する

## ステータス

Accepted (2026-06-08)

提案モード粒度(連続入力を1提案として束ね、5秒の無操作で切る)は、要件定義書 第8章(項番10)においてMECE分析の根拠付きで確定した。本ADRはこの確定値を前提とし、Acceptedとして確定する。

## 文脈

要件定義書FR-05により、提案モードでは利用者の編集を本文に直接反映せず、提案として保留する。Google文書の「提案モード」相当の機能である。提案は1件単位で受諾と却下が可能であり、本文の編集が並行して進む中でも提案の意図が保たれる必要がある。

設計上の論点は、提案を本文のOTに混在させるか、本文OTから分離して管理するかである。

## 検討した選択肢

### 選択肢A: 提案を本文OTとは独立したテーブルで管理する

提案は `suggestions` テーブルに保存し、提案の本体はQuill.jsのDelta操作集合として保持する。本文への適用は、受諾時にDeltaをShareDBの `submitOp` で反映する。提案集合は本文OTの履歴とは独立した状態として扱う。

長所として、本文のOT処理が単純な状態を保てる点、提案の受諾と却下が文書OTに副作用を起こさない点、提案件数や状態の集計をSQLで容易に行える点が挙げられる。短所として、本文が編集された後に提案を受諾する際の追従処理が必要になる点がある。

### 選択肢B: 提案をShareDBの操作タイプとして本文OTに埋め込む

ShareDBのカスタム操作タイプを定義し、提案の追加、受諾、却下を本文OTの操作として扱う。

長所として、本文と提案を単一のOTストリームで扱える点が挙げられる。短所として、ShareDBの標準 `rich-text` 操作タイプを拡張する必要があり、変換規則の整合性検証が大きな負担となる点と、提案のテーブル集計が困難になる点が決定的である。

### 選択肢C: 提案をクライアント側のみで保持し、サーバーには受諾時のみ反映する

提案中の編集をクライアント側のローカルバッファに溜め、受諾時に本文OTへ反映する。サーバー側には提案の永続化を行わない。

長所として、サーバー実装が単純である点が挙げられる。短所として、他者の閲覧が必要な提案モードの要件(FR-05の提案表示)を満たせない点と、ブラウザを閉じると提案が消える点が課題である。

## 決定

選択肢Aを採用する。提案は本文OTから分離した `suggestions` テーブルで管理し、ShareDBの `rich-text` 操作タイプは拡張しない。

論理エンティティの概念モデルは以下のとおりとする。提案は文書ID、提案者ID、`base_version`、Deltaオペレーション、状態(`pending`/`accepting`/`accepted`/`rejected`/`stale`/`expired`)、楽観排他バージョン、作成時刻、状態変更時刻、状態変更者の組で表現する。`accepting` は受諾の二段CASで第一CASが占有する一時状態である。状態列に時刻別カラムは持たず、状態と状態変更時刻の組で表現する。

提案の粒度は要件定義書FR-05に従い、連続する入力を5秒の無操作で区切って1提案として束ねる(要件定義書 第8章で確定)。

受諾フローは二段CAS方式で原子性を確保する。本文submitOpの前後を第一CAS(claim)と第二CAS(confirm)で挟み、submitOp失敗時には補償CASで `pending` へ復帰させる。これにより本文submitOpと提案状態が任意時点で常に整合し、リトライ時の二重適用を不可能にする。一段方式(submitOp成功後に1回CAS)は中間状態の観測可能性により採用しない。

却下はDelta適用を行わず状態を `rejected` に更新する。自動stale化バッチは10分間隔で `base_version` から現在版までのopsが追従可能か判定し、追従不能な提案を `stale` へ遷移させて提案者と編集権限保持者へ通知する。

各CAS SQLの具体形、`suggestions` のCREATE文とCHECK制約、エラーコード、Deltaサイズ上限、インデックスは[詳細設計05提案モード二段CAS](../detailed/05-suggestion-two-phase-cas.md)で確定する。本ADRは決定とその根拠を保持し、契約は詳細設計の正本に従う。

## 根拠

採用根拠は以下のとおりである。

1. ShareDBの `rich-text` 操作タイプを拡張せずに済むため、ADR-0002のリスク(操作タイプ拡張時の整合性問題)を回避できる。
2. 提案のSQL集計(未処理件数、提案者別件数、解決時間など)を業務テーブルとして自然に扱える。
3. 本文の編集と提案が独立して進むため、編集ロックを取らずに済む。
4. 受諾と却下の操作が文書履歴の副作用にならない設計とできる。

## 影響

### ポジティブな影響

1. 本文OTの設計が単純なまま保たれる。
2. 提案の状態管理(保留、受諾、却下)が業務データとしてSQLで扱える。
3. 提案の永続化、検索、集計が容易である。

### ネガティブな影響

1. 本文編集が進んだ後に提案を受諾する際、提案Deltaの位置追従が必要になる場合がある。
2. 提案中の編集が他者から見える設計の場合、提案集合の同期に別途プレゼンス相当の仕組みが必要となる。
3. 提案と本文OTの間の整合性保証はアプリケーション側の責務となる。

### リスクと緩和策

1. リスクとして、本文編集が進んだ後の受諾時に、提案Deltaが古い文書状態を前提としているために整合性が崩れる可能性がある。緩和策として、受諾時には現在の文書状態のバージョンと提案作成時のバージョンを比較し、必要に応じてQuill.jsのDelta `transform` で位置を追従させる。追従不能な場合(本文側で当該範囲が削除されたなど)は利用者へ警告を表示し、手動操作を促す。
2. リスクとして、提案集合のリアルタイム同期が遅延する可能性がある。緩和策として、提案の追加と状態変更はShareDBのプレゼンスチャネルで通知し、クライアントが即座に再描画する。
3. リスクとして、提案件数が増えるとUIが煩雑になる。緩和策として、解決済み提案はデフォルトで折り畳み表示とし、未解決のみを表示する。提案数の上限はソフト警告で運用に委ねる。

## 関連ADR

2. ADR-0010認可判定をREST層とWebSocket層の二重で実施する

## 自己レビュー記録

### レビュー観点1: 受諾時の位置追従

問題点として、初版では本文編集後の受諾時に提案位置がずれる問題への対処が示されていなかった。是正として、リスクと緩和策にQuill.jsのDelta transformによる位置追従と、追従不能時の利用者警告を明示した。

### レビュー観点2: 提案のリアルタイム共有

問題点として、初版では提案を他クライアントへリアルタイム共有する手段が曖昧だった。是正として、ShareDBのプレゼンスチャネル経由で通知する方針を明記した。

### レビュー観点3: 操作タイプ拡張回避の重要性

問題点として、初版では `rich-text` 拡張を避ける根拠がADR-0002のリスクと紐付けられていなかった。是正として、根拠に明示しADR-0002との関連を強調した。

### レビュー観点4: 提案件数増大への配慮

問題点として、初版では提案件数が増えた場合のUI挙動が触れられていなかった。是正として、解決済み折り畳みのデフォルトをリスク緩和策に追加した。
