# ADR-0029 フェーズ分割によりMVPを現行実装水準へ再定義する

## ステータス

Accepted (2026-07-05)

2026-07-05のユーザー決定により、現行実装をフェーズ1のMVPとして正式化した。本ADRはフェーズ区分の正本である。

## 文脈

設計文書群(ADR-0001からADR-0028、要件定義書、基本設計書、詳細実装指示書)は、操作変換(OT)による同時編集統合、ShareDB、Quill.js、PostgreSQL永続化、Redis Pub/Sub、S3互換オブジェクトストレージを前提とした構成を規定している。一方、現行実装はin-memory WebSocketリレーとcontentEditableによる簡易同期であり、永続化層と日本語IME確定前の送信ガードを持たない。

この設計と実装の乖離により、受け入れレビューで「設計文書と実装が違う」という指摘が繰り返し発生し、レビューが収束しない状態が続いていた。乖離の解消方向(実装を設計へ寄せるか、設計の適用時期を段階化するか)を確定する必要があった。

## 検討した選択肢

### 選択肢A: フェーズ分割により現行実装をフェーズ1のMVPとして正式化する

長所として、既存ADRの決定と根拠を破棄せずに保持したまま、受け入れ判定の基準線を現行実装水準で一意に定められる。短所として、既存文書の「MVP」の語に適用フェーズの読み替えが必要になり、対象ADRへの注記追加という文書改訂の手間が発生する。

### 選択肢B: 実装を設計文書へ追従させる

長所として、文書改訂が不要である。短所として、ShareDB、PostgreSQL、Redis、S3互換ストレージの導入は実装規模が大きく、レビュー収束までの期間を確定できない。乖離の指摘が解消するまで受け入れ判定が停止し続ける点が決定的である。

### 選択肢C: 設計文書を現行実装水準まで縮退させ、将来分の決定を破棄する

長所として、文書と実装が即時に一致する。短所として、既存ADR全28件が保持する決定と根拠と検討経緯を失い、フェーズ2以降の実装着手時に同じ検討を繰り返す手戻りが発生する。

## 決定

選択肢Aを採用する。フェーズ区分は以下のとおりとする。

フェーズ1(MVP、現行実装水準)は、in-memory WebSocketリレーによるオンライン共同編集同期、contentEditableエディタと基本装飾(太字、斜体、下線、ハイライト)、プレゼンス人数表示、メモリ内保持のコメント、PDFとDOCXとMarkdownのエクスポート、セキュリティヘッダ付与、ブラウザ言語自動切替で構成する。永続化は行わない。フェーズ1の機械判定項目はリポジトリ直下のACCEPTANCE.mdのv1合格基準に列挙する。

以下の設計項目はフェーズ2以降へ移す。

| 設計項目                             | 正本                                    |
| ------------------------------------ | --------------------------------------- |
| OTによる同時編集統合                 | ADR-0001                                |
| ShareDB採用                          | ADR-0002                                |
| Quill.js採用                         | ADR-0003                                |
| PostgreSQL永続化                     | ADR-0004                                |
| Redis Pub/Subによる複数サーバー同期  | ADR-0005                                |
| S3互換オブジェクトストレージ         | ADR-0007                                |
| カバレッジ閾値ゲートの強制           | ADR-0009、要件定義書NFR-10              |
| 日本語IME確定前テキストの送信ガード  | ADR-0022、要件定義書NFR-09              |
| 性能予算(同期遅延の数値基準)         | 要件定義書NFR-01                        |

各項目の要件と設計の記述は破棄せず、適用時期のみを繰り延べる。対象ADRのステータスはAcceptedのまま維持し、ステータス節に「適用はフェーズ2以降」の注記を追加する。機能要件と非機能要件のフェーズ区分表は要件定義書 第8章に置き、同表を唯一の正本とする。

本ADRの対象は2026-07-05のユーザー決定で凍結された上記項目のみである。上記以外のADRの適用時期は本ADRでは変更しない。

## 根拠

1. 設計と実装の乖離がレビュー非収束の根本原因であり、フェーズ区分の明文化によって受け入れ判定の基準線が一意になる。
2. 現行実装はユニットテスト30件(server 14件、web 14件、export-worker 2件)とE2Eの主要導線が検証済みであり、MVPとして提示できる動作水準にある。
3. 既存ADRの決定と根拠を破棄せず保持するため、フェーズ2以降の実装着手時に再検討の手戻りが発生しない。
4. 受け入れ基準(ACCEPTANCE.md)の段階基準と同一のフェーズ境界を文書体系全体で共有でき、境界衝突の検査をgrepで機械化できる。

## 影響

### ポジティブな影響

- 受け入れレビューの合否判定が現行実装水準で一意に定まる。
- 要件定義書のMVP記述と実装の不一致という指摘が構造的に解消する。

### ネガティブな影響

- 既存文書の「MVP」の語は、要件定義書 第8章のフェーズ区分表との読み替えが必要になる。
- 対象ADR(ADR-0001、ADR-0002、ADR-0003、ADR-0004、ADR-0005、ADR-0007、ADR-0009、ADR-0022)へ注記が追加され、文書量が増える。

### リスクと緩和策

- リスク: フェーズ境界の解釈揺れにより、フェーズ2以降の項目を「フェーズ1で実装する」文脈で言及する記述が混入する。緩和策: 封緘前にフェーズ境界キーワードのgrep検査を実施し、要件定義書 第8章のフェーズ区分表を唯一の正本として照合する。
- リスク: フェーズ2着手時に対象ADRの前提(ライブラリの版や周辺環境)が陳腐化している。緩和策: 各フェーズ着手時に対象ADRの現行性を再確認し、必要なら差し替えADRを起票する。

## 自己レビュー記録

### レビュー観点1 (ADR-0029): 既存ADRとの決定矛盾

対象ADRの決定内容は変更せず適用時期のみを繰り延べるため、決定の矛盾は発生しない。対象ADRのステータス節に本ADRへの参照注記を追加し、関連ADR節の双方向参照を成立させた。

### レビュー観点2 (ADR-0029): MVP境界の一意性

フェーズ1の適用範囲は要件定義書 第8章のフェーズ区分表を正本とし、本ADRの決定節の列挙と同一内容であることをgrepで照合した。

### レビュー観点3 (ADR-0029): 受け入れ基準との整合

ACCEPTANCE.mdのv1合格基準が現行実装で検証済みの項目のみで構成されることを確認し、日本語IME、性能予算、カバレッジ閾値、永続化、OTの各項目が次フェーズの一覧へ移されていることを照合した。
