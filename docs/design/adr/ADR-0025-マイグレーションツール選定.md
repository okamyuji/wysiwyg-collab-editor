# ADR-0025 マイグレーションツールにnode-pg-migrateを採用する

## ステータス

Accepted (2026-06-26)

## 文脈

PostgreSQLのスキーママイグレーション管理ツールを選定する。

## 検討した選択肢

### 選択肢A: node-pg-migrate(本案)

プレーンSQL中心で軽量

### 選択肢B: Knex

アクティブメンテナンスがADR策定時点で停滞しているため不採用

### 選択肢C: Prisma Migrate

本MVPはORMにPrismaを採用しないため不整合

### 選択肢D: Drizzle Kit

TypeScript型生成は魅力的だが本MVPの素直なSQL運用と不整合

### 選択肢E: Atlas

Goバイナリの追加で運用負担が増加するため不採用

### 選択肢F: TypeORM

本MVPはORM不採用方針のため不整合

### 選択肢G: Flyway

Java実行環境の追加で運用負担が増加するため不採用

## 決定

選択肢Aを採用する。具体仕様は要件定義書および基本設計書の該当節に置き、本ADRは決定の所在と根拠を保持する。

### 確定値

| 項目         | 値                                                    |
| ------------ | ----------------------------------------------------- |
| ツール       | node-pg-migrate                                       |
| 形式         | プレーンSQL                                           |
| ロールバック | 各マイグレーションにdown必須                          |
| CI統合       | `pnpm migrate:dry` をADR-0009のscan-secrets後段で実行 |
| 命名規則     | `YYYYMMDDHHMMSS_<slug>.sql`                           |

## 根拠

採用根拠は以下のとおりである。

1. プレーンSQLでマイグレーションを管理できる。
2. Node.jsエコシステム内で完結する。
3. ロールバック手順が単純である。
4. 本MVPの規模と運用体制に十分な機能を提供する。

## 影響

### ポジティブな影響

1. 主要設計決定のMECEな整理に寄与する。
2. 要件定義書および基本設計書とのトレーサビリティが明確になる。
3. 当該分野の確定値を1箇所に集約することで、将来の差し替えADR起票時の起点が明確になる。

### ネガティブな影響

1. 詳細仕様を要件定義書と基本設計書に委ねるため、ADR単体での自己完結性は限定的である。
2. 関連文書の更新時には本ADRも見直す必要がある。

### リスクと緩和策

1. 並行マイグレーション衝突に対しては、本番デプロイ時のシリアル実行を運用ルール化する。
2. メンテナンス状況の継続的な確認が必要となるため、四半期ごとに代替候補を再評価する。
3. 複雑なオブジェクト変更(マテリアライズドビュー、パーティショニング変更など)では手動SQLが必要となる場合がある。
