# 詳細設計

本ディレクトリは実装直前の契約(DDL、API、CAS SQL、エラーコード、定数、アルゴリズムパラメータ)を集約する。各章末に「対応要件 / 対応基本設計節 / 対応テストID」のトレーサビリティ表を必ず置く。

詳細設計01〜30章はフェーズ2以降の目標アーキテクチャを規定する。フェーズ1(現行実装水準)の適用範囲はADR-0029と要件定義書 第8章のフェーズ区分表を正本とし、各章の「本MVP」という表現はこの区分に読み替える。

## 文書階層の粒度ポリシー

文書は3層構造で粒度を分離する。各層には載せてよい内容と禁止項目がある。

### ADR (`docs/design/adr/`)

載せる内容は決定、根拠、検討した代替案、影響、関連ADRの5項目に限る。具体実装値は載せない。

| 載せる                                             | 載せない                                   |
| -------------------------------------------------- | ------------------------------------------ |
| 決定の言明(例: 「セッションCookie方式を採用する」) | Cookie属性の具体値、Argon2idのメモリ反復値 |
| 検討した選択肢A/B/Cの長短                          | CAS SQL本文、HTTPステータスコード          |
| 根拠と影響                                         | DDLカラム型、エラーコード一覧              |
| 関連ADRへのリンク                                  | アルゴリズムの具体パラメータ               |

### 基本設計(`docs/design/03-basic-design.md`)

載せる内容はコンポーネント分割、責務、論理データモデル、主要フロー(順序図相当)、依存関係に限る。

| 載せる                                   | 載せない                                     |
| ---------------------------------------- | -------------------------------------------- |
| ER相当の論理モデル(エンティティ、関係)   | DDL、PostgreSQL固有型、CHECK制約本文         |
| 主要フローの順序(どの順で何が呼ばれるか) | CAS SQL、HTTPステータスコード、エラーコード  |
| コンポーネント境界と責務                 | アルゴリズムの具体パラメータ(Argon2id、HMAC) |
| 依存ライブラリ名                         | バージョン番号の具体値、HEX値                |

### 詳細設計(本ディレクトリ)

実装に必要な契約をすべて集約する。

| 載せる                                              | 載せない                       |
| --------------------------------------------------- | ------------------------------ |
| 全DDL (CREATE文、CHECK制約、インデックス、トリガ)   | 検討した選択肢(ADR管轄)        |
| 全API契約(REST/WSの入出力スキーマ、エラーコード)    | 採用根拠(ADR管轄)              |
| 全CAS SQL、二段CASのstep手順                        | 論理モデル概観(基本設計管轄)   |
| アルゴリズムパラメータ(Argon2id、HMAC、CSPハッシュ) | 主要フロー全体像(基本設計管轄) |
| 定数表(HEX、URL、上限値、TTL)                       |                                |
| エラーコード一覧                                    |                                |
| テストID紐付け表                                    |                                |

## 並行開発のための層構造と互いに素規約

詳細設計は並行開発を可能にするため、4層構造で依存方向を単方向化する。下位層は上位層を知らない。同じ層内の章は互いに素(mutually exclusive)であり、並行開発可能である。

### 層構造

| 層         | 説明                                 | 凍結順序         | 並行可否     |
| ---------- | ------------------------------------ | ---------------- | ------------ |
| 基盤層     | 全章から参照される共有契約           | 1番目に凍結      | 順次         |
| 横断層     | 全体に薄く効く規約。基盤層のみに依存 | 2番目に凍結      | 章間並行     |
| ドメイン層 | 機能単位。互いに素                   | 3番目以降        | 全章完全並行 |
| データ層   | DDLは各ドメイン章が所有              | ドメイン章と同時 | 章間並行     |

### 互いに素にするための3規約

1. テーブル所有権規約: 各テーブルは1つの所有章を持ち、所有章だけがDDLを書く。他章はAPI経由でのみ読む。`01-data-model-ddl.md` はテーブル所有マップを持つ目次に縮退する。
2. エラーコード名前空間規約: 章接頭辞(`SUG-`、`EXP-`、`IMG-` 等)でコード文字列を分離する。`19-error-code-registry.md` は各章PRが自分の接頭辞でエントリを追記する登録簿である。
3. 章間連携はイベントのみとする規約: 関数の直接呼び出しは禁止。`audit_logs` 追記、ShareDB pub/sub、`image_purge_queue` などのキューで連携する。

## 章テンプレート

各章は以下の構成で書く。テンプレートは `_template.md` を参照。

1. スコープ(この章が責任を持つ契約の範囲)
2. 契約(DDL、API、CAS、エラーコード、定数、アルゴリズムパラメータのいずれか)
3. テストID紐付け表
4. トレーサビリティ表(対応要件 / 対応基本設計節 / 対応ADR)

## トレーサビリティ

`docs/_quality/TRACEABILITY_MATRIX.md` でFR/NFR → 基本設計§ → 詳細設計§ → テストIDの逆引きができる。詳細設計の各章末尾の表と機械的に突合する。

## 実装着手判定基準

以下4点が同時に満たされたとき、実装フェーズに進める。

1. 全章が状態「整備済み」になり、各章が契約+テストID紐付け表を持つ。
2. `docs/_quality/TRACEABILITY_MATRIX.md` で全FR/NFRが詳細設計章とテストIDの両方に対応する。
3. doclint 0/0/0/0、ADR交差決定green、MVPボーダリgreen、コーパス腐敗green、Rubricフレッシュネスgreen。
4. 対象フェーズの確認(ADR-0029)。着手対象の章が要件定義書 第8章のフェーズ区分表でどのフェーズに属するかを確認し、フェーズ2以降の章はフェーズ1の受け入れ判定に含めない。
